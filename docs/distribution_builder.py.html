Technische Dokumentation: distribution_builder.py

1. Überblick

Die Datei distribution_builder.py dient als zentrales Skript zur Erstellung von Distributionen basierend auf einem gegebenen Projektverzeichnis. Es definiert die Logik für die Auswahl des Build-Flavors, das Skripten von Abhängigkeiten, das Kompaktieren von Dateien und das Erstellen von Archivdateien für verschiedene Zielsysteme (z.B. Windows Server, Linux). Das Skript verwendet Dataclasses zur strukturierten Speicherung von Build-Ergebnissen und definiert Konfigurationskonstanten für Flavors und zu überspannende Dateien/Ordner.

2. Konfiguration und Konstanten

Das Skript definiert mehrere globale Konstanten, die die Build-Strategie steuern:

2.1. PACKAGE_FLAVORS

Definiert die unterstützten Build-Flavors, die das Skript verarbeiten kann:

  • "distribution"
  • "windows-server-package"
  • "linux-server-package"
  • "linux-project"

2.2. SKIP_NAMES

Eine Menge von Dateinamen oder Ordnernamen, die während des Build-Prozesses ignoriert werden sollen, um unnötige Dateien oder Abhängigkeiten zu vermeiden:

  • .git
  • .github
  • .pytest_cache
  • __pycache__
  • .nova
  • LIT
  • node_modules
  • dist
  • build

2.3. SKIP_FILE_SUFFIXES

Eine Menge von Dateiendungen, die als temporär oder nicht relevant für die Distribution betrachtet werden:

  • .pyc
  • .pyo
  • .pyd
  • .db
  • .shm
  • .wal
  • .zip
  • .litertlm
  • .gguf
  • .task

2.4. SKIP_FILE_NAMES

Eine Menge spezifischer Dateien, die explizit ignoriert werden sollen:

  • analysis_dump.md
  • Projektcode_analysis_dump.md
  • info.png

3. Datenstrukturen (Dataclasses)

Das Skript verwendet drei dataclass-Definitionen zur strukturierten Rückgabe von Build-Ergebnissen:

3.1. DistributionBuildResult

Speichert die Ergebnisse eines vollständigen Distribution-Builds:

  • version (str): Die Version des gebildeten Artefakts.
  • flavor (str): Der verwendete Build-Flavor.
  • archive_path (Path): Der Pfad zur finalen Archivdatei.
  • staging_root (Path): Der Pfad zum temporären Staging-Verzeichnis.

3.2. DistributionMaterializeResult

Speichert die Ergebnisse, wenn Materialien (z.B. spezifische Komponenten) in ein Zielverzeichnis materialisiert werden:

  • version (str): Die Version.
  • flavor (str): Der verwendete Build-Flavor.
  • target_root (Path): Das Zielverzeichnis für die Materialisierung.

3.3. LinuxProjectBuildResult

Speichert die Ergebnisse eines spezifischen Builds für Linux-Projekte:

  • version (str): Die Version.
  • archive_path (Path): Der Pfad zur Archivdatei.
  • staging_root (Path): Der Pfad zum temporären Staging-Verzeichnis.

4. Funktionen

4.1. detect_project_version(base_path: Path) -> str

Beschreibung: Diese Funktion liest die pyproject.toml Datei im angegebenen base_path und versucht, die Versionsnummer zu extrahieren. Logik:

  1. Sie liest den gesamten Inhalt der pyproject.toml als Text.
  2. Sie durchläuft jede Zeile und prüft, ob diese mit "version = " beginnt.
  3. Wenn eine solche Zeile gefunden wird, wird der Wert nach dem Gleichheitszeichen (=) extrahiert und zurückgegeben.
  4. Falls keine Versionszeile gefunden wird, wird standardmäßig "0.1.0" zurückgegeben.

    5. Parameter:

  • base_path: Path: Der Pfad zum Verzeichnis, das die pyproject.toml enthält.

    • Rückgabe:

  • str: Die extrahierte Versionsnummer.

5. Zusammenfassung der Struktur

Das Skript ist modular aufgebaut. Die Konfiguration ist klar getrennt, die Ergebnisse sind durch spezifische Dataclasses definiert, und die Logik für die Versionserkennung ist isoliert. Die Hauptfunktion (die im bereitgestellten Ausschnitt nicht vollständig sichtbar ist) wird wahrscheinlich diese Komponenten nutzen, um den Build-Prozess für einen gewählten Flavor zu orchestrieren.

Technische Dokumentation: build_distribution_archive Funktion

Übersicht

Die Funktion build_distribution_archive ist eine zentrale Methode zur Erstellung einer vollständigen Installations- oder Distributions-Archivdatei (ZIP-Datei) für ein Projekt. Sie übernimmt den gesamten Prozess, von der Kopie des Quellverzeichnisses bis hin zur finalen Komprimierung, wobei spezifische Anpassungen basierend auf der gewählten flavor (z.B. "distribution") vorgenommen werden.

Funktionssignatur

def build_distribution_archive(
base_path: Path,
output_dir: Path | None = None,
version: str | None = None,
flavor: str = "distribution",
) -> DistributionBuildResult:

Parameter

Parameter Typ Beschreibung Standardwert
base_path Path Der absolute oder rel relative Pfad zum Quellverzeichnis des Projekts, das verpackt werden soll. Erforderlich
output_dir `Path None` Der Pfad, an dem die finale ZIP-Datei gespeichert werden soll. Wenn None angegeben wird, wird das Quellverzeichnis (base_path) als Basis für den Output-Pfad verwendet. None
version `str None` Eine spezifische Versionsnummer, die in den Namen des Archivs enthalten sein soll. Wenn None ist, wird automatisch die Projektversion über detect_project_version(base_path) ermittelt. None
flavor str Definiert den Typ oder die Zielumgebung der Distribution (z.B. "distribution"). Wird zur Namensgenerierung und zur Anpassung der Struktur verwendet. "distribution"

Rückgabewert

Typ Beschreibung
DistributionBuildResult Ein Objekt, das die Ergebnisse des Build-Prozesses zusammenfasst. Enthält:
version Die finale Versionsnummer, die im Archiv verwendet wurde.
flavor Der normalisierte Flavor-Name, der verwendet wurde.
archive_path Der vollständige Pfad zur erstellten ZIP-Datei.
staging_root Der Pfad zum temporären Verzeichnis, in dem die Distribution vor der Komprimierung aufgebaut wurde.

Funktionsweise (Workflow)

Die Funktion folgt einem mehrstufigen Prozess, der innerhalb eines temporären Verzeichnisses abläuft:

  1. Pfad-Normalisierung und Namensgenerierung:
  • base_path wird auf einen absoluten Pfad normalisiert.
  • Die Versionsnummer wird entweder explizit über version oder automatisch über detect_project_version(base_path) bestimmt.
  • output_dir wird auf einen absoluten Pfad normalisiert.
  • Der flavor wird durch _normalize_flavor(flavor) bereinigt.
  • Der finale package_name wird generiert im Format: Nova-School-Server-v[version_text]-[flavor_text].
  • Der Zielpfad für die ZIP-Datei (archive_path) wird definiert.
  1. Temporäre Arbeitsumgebung (Staging):
  • Ein temporäres Verzeichnis wird mit dem Präfix nova-school-distribution- erstellt (tempfile.TemporaryDirectory).
  • Innerhalb dieses Verzeichnisses wird das staging_root (basierend auf dem package_name) erstellt.
  1. Projektstruktur-Kopie:
  • Der gesamte Quellbaum (base_path) wird in das temporäre Staging-Verzeichnis kopiert (_copy_project_tree).
  1. Distribution-Scaffolding:
  • Eine spezifische Struktur für die Distribution wird in das Staging-Verzeichnis erstellt (_create_distribution_scaffold), die auf der Version und dem Flavor basiert.
  1. Flavor-spezifische Bereinigung:
  • Elemente, die für den gewählten Flavor nicht benötigt werden, werden aus dem Staging-Verzeichnis entfernt (_prune_for_flavor).
  1. Binärkopie:
  • Optional vorhandene Linux-Runtime-Binärdateien werden basierend auf dem Flavor kopiert (_copy_optional_linux_runtime_binaries).
  1. Archivierung:
  • Falls die Ziel-ZIP-Datei bereits existiert, wird sie gelöscht.
  • Der gesamte Inhalt des staging_root wird in die finale ZIP-Datei komprimiert (_zip_tree).
  1. Rückgabe:
  • Die Funktion gibt ein DistributionBuildResult zurück, das alle wichtigen Pfade und Metadaten enthält.

Abhängigkeiten (Impliziert)

  • pathlib.Path: Für die Verwaltung von Dateipfaden.
  • tempfile: Für die Erstellung temporärer Verzeichnisse.
  • _copy_project_tree(...): Eine interne Funktion zur Kopie des Quellverzeichnisses.
  • _create_distribution_scaffold(...): Eine interne Funktion zur Erstellung der Distributionenstruktur.
  • _prune_for_flavor(...): Eine interne Funktion zur Entfernung unnötiger Dateien.
  • _copy_optional_linux_runtime_binaries(...): Eine interne Funktion zur Kopie von Systembinären.
  • _zip_tree(...): Eine interne Funktion zur Komprimierung des Verzeichnisses.
  • detect_project_version(...): Eine externe Funktion zur Versionserkennung.
  • _normalize_flavor(...): Eine externe Funktion zur Normalisierung des Flavor-Namens.
  • DistributionBuildResult: Eine definierte Datenstruktur für die Rückgabe.

Anwendungsfälle

Diese Funktion ist ideal für den automatisierten Build-Prozess von Distributionen, bei denen:

  1. Versionskontrolle zentralisiert werden muss.
  2. Flavor-spezifische Anpassungen (z.B. Entfernen von Desktop-Komponenten oder Hinzufügen von Server-Binärs) erforderlich sind.
  3. Ein konsistentes Archivformat für die Bereitstellung benötigt wird.

Funktion: build_linux_project_archive

Diese Funktion dient dazu, den Prozess des Erstellens eines Archivs für ein Linux-Projekt zu initiieren und die Ergebnisse zu strukturieren. Sie ruft die Kernfunktion build_distribution_archive auf und fasst das Ergebnis in eine spezifische LinuxProjectBuildResult-Klasse zusammen. ---

Signaturen

def build_linux_project_archive(
base_path: Path,
output_dir: Path | None = None,
version: str | None = None,
) -> LinuxProjectBuildResult:

Beschreibung

Diese Funktion orchestriert die Erstellung eines Distributionsarchivs für ein Linux-Projekt. Sie übernimmt die Bereitstellung der notwendigen Parameter an die zugrunde liegende Funktion build_distribution_archive und transformiert das zurückgegebene Ergebnis in ein Objekt vom Typ LinuxProjectBuildResult.

Parameter

Parameter Typ Beschreibung Pflicht
base_path Path Der Pfad zum Basisverzeichnis des Linux-Projekts, aus dem das Archiv erstellt werden soll. Ja
output_dir `Path None` Optionaler Pfad, in dem das finale Archiv gespeichert werden soll. Wenn None angegeben wird, wird der Standardpfad für die Ausgabe verwendet. Nein
version `str None` Optionaler Versionsstring, der in das generierte Archiv integriert werden soll. Nein

Rückgabe (Return Value)

Typ Beschreibung
LinuxProjectBuildResult Ein Objekt, das die Schlüsselinformationen des erfolgreichen Build-Prozesses enthält:
version Die generierte Versionsnummer des Builds.
archive_path Der vollständige Pfad zum erstellten Archiv.
staging_root Der Pfad zum Staging-Verzeichnis, das für den Build verwendet wurde.

Abhängigkeiten

  • build_distribution_archive(...): Eine interne oder externe Funktion, die den eigentlichen Build-Prozess durchführt.
  • Path: Ein Objekt aus dem pathlib-Modul, das für Pfadmanipulation verwendet wird.
  • LinuxProjectBuildResult: Eine definierte Klasse, die die strukturierten Ergebnisse des Builds hält.

Workflow

  1. Die Funktion ruft build_distribution_archive auf, wobei base_path, output_dir, version und der feste Flavor "linux-project" als Argumente übergeben werden.
  2. Das Ergebnis der build_distribution_archive-Funktion wird entgegennimmt.
  3. Die zurückgegebene LinuxProjectBuildResult-Instanz wird neu konstruiert, wobei die Attribute version, archive_path und staging_root aus dem Ergebnis der Kernfunktion extrahiert werden.
  4. Das konstruierte LinuxProjectBuildResult wird zurückgegeben.

    5. ---

Beispielanwendung (Konzeptuell)

from pathlib import Path
# Angenommen, die notwendigen Klassen und Funktionen sind importiert
# from .core import build_distribution_archive, LinuxProjectBuildResult
# Beispielpfade
project_root = Path("/home/user/my_linux_project")
output_location = Path("/tmp/builds")
build_version = "1.2.3-beta"
# Aufruf der Funktion
try:
build_result = build_linux_project_archive(
base_path=project_root,
output_dir=output_location,
version=build_version
)
print(f"Build erfolgreich abgeschlossen.")
print(f"Versionsnummer: {build_result.version}")
print(f"Archiv gespeichert unter: {build_result.archive_path}")
except Exception as e:
print(f"Fehler beim Bauen des Archivs: {e}")

Technische Dokumentation: materialize_distribution_directory

1. Übersicht

Die Funktion materialize_distribution_directory ist verantwortlich für die Erstellung einer vollständigen, versionierten und flavor-spezifischen Distribution-Struktur an einem Zielort. Sie kopiert den Projektbaum, erstellt eine Distribution-Skelettstruktur und fügt optional Betriebssystem-spezifische Laufzeitbinärbestände hinzu, bevor sie das Zielverzeichnis ersetzt.

2. Funktion

Name: materialize_distribution_directory Beschreibung: Materialisiert die Projektstruktur in ein definiertes Zielverzeichnis, basierend auf dem Quellprojekt, einer optionalen Version und einer Flavor-Konfiguration. Signature:

def materialize_distribution_directory(
base_path: Path,
target_root: Path,
*,
version: str | None = None,
flavor: str = "distribution",
) -> DistributionMaterializeResult:

Parameter:

Parameter Typ Beschreibung Wertsatz
base_path Path Der Pfad zum Quellprojekt (Root-Verzeichnis des Projekts). Erforderlich
target_root Path Der Pfad, an dem die fertige Distribution erstellt werden soll. Erforderlich
version `str None` Eine spezifische Version, die für die Distribution verwendet werden soll. Wenn None, wird die Version automatisch aus dem Projekt (detect_project_version) abgeleitet. None
flavor str Die gewünschte Flavor-Konfiguration für die Distribution (z.B. "distribution"). "distribution"

Werte:

  • Rückgabe: DistributionMaterializeResult – Ein Objekt, das die erstellten Version, Flavor und den Zielpfad zurückgibt.

3. Funktionsweise und Logik

Die Funktion folgt einem mehrstufigen Prozess, der in einem temporären Verzeichnis abläuft, um die Integrität des Quellprojekts zu gewährleisten.

  1. Pfadauflösung und Normalisierung:
  • base_path und target_root werden aufgelöst (resolve(strict=False)), um relative Pfade zu absoluten Pfaden umzuwandeln.
  • Die version wird entweder explizit übergeben oder durch die Funktion detect_project_version(base_path) bestimmt.
  • Die flavor wird durch _normalize_flavor(flavor) bereinigt.
  1. Temporäre Arbeitsumgebung:
  • Ein temporäres Verzeichnis wird erstellt (tempfile.TemporaryDirectory), um die Materialisierung sicher zu isolieren.
  • Innerhalb dieses Verzeichnisses wird das staging_root für die Materialisierung definiert.
  1. Projektkopierung:
  • Ausschluss von Pfaden: Es wird geprüft, ob das target_root relativ zum base_path liegt. Falls ja, wird dieser relative Pfad in eine Liste von excluded_relative Pfaden aufgenommen, um ihn beim Kopieren auszuschließen.
  • Kopieren des Projektbaums: Die Funktion _copy_project_tree kopiert den gesamten Projektbaum von base_path in das temporäre staging_root, wobei die identifizierten relativen Pfade ausgeschlossen werden.
  1. Distribution-Skelett-Erstellung:
  • Skelett-Erstellung: _create_distribution_scaffold erstellt die notwendige Grundstruktur für die Distribution basierend auf der Version und dem Flavor.
  • Flavor-Pruning: _prune_for_flavor entfernt oder modifiziert Teile des Skeletts, die für das spezifische Flavor nicht relevant sind.
  1. Optionaler Code-Kopierer:
  • Laufzeitbinärbestände: _copy_optional_linux_runtime_binaries kopiert optional spezifische Binärdateien für Linux-Server-Pakete, basierend auf dem flavor_text.
  1. Ziel-Deployment:
  • Ziel-Bereinigung: Falls das target_root bereits existiert, wird es entfernt (shutil.rmtree(target_root, ignore_errors=True)).
  • Ziel-Erstellung: Das Verzeichnis-Elternverzeichnis des target_root wird erstellt.
  • Endkopierer: shutil.copytree(staging_root, target_root, dirs_exist_ok=True) kopiert den gesamten vorbereiteten staging_root in das finale target_root.
  1. Rückgabe:
  • Die Funktion gibt ein DistributionMaterializeResult zurück, das die erfolgreich verwendeten Version, Flavor und den finalen Zielpfad enthält.

4. Abhängigkeiten

  • pathlib.Path
  • tempfile
  • shutil
  • (Implizit: detect_project_version, _normalize_flavor, _copy_project_tree, _create_distribution_scaffold, _prune_for_flavor, _copy_optional_linux_runtime_binaries)

5. Fehlerbehandlung

  • Pfadfehler: Die Behandlung des ValueError beim Versuch, target_root relativ zu base_path zu berechnen, stellt sicher, dass die Funktion robust ist, falls die Pfade nicht direkt vergleichebar sind.
  • Kopierfehler: Die Verwendung von ignore_errors=True bei shutil.rmtree stellt sicher, dass das Skript nicht abstürzt, falls das Zielverzeichnis bereits existiert und nicht korrekt gelöscht werden kann.

6. Verwendungshinweise

Diese Funktion sollte verwendet werden, um ein vollständiges, deploybares Artefakt (eine Distribution) aus einem Quellprojekt zu generieren. Stellen Sie sicher, dass base_path und target_root gültige, existierende Pfade sind.

Technische Dokumentation: distribution_builder.py

Dieses Dokument beschreibt die Funktionen aus der Datei distribution_builder.py, die für die Normalisierung von Distributionen und das Kopieren von Projektstrukturen zuständig sind. ---

1. Funktion: _normalize_flavor(flavor: str) -> str

Diese interne Funktion dient dazu, einen eingegebenen String, der eine Distribution oder einen Flavor darstellt, in einen standardisierten, internen Flavor-Namen umzuwandeln. Zweck: Standardisierung von Distributionen und deren Aliasnamen, um eine konsistente Verarbeitung in nachfolgenden Prozessen zu gewährleisten. Parameter:

  • flavor: str: Der eingegebene String, der die Distribution oder den Flavor darstellt (z.B. "Windows Server", "Linux").

    • Rückgabe:

  • str: Der normalisierte Flavor-Name, der in PACKAGE_FLAVORS definiert ist.

    • Logik:

  1. Textvorverarbeitung: Der eingegebene flavor wird in einen String umgewandelt, entfernt Leerzeichen und wird kleingestellt (.strip().lower()). Standardmäßig wird "distribution" verwendet, falls flavor leer ist.
  2. Alias-Mapping: Es wird ein Dictionary (aliases) verwendet, das verschiedene Eingaben (z.B. "windows", "linux-server") mit ihren internen Schlüsselwörtern (z.B. "windows-server-package", "linux-server-package") verknüpft.
  3. Normalisierung: Es wird versucht, den Text im aliases-Dictionary zu finden.
  4. Validierung: Wenn der normalisierte Name nicht in der globalen Konstante PACKAGE_FLAVORS enthalten ist, wird eine ValueError mit einer Fehlermeldung ausgelöst, die den ungültigen Flavor angibt.
  5. Rückgabe: Der gefunden und validierte normalisierte Name wird zurückgegeben.

    6. Wichtige Abhängigkeiten:

  • PACKAGE_FLAVORS: Eine globale Konstante (nicht im bereitgestellten Code sichtbar), die die Liste der gültigen, normalisierten Flavor-Namen enthält.

    • ---

2. Funktion: _copy_project_tree(source_root: Path, target_root: Path, *, excluded_relative_paths: Iterable[Path] | None = None) -> None

Diese interne Funktion kopiert die gesamte Projektstruktur (Dateien und Verzeichnisse) von einem Quellverzeichnis (source_root) in ein Zielverzeichnis (target_root), wobei bestimmte Pfade ausgeschlossen werden können. Zweck: Repräsentiert den Kernprozess des Projekt- oder Verzeichniskopierens, einschließlich der Anwendung von Exklusionsregeln. Parameter:

  • source_root: Path: Der Pfad zum Quellverzeichnis, dessen Inhalt kopiert werden soll.
  • target_root: Path: Der Pfad, in den die Kopie des Quellverzeichnisses kopiert werden soll.
  • excluded_relative_paths: Iterable[Path] | None = None: Eine optional übergeordnete Liste von relativen Pfaden, die im Quellverzeichnis ausgeschlossen werden sollen. Wenn None übergeben wird, werden keine Exklusionen angewendet.

    • Rückgabe:

  • None

    • Logik:

  1. Exklusionsvorbereitung:
  • Die übergebenen excluded_relative_paths werden bereinigt und normalisiert (_normalize_relative_path), um sicherzustellen, dass sie mit den Pfaden im Quellverzeichnis verglichen werden können.
  • Die Ergebnisse werden durch _is_excluded_relative_path gefiltert.
  1. Iteration über Quellverzeichnis:
  • Jedes Element (item) im source_root wird iteriert.
  • Relative Pfadberechnung: Der relative Pfad des aktuellen Elements zum source_root wird berechnet (item.relative_to(source_root) und dann _normalize_relative_path).
  • Exklusionsprüfung: Es wird geprüft, ob der berechnete relative Pfad in der Liste der ausgeschlossenen Pfade enthalten ist (_is_excluded_relative_path). Wenn ja, wird das Element übersprungen (continue).
  • Root-Prüfung: Es wird geprüft, ob das Element die Wurzel des Quellverzeichnisses ist (_should_skip_root_entry). Wenn ja, wird übersprungen.
  1. Kopierprozess:
  • Verzeichnis: Wenn das Element ein Verzeichnis (item.is_dir()) ist, wird es mithilfe der internen Funktion _copy_directory in das Zielverzeichnis kopiert.
  • Datei: Wenn das Element eine Datei (item.is_file()) ist:
  • Es wird sichergestellt, dass das Zielverzeichnis der Datei existiert (destination.parent.mkdir(parents=True, exist_ok=True)).
  • Die Datei wird mithilfe von shutil.copy2 kopiert (um Metadaten wie Erstellungsdatum zu erhalten).

    • Wichtige Abhängigkeiten:

  • _normalize_relative_path(...): Eine interne Funktion zur Normalisierung von relativen Pfaden.
  • _is_excluded_relative_path(...): Eine interne Funktion zur Überprüfung der Exklusionsregeln.
  • _should_skip_root_entry(...): Eine interne Funktion zur Entscheidung, ob das Quellverzeichnis selbst übersprungen werden soll.
  • _copy_directory(...): Eine interne Funktion zum Kopieren von Verzeichnissen.
  • shutil.copy2: Standardmodul für das Kopieren von Dateien.

    • ---

1. Übersicht

Die Datei distribution_builder.py enthält Funktionen, die für den Aufbau oder die Erstellung einer Verteilung (wahrscheinlich eines Modells oder einer Anwendung) zuständig sind. Der Fokus liegt auf dem rekursiven Kopieren von Dateien und Verzeichnissen aus einer Quellstruktur in ein Zielverzeichnis, unter Anwendung verschiedener Filter- und Skipsregeln.

2. Funktionen

2.1. _copy_directory(source_dir: Path, target_dir: Path, *, source_root: Path, excluded_relative_paths: list[Path]) -> None

Diese interne Funktion ist verantwortlich für das rekursive Kopieren von Inhalten eines Quellverzeichnisses (source_dir) in ein Zielverzeichnis (target_dir). Parameter:

  • **source_dir: Path**: Der Pfad zum Verzeichnis, dessen Inhalt kopiert werden soll.
  • **target_dir: Path**: Der Pfad des Zielverzeichnisses, in das die Dateien kopiert werden.
  • **source_root: Path**: Der Stammverzeichnis-Pfad, der zur Normalisierung der relativen Pfade verwendet wird.
  • **excluded_relative_paths: list[Path]**: Eine Liste von relativen Pfaden, die während des Kopiervorgangs explizit ausgeschlossen werden sollen.

    • Logik:

  1. Iteration: Durchläuft alle Elemente (item) im source_dir.
  2. Relative Pfad Normalisierung: Berechnet den relativen Pfad des aktuellen Elements zum source_root mittels _normalize_relative_path().
  3. Ausschlussprüfung (Relative Pfade): Prüft, ob der normalisierte relative Pfad in der Liste excluded_relative_paths enthalten ist. Bei Übereinstimmung wird das Element übersprungen (continue).
  4. Skipsprüfung (Elemente): Prüft, ob das aktuelle Element durch _should_skip_entry(item, relative_path) übersprungen werden soll. Bei Übereinstimmung wird das Element übersprungen (continue).
  5. Zielpfad Bestimmung: Bestimmt den Zielpfad (destination) basierend auf dem Namen des Elements im target_dir.
  6. Verzeichnis-Kopieren:
  • Wenn das Element ein Verzeichnis (item.is_dir()) ist, wird die Funktion rekursiv aufgerufen, um den Inhalt des Verzeichnisses zu kopieren.
  • Wenn das Element eine Datei (item.is_file()) ist:
  • Stellt sicher, dass das übergeordnete Verzeichnis im Zielverzeichnis existiert (destination.parent.mkdir(parents=True, exist_ok=True)).
  • Kopiert die Datei mithilfe von shutil.copy2 (behält Metadaten wie Änderungszeit und Berechtigungen bei).

2.2. _should_skip_root_entry(path: Path, relative_path: Path) -> bool

Diese Funktion dient dazu, spezifische Elemente im Quellverzeichnis zu ignorieren, die sich im Wurzelverzeichnis befinden. Parameter:

  • **path: Path**: Der vollständige Pfad des Elements im Quellverzeichnis.
  • **relative_path: Path**: Der normalisierte relative Pfad des Elements zum Stammverzeichnis.

    • Logik:

  • **data:** Wenn der Name des Elements "data" ist, wird es übersprungen.
  • **Model:** Wenn der Name des Elements "Model" ist, wird es übersprungen.
  • Rekursion: Ruft _should_skip_entry(path, relative_path) auf, um weitere Skipsregeln zu prüfen.

2.3. _should_skip_entry(path: Path, relative_path: Path) -> bool

Diese zentrale Funktion entscheidet, ob ein spezifisches Element (path) basierend auf seinem Namen und seiner Position im Verhältnis zum Stammverzeichnis (relative_path) übersprungen werden soll. Parameter:

  • **path: Path**: Der vollständige Pfad des Elements.
  • **relative_path: Path**: Der normalisierte relative Pfad des Elements zum Stammverzeichnis.

    • Logik:

  1. Name-basierte Skips:
  • Wenn der Name des Elements in der globalen Konstante SKIP_NAMES enthalten ist, wird es übersprungen.
  1. Struktur-basierte Skips:
  • Wenn die ersten zwei Teile des relative_path "Linux" und "project" sind, wird es übersprungen.
  1. **Dateispezifische Skips (path.is_file()):**
  • Name-basierte Skips: Wenn der Dateiname in der globalen Konstante SKIP_FILE_NAMES enthalten ist, wird es übersprungen.
  • Temporäre Dateien: Wenn der Dateiname mit _tmp_ oder tmp_ beginnt, wird es übersprungen.
  • Spezifische Dateiendungen: Wenn der Dateiname eine der Marker (_analysis_dump.md oder .codedump.md) endet, wird es übersprungen.
  • Dateiendungs-Filter: Wenn die Dateiendung (kleingeschrieben) in der globalen Konstante SKIP_FILE_SUFFIXES enthalten ist, wird es übersprungen.
  1. Standardfall: Wenn keine der oben genannten Bedingungen zutrifft, wird False zurückgegeben (Element wird kopiert).

3. Abhängigkeiten und Konstanten (Impliziert)

Die Funktion setzt auf folgende externe Komponenten und Konstanten, die in der Dokumentation nicht explizit definiert sind, aber für die korrekte Funktion sind notwendig:

  • **Path**: Typ für Dateipfade (wahrscheinlich aus dem pathlib-Modul).
  • **shutil**: Modul für Dateitransaktionen (speziell shutil.copy2 und Path.mkdir).
  • **_normalize_relative_path(item: Path) -> Path**: Eine externe Funktion zur Bereinigung und Normalisierung relativer Pfade.
  • **_is_excluded_relative_path(relative_path: Path, excluded_relative_paths: list[Path]) -> bool**: Eine externe Funktion zur Überprüfung auf explizite Ausnahmen.
  • **_should_skip_entry(path: Path, relative_path: Path) -> bool**: Die Hauptlogik für die Skipsprüfung.
  • Konstanten:
  • SKIP_NAMES: Eine set oder list von String-Namen, die Dateien/Verzeichnisse ignorieren sollen.
  • SKIP_FILE_NAMES: Eine set oder list von String-Namen, die Dateien ignorieren sollen.
  • SKIP_FILE_SUFFIXES: Eine set oder list von String-Suffixen, die Dateien ignorieren sollen.

1. _normalize_relative_path(path: Path) -> Path

Beschreibung: Diese interne Funktion nimmt einen Path-Objekt entgegen und normalisiert ihn, indem sie leere Strings und Punkte (.) aus den Komponenten entfernt. Parameter:

normalized_path = _normalize_relative_path(path_input)
# Erwartete Ausgabe: Path('a/b/c') (oder ähnlich, abhängig von der genauen Implementierung von Path(*...) bei leeren Komponenten)

2. _is_excluded_relative_path(relative_path: Path, excluded_relative_paths: list[Path]) -> bool

Beschreibung: Diese interne Funktion prüft, ob ein gegebenes relative_path von einer Liste von excluded_relative_paths ausgeschlossen ist. Die Prüfung erfolgt auf zwei Bedingungen:

  1. Exakte Übereinstimmung: Der relative_path ist exakt gleich einem der ausgeschlossenen Pfade.
  2. Präfix-Übereinstimmung: Die ersten Teile des relative_path entsprechen den ersten Teilen eines ausgeschlossenen Pfades (d.h., der relative_path beginnt mit dem ausgeschlossenen Pfad).

    3. Parameter:

  • relative_path: Path: Der Pfad, der geprüft werden soll.
  • excluded_relative_paths: list[Path]: Eine Liste von Path-Objekten, die als Ausnahmen definiert sind.

    • Rückgabe:

  • bool: True, wenn der relative_path ausgeschlossen ist (entweder exakt oder als Präfix), False andernfalls.

    • Details:

  • Die Funktion iteriert durch jede excluded Pfad in der Liste.
  • Sie ignoriert Pfade, die keine Komponenten haben (if not excluded.parts: continue).
  • Die Prüfung erfolgt, ob relative_path == excluded oder ob die ersten Teile von relative_path mit den ersten Teilen von excluded übereinstimmen (relative_path.parts[: len(excluded.parts)] == excluded.parts).

    • Beispiel: Angenommen:

  • relative_path = Path("data/config/settings.json")
  • excluded_relative_paths = [Path("data/config"), Path("other/file")]

    • Die Funktion würde True zurückgeben, da relative_path.parts[: len(Path("data/config").parts)] gleich Path("data/config").parts ist.

Abhängigkeiten

  • pathlib.Path: Für die Manipulation von Pfadstrukturen.

Anmerkungen

  • Die Funktionen sind als interne Hilfsfunktionen (_) gekennzeichnet, was darauf hindeutet, dass sie primär zur Unterstützung komplexerer Logik innerhalb der Hauptklasse oder Funktion dienen.
  • Die Normalisierung (_normalize_relative_path) ist entscheidend, um konsistente Vergleiche von Pfaden zu gewährleisten.
  • Die Ausschlusslogik (_is_excluded_relative_path) implementiert eine gängige Mustererkennung für Pfad- oder Verzeichnis-Ausschlussmechanismen.

Funktion: _prune_for_flavor(staging_root: Path, flavor: str) -> None

Beschreibung

Diese interne Funktion dient dazu, die staging_root-Verzeichnisstruktur basierend auf dem angegebenen flavor-Typ zu bereinigen (Pruning). Sie entfernt spezifische Skripte, die für bestimmte Betriebssystem- oder Projekt-Flavors nicht relevant sind.

Parameter

  • **staging_root: Path**: Der absolute oder relative Pfad zum Verzeichnis, das bereinigt werden soll (das Staging-Verzeichnis).
  • **flavor: str**: Der spezifische Flavor-Typ, der die Bereinigungslogik steuert (z.B. "windows-server-package", "linux-server-package", "linux-project").

Rückgabe

  • None: Die Funktion führt keine Rückgabe aus; sie modifiziert das staging_root-Verzeichnis direkt.

Verhalten

Die Funktion implementiert eine bedingte Logik, um Dateien zu entfernen, die für einen bestimmten Flavor nicht benötigt werden.

  1. **Wenn flavor gleich "windows-server-package" ist:**
  • Es werden die folgenden relativen Pfade im staging_root überprüft und entfernt, falls sie existieren:
  • start_server.sh
  • start_worker.sh
  • run_tests.sh
  1. **Wenn flavor entweder "linux-server-package" oder "linux-project" ist:**
  • Es werden die folgenden relativen Pfade im staging_root überprüft und entfernt, falls sie existieren:
  • start_server.ps1
  • start_worker.ps1
  • run_tests.ps1

    • Unterstützte Flavors:

  • "windows-server-package"
  • "linux-server-package"
  • "linux-project"

Abhängigkeiten

  • **Path (von pathlib)**: Für die Manipulation von Dateipfaden.
  • **_remove_if_exists(Path)**: Eine interne Hilfsfunktion (nicht definiert in diesem Ausschnitt), die prüft, ob ein Pfad existiert, und ihn bei Existenz entfernt.

Verwendung (Beispiel)

from pathlib import Path
# Angenommen, staging_root ist ein Path-Objekt und flavor ist ein String
# Beispiel 1: Bereinigung für Windows Server Package

flavor_win = "windows-server-package"
_prune_for_flavor(staging_dir, flavor_win)
# Entfernt: start_server.sh, start_worker.sh, run_tests.sh
# Beispiel 2: Bereinigung für Linux Project
staging_dir_linux = Path("/tmp/staging/linux_project_build")
flavor_linux = "linux-project"
_prune_for_flavor(staging_dir_linux, flavor_linux)
# Entfernt: start_server.ps1, start_worker.ps1, run_tests.ps1

  • path: Path: Der Pfad zur Datei oder zum Verzeichnis, das überprüft werden soll.
