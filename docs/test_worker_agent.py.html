Technische Dokumentation: test_worker_agent.py

1. Überblick

Diese Dokumentation beschreibt die Struktur und den Zweck der Python-Datei test_worker_agent.py. Diese Datei dient primär der Unit-Testing-Abdeckung der Klasse WorkerAgent aus dem Modul nova_school_server.worker_agent. Sie verwendet das unittest-Framework sowie unittest.mock zur Simulation von Abhängigkeiten und zur Validierung der Funktionalität des Agenten.

2. Voraussetzungen

Um diese Tests auszuführen, sind folgende Abhängigkeiten erforderlich:

  • Python 3.x
  • Die Bibliothek nova_school_server (insbesondere das Modul worker_agent) muss installiert sein.

Technische Dokumentation: WorkerAgent Tests

Diese Dokumentation beschreibt die Testfälle der Klasse WorkerAgentTests aus der Datei test_worker_agent.py. Diese Tests dienen dazu, die korrekte Generierung von Befehlen durch den WorkerAgent für verschiedene Szenarien (Container-Befehle, Dateigrößenbeschränkungen, Integritätsprüfung von Artefakten) zu validieren. ---

1. Überblick

Die Klasse WorkerAgentTests verwendet das unittest-Framework, um die Funktionalität des WorkerAgent-Klassens zu testen. Die Tests konzentrieren sich auf folgende Hauptbereiche:

  1. Container-Befehlsgenerierung: Überprüfung, wie der Agent Befehle für Container-Workloads erstellt, insbesondere im Hinblick auf die Verwendung von Materialized Workspaces und die korrekte Handhabung von Container-spezifischen Parametern.
  2. Parameterkonservierung und Konvertierung: Überprüfung, dass wichtige Pfade und Dateigrößenlimits korrekt im generierten Befehl beibehalten oder in die erforderliche Formatierung umgewandelt werden.
  3. Artefaktintegritätsprüfung: Validierung der Methoden zur Überprüfung der Integrität von Artefakten basierend auf SHA256-Hashes.

2. Testfälle im Detail

2.1. test_container_command_uses_materialized_workspace_without_copy_wrapper

Ziel: Verifizieren, dass der _build_command-Methode bei der Erstellung eines Container-Befehls, der einen Materialized Workspace verwendet, keine unnötigen Kopieroperationen (cp -a) durchführt und die korrekten Container-spezifischen Komponenten (z.B. runsc) verwendet. Setup:

  • Ein temporäres Verzeichnis (work_root) wird erstellt.
  • Ein Python-Skript (main.py) wird in diesem Workspace platziert.
  • Ein WorkerAgent wird mit spezifischen Konfigurationen initialisiert, die auf einen Container-Backend ("container") und einen Docker-Container ("docker") abzielen.

    • Erwartete Ergebnisse (Assertions):

  • Der generierte Befehl (command) muss den Parameter --runtime enthalten.
  • Der generierte Befehl muss den Parameter runsc enthalten.
  • Der generierte Befehl darf den Pfad /workspace-src nicht enthalten.
  • Der generierte Befehl darf den Befehl cp -a nicht enthalten.
  • Es muss bestätigt werden, dass das Zielverzeichnis für den Container-Workspace (runtime_root.parent / "container-workspace" / "main.py") existiert.

2.2. test_container_command_preserves_image_path_and_converts_file_limit

Ziel: Testet die korrekte Verarbeitung von Rust-Projekten innerhalb eines Containers, insbesondere die Beibehaltung des Image-Pfads und die korrekte Konvertierung von Dateigrößenlimits in die Befehlsargumente. Setup:

  • Ein temporäres Verzeichnis wird erstellt, das eine Rust-Projektstruktur (Cargo.toml, src/main.rs) enthält.
  • Der WorkerAgent wird mit Konfigurationen für eine Rust-Runtime und spezifischen Container-Parametern initialisiert, einschließlich container_file_size_limit_kb.

    • Erwartete Ergebnisse (Assertions):

  • Der generierte Befehl muss den Parameter fsize=67108864:67108864 enthalten (was die Konvertierung des Limits in Bytes darstellt).
  • Der generierte Befehl darf den Pfad PATH=/usr/local nicht enthalten.
  • Der generierte Befehl darf den spezifischen Windows-Pfad PATH=C:\\Windows\\System32;C:\\Rust nicht enthalten (was auf eine korrekte Abstraktion hindeutet).

2.3. test_verify_artifact_integrity_rejects_hash_mismatch

Ziel: Validiert, dass die Methode _verify_artifact_integrity korrekt funktioniert, indem sie einen Hash-Mismatch ablehnt. Setup:

  • Ein temporäres Artefakt (payload.zip) wird erstellt.
  • Das Artefakt wird mit einem falschen SHA256-Hash ("0" * 64) versehen.

    • Erwartete Ergebnisse (Assertions):

  • Es wird erwartet, dass die Methode eine RuntimeError wirft, wenn der bereitgestellte Hash mit dem tatsächlichen Hash des Artefakts nicht übereinstimmt.

2.4. test_verify_artifact_integrity_accepts_matching_hash

Ziel: Validiert, dass die Methode _verify_artifact_integrity korrekt funktioniert, wenn der bereitgestellte Hash mit dem tatsächlichen Hash des Artefakts übereinstimmt. Setup:

  • Ein temporäres Artefakt (payload.zip) wird erstellt.
  • Der tatsächliche SHA256-Hash des Artefakts wird berechnet.
  • Die Methode wird mit dem korrekten Hash aufgerufen.

    • Erwartete Ergebnisse (Assertions):

  • Es wird erwartet, dass die Überprüfung erfolgreich ist (kein Fehler wird geworfen), da der Hash übereinstimmt.

3. Zusammenfassung und Implikationen

Diese Testsuite stellt sicher, dass der WorkerAgent folgende kritische Aspekte korrekt implementiert:

  1. Konformität mit Container-Spezifikationen: Der Agent generiert Befehle, die die Anforderungen des Container-Backends (z.B. runsc) erfüllen.
  2. Robustheit bei Pfad- und Limit-Handling: Der Agent kann komplexe Pfade und spezifische Konfigurationsparameter (wie Dateigrößenlimits) korrekt in die ausführbaren Befehle übersetzen, ohne unerwünschte Systemoperationen (wie unnötige Kopierbefehle) einzuführen.
  3. Sicherheit und Zuverlässigkeit: Die Integritätsprüfung von Artefakten ist funktional und zuverlässig, indem sie Hash-Vergleiche korrekt durchführt.

    4. Diese Tests sind entscheidend für die Verlässlichkeit des Agenten bei der Orchestrierung von Workloads in Container-Umgebungen.
