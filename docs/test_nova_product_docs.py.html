Technische Dokumentation: Nova School Produktinformationssystem (Basierend auf Testfällen)

Dieses Dokument beschreibt die Funktionsweise und die erwarteten Ergebnisse des Systems, das durch die bereitgestellten Unit-Tests (test_nova_product_docs.py) validiert wird. Es konzentriert sich auf die Dokumentengenerierung, Indexierung und die Verwaltung von Referenzbibliotheken. ---

1. Überblick

Das System dient dazu, technische Dokumentationen (z.B. für Produkte oder Systeme wie "Nova School") zu verarbeiten, zu indexieren und daraus verschiedene Outputs (z.B. ein generiertes Dokument, ein Index und ein Manifest) zu erstellen. Die Tests validieren insbesondere die folgenden Kernfunktionen:

  1. Dokumenten-Builder: Generierung von Dokumenten, Indexen und Manifesten aus Quell-Dokumenten.
  2. Referenzbibliotheks-Management: Kennzeichnung einer spezifischen Dokumentation als "offiziell lokal" innerhalb einer Referenzbibliothek.

2. Komponenten und Schlüsselkonzepte

2.1. NovaSchoolProductDocsBuilder (Dokumenten-Builder)

Diese Klasse ist verantwortlich für die Verarbeitung von Quell-Dokumenten (source_root) und die Erstellung von Outputs im Zielpfad (pack_root). **Funktionalität (Validiert durch test_builder_generates_index_and_expands_permission_tokens):**

  • Indexgenerierung: Erzeugt eine index.md Datei, die Suchpfade und Dokumentenreferenzen enthält (z.B. /reference?area=nova-school&doc=01_Test.md).
  • Dokumentenerstellung: Generiert ein Hauptdokument (01_Test.md) im Zielpfad, das spezifische Inhalte (z.B. {{PERMISSION_TABLE}}, {{ROLE_DEFAULTS_TABLE}}) enthält.
  • Token-Expansion: Führt die notwendige Expansion von Platzhaltern (wie {{...}}) durch.
  • Manifest-Erstellung: Erstellt eine manifest.json Datei, die Metadaten über die generierten Dokumente speichert (z.B. document_count).

    • Erwartete Ergebnisse (Test-Ergebnisse):

  • Das generierte Dokument enthält spezifische Schlüsselwörter ("project.create") und strukturelle Elemente ("| Recht | Student | Teacher | Admin |").
  • Der Index enthält einen gültigen Suchpfad, der die Quelle und das Dokument identifiziert.
  • Das Manifest spiegelt die Anzahl der verarbeiteten Dokumente korrekt wider.

2.2. ReferenceLibraryService (Referenzbibliotheksdienst)

Dieser Dienst verwaltet die Metadaten und den Status von Dokumenten innerhalb einer definierten Referenzbibliothek. **Funktionalität (Validiert durch test_reference_library_marks_nova_school_as_official_local):**

  • Statuszuweisung: Kann den Status einer bestimmten Dokumentation (hier: "nova-school") auf "official-local" setzen, wenn die entsprechenden Bedingungen erfüllt sind.
  • Dokumentensuche: Kann Dokumente basierend auf Schlüsselwörtern (search) oder Kategorien (container) innerhalb einer bestimmten area finden.
  • Portal-Generierung: Kann auf Basis einer Dokumenten-ID und einer Suchanfrage (query) eine spezifische Ansicht (Portal) generieren, die den Status (official-local) widerspiegelt.

    • Erwartete Ergebnisse (Test-Ergebnisse):

  • Die Kennzeichnung "nova-school" erhält den Status "official-local".
  • Die Anzahl der Dokumente in der Bibliothek ist korrekt (mindestens 2).
  • Spezifische Dokumente (01_System.md) werden erfolgreich durch Such- und Portal-Funktionen gefunden.
  • Die generierte Portal-Ansicht enthält sowohl den Titel der Dokumentation als auch den Status "official-local".

3. Testfälle im Detail

Testfall 1: Dokumenten-Builder (Integrationstest)

Ziel: Überprüfung, ob der NovaSchoolProductDocsBuilder die Quell-Dokumentation korrekt in ein strukturiertes, indexiertes und manifestiertes Output-Format umwandelt. Schlüsselprüfungen:

  • Inhaltliche Korrektheit: Überprüfung, ob spezifische Platzhalter und Strukturmerkmale im Output vorhanden sind.
  • Index-Validität: Überprüfung, ob der generierte Index einen korrekten Pfad enthält.
  • Metadaten-Konsistenz: Überprüfung, ob die Anzahl der Dokumente im Manifest mit der tatsächlichen Anzahl der verarbeiteten Dateien übereinstimmt.

Testfall 2: Referenzbibliotheks-Management (Service-Test)

Ziel: Überprüfung der Fähigkeit des ReferenceLibraryService, die Statusverwaltung und die Dokumenten-Retrieval-Funktionen korrekt durchzuführen. Schlüsselprüfungen:

  • Status-Setzung: Bestätigt, dass die spezifische Dokumentation korrekt als "official-local" markiert wird.
  • Retrieval-Funktionalität: Bestätigt, dass Dokumente erfolgreich über verschiedene Suchkriterien (Themen, Area) abgerufen werden können.
  • Kontextuelle Generierung: Bestätigt, dass die Kombination von Dokument-ID, Area und Suchanfrage zu einer korrekten, kontextsensitiven Portal-Generierung führt.

4. Zusammenfassung und Implikationen

Die Tests deuten darauf hin, dass das System eine robuste Architektur für die Transformation von Rohdaten in strukturierte, indexierbare und kontextabhängige Artefakte besitzt. Die Trennung zwischen der Generierung (Builder) und der Metadaten-/Statusverwaltung (ReferenceLibraryService) ist klar definiert und erfolgreich getestet. Wichtige Abhängigkeiten:

  • Erfolgreiche Implementierung der Token-Erweiterung im Builder.
  • Korrekte Mapping-Logik zwischen Dokumenten-IDs und Referenzbibliotheks-Status.
  • Effiziente Indexierungsmechanismen für die Suchfunktionen.
