Technische Dokumentation: DocumentationCatalog Klasse

1. Einführung

Die Klasse DocumentationCatalog dient dazu, eine zentrale Verwaltung und Interaktion mit einer Sammlung von Dokumentationen zu ermöglichen. Sie ist verantwortlich für die Strukturierung, Erstellung und Abfrage von Dokumenten, die lokal in einem definierten Verzeichnis gespeichert sind.

2. Klasse: DocumentationCatalog

2.1. Beschreibung

Die Klasse DocumentationCatalog initialisiert ein Verzeichnis für die Dokumente und bietet Methoden zur Sicherstellung der Existenz von Dokumenten, zur Auflistung aller verfügbaren Dokumente und zur Abfrage des Inhalts eines spezifischen Dokuments.

2.2. Abhängigkeiten

  • pathlib.Path: Für die Verwaltung von Dateipfaden und Verzeichnissen.
  • templates.OFFLINE_DOCS: Eine externe Struktur (vermutlich ein Dictionary), die die Metadaten und Inhalte der Dokumente definiert.

2.3. Attribute

Attribut Typ Beschreibung
self.docs_path Path Der Pfad zum Verzeichnis, in dem die Dokumentendateien gespeichert werden.

2.4. Methoden

2.4.1. __init__(self, docs_path: Path) -> None

Zweck: Initialisiert den Katalog und stellt sicher, dass das Zielverzeichnis für die Dokumente existiert. Parameter:

  • docs_path: Path: Der Pfad zum Verzeichnis, in dem die Dokumente abgelegt werden sollen.

    • Ablauf:

  1. Speichert den bereitgestellten docs_path als Attribut.
  2. Verwendet self.docs_path.mkdir(parents=True, exist_ok=True), um das Verzeichnis zu erstellen.
  • parents=True: Erstellt notwendige übergeordnete Verzeichnisse.
  • exist_ok=True: Verhindert einen Fehler, falls das Verzeichnis bereits existiert.

2.4.2. ensure_seed_docs(self) -> None

Zweck: Stellt sicher, dass alle Dokumente, die in der OFFLINE_DOCS-Konfiguration definiert sind, physisch als Markdown-Dateien (.md) im Speicherdirektor existieren. Ablauf:

  1. Iteriert über alle Schlüssel-Wert-Paare (slug, payload) im globalen Dictionary OFFLINE_DOCS.
  2. Für jeden Eintrag konstruiert den Zielpfad (target) für die Datei (z.B. slug.md).
  3. Prüfung: Wenn die Datei bereits existiert (target.exists()), wird übersprungen.
  4. Erstellung: Wenn die Datei nicht existiert, wird der Inhalt (payload["content"]) in die Datei geschrieben, wobei die Zeichenkodierung utf-8 verwendet wird.

2.4.3. list_docs(self) -> list[dict[str, object]]

Zweck: Gibt eine Liste aller definierten Dokumente zurück, zusammen mit ihren Metadaten und Pfaden. Ablauf:

  1. Erzeugt eine Liste von Dictionaries.
  2. Iteriert über alle Einträge in OFFLINE_DOCS.
  3. Für jeden Eintrag erstellt ein Dictionary, das folgende Informationen enthält:
  • slug: Der eindeutige Schlüssel.
  • title: Der Titel aus dem Payload.
  • tags: Eine Liste der Tags, falls vorhanden.
  • path: Der vollständige Pfad zur Markdown-Datei.

    • Rückgabe: Eine Liste von Dictionaries, wobei jedes Dictionary eine strukturierte Zusammenfassung eines Dokuments darstellt.

2.4.4. get_doc(self, slug: str) -> dict[str, object]

Zweck: Lese und gibt den vollständigen Inhalt eines spezifischen Dokuments basierend auf seinem Slug zurück. Parameter:

  • slug: str: Der eindeutige Slug des gesuchten Dokuments.

    • Ablauf:

  1. Existenzprüfung: Prüft, ob der slug in der OFFLINE_DOCS-Konfiguration existiert.
  • Fehlerbehandlung: Wenn der Slug nicht gefunden wird, wirft die Ausnahme FileNotFoundError mit einer spezifischen Fehlermeldung.
  1. Pfadkonstruktion: Konstruiert den vollständigen Dateipfad (path) für die gesuchte Datei.
  2. Inhaltsabruf: Liest den gesamten Textinhalt der Datei (path.read_text(encoding="utf-8")).
  3. Rückgabe: Gibt ein Dictionary zurück, das folgende Informationen enthält:
  • slug: Der Slug.
  • title: Der Titel.
  • tags: Die Liste der Tags.
  • content: Der vollständige Textinhalt der Markdown-Datei.

3. Verwendung (Beispiel)

from pathlib import Path
# Angenommen, OFFLINE_DOCS ist definiert und die Klasse DocumentationCatalog ist importiert
# 1. Initialisierung
DOCS_DIRECTORY = Path("./documentation_files")
catalog = DocumentationCatalog(docs_path=DOCS_DIRECTORY)
# 2. Sicherstellen, dass alle Dateien existieren (falls sie fehlen)
catalog.ensure_seed_docs()
# 3. Auflisten aller verfügbaren Dokumente
print("--- Verfügbare Dokumente ---")
docs_list = catalog.list_docs()
for doc in docs_list:
print(f"Slug: {doc['slug']}, Titel: {doc['title']}, Pfad: {doc['path']}")
# 4. Ein spezifisches Dokument abrufen
try:
specific_doc = catalog.get_doc("introduction")
print("\n--- Inhalt des Dokuments 'introduction' ---")
print(f"Titel: {specific_doc['title']}")
print(f"Tags: {specific_doc['tags']}")
print("--- Inhalt ---")
print(specific_doc['content'][:500] + "...") # Zeigt die ersten 500 Zeichen
except FileNotFoundError as e:
print(f"Fehler beim Abrufen: {e}")

4. Fehlerbehandlung

Ausnahme Ursache Wann wird sie ausgelöst?
FileNotFoundError Der angeforderte slug existiert nicht in der OFFLINE_DOCS-Konfiguration. In der Methode get_doc(self, slug: str).

5. Design-Überlegungen

  • Separation von Sorgen: Die Klasse trennt die Logik der Dateisystemoperationen (Erstellen, Lesen, Listen) von der Definition der Dokumentenstruktur (OFFLINE_DOCS).
  • Pathlib-Nutzung: Die Verwendung von pathlib.Path sorgt für eine plattformübergreifende und leserfreundliche Handhabung von Dateipfaden.
  • Immutabilität der Metadaten: Die Metadaten (Titel, Tags) werden bei der Abfrage aus der zentralen Konfiguration (OFFLINE_DOCS) abgerufen, während der eigentliche Inhalt dynamisch aus dem Dateisystem gelesen wird.
