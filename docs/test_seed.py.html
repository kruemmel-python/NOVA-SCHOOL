Technische Dokumentation: test_seed.py

1. Übersicht

Diese Datei enthält eine Testklasse SeedBootstrapTests, die die Funktionalität des Seed-Bootstrap-Prozesses in der Anwendung validiert. Der Fokus liegt darauf, sicherzustellen, dass der Bootstrap-Prozess Benutzer, Gruppen und deren Berechtigungen korrekt initialisiert oder zurückgesetzt werden, selbst wenn vorherige, möglicherweise fehlerhafte oder überlagernde Einstellungen existieren.

2. Zielsetzung der Tests

Das Hauptziel dieser Testsuite ist es zu verifizieren, dass die Funktion bootstrap_application (die in der Testmethode aufgerufen wird) folgende Zustände erzwingt:

  1. Benutzer-Status: Benutzer, die zuvor mit spezifischen Rollen und Statusen erstellt wurden, werden auf den Standardstatus "active" gesetzt.
  2. Benutzer-Display-Namen: Spezifische Display-Namen werden auf vordefinierte Werte ("Lehrkraft", "Schueler Demo") gesetzt.
  3. Benutzer-Berechtigungen: Die spezifischen Berechtigungen, die zuvor dem Benutzer zugewiesen wurden, werden auf ein leeres Set ({}) zurückgesetzt.
  4. Gruppen-Berechtigungen: Die Berechtigungen einer Gruppe werden auf ein spezifisches, vordefiniertes Set ({"workspace.group": True, "chat.use": True, "docs.read": True}) gesetzt.

3. Code-Analyse

3.1. Klasse: SeedBootstrapTests

Diese Klasse implementiert die Testlogik und nutzt das unittest-Framework.

3.1.1. setUp(self) -> None

Die setUp-Methode wird vor jeder Testaufgabe ausgeführt und dient dazu, alle notwendigen Abhängigkeiten und Ressourcen für die Tests einzurichten. Aktion:

  • Temporäre Umgebung: Ein temporäres Verzeichnis (self.tmp) wird erstellt (tempfile.TemporaryDirectory).
  • Konfiguration: Eine ServerConfig wird aus dem temporären Pfad geladen.
  • Repository: Eine Instanz von SchoolRepository wird mit dem Datenbankpfad der Konfiguration initialisiert.
  • Sicherheit: Eine Instanz von EmbeddedSecurityPlane wird mit dem temporären Pfad initialisiert.
  • Authentifizierung: Eine Instanz von AuthService wird mit dem Repository, der Sicherheit, dem Tenant-ID und der Session-TTL initialisiert.
  • Dokumentation: Eine Instanz von DocumentationCatalog wird mit dem Dokumentationspfad initialisiert.
  • Workspace Management: Eine Instanz von WorkspaceManager wird mit der Konfiguration initialisiert.

3.1.2. tearDown(self) -> None

Die tearDown-Methode wird nach jedem Testaufruf ausgeführt und sorgt für die saubere Bereinigung der Ressourcen. Aktion:

  • Repository Schließen: self.repository.close() wird aufgerufen.
  • Sicherheit Schließen: self.security.close() wird aufgerufen.
  • Temporäre Umgebung Bereinigen: self.tmp.cleanup() wird aufgerufen, um das temporäre Verzeichnis zu löschen.

3.1.3. Testmethode: test_bootstrap_resets_demo_accounts_to_role_defaults_even_when_old_overrides_exist

Diese Methode simuliert einen Szenario, in dem Benutzerdaten und Gruppen existieren, die möglicherweise nicht den gewünschten Bootstrap-Zuständen entsprechen, und prüft, ob der Bootstrap-Prozess diese korrigiert. Schritte im Test:

  1. Dateninitialisierung (Setup-Daten):
  • Benutzer "teacher": Ein Benutzer wird erstellt mit spezifischen Rollen, Berechtigungen und dem Status "suspended".
  • Benutzer "student": Ein Benutzer wird erstellt mit spezifischen Rollen, Berechtigungen und dem Status "inactive".
  • Gruppe "class-1a": Eine Gruppe wird erstellt mit spezifischen Berechtigungen.
  1. Bootstrap-Ausführung:
  • Die Funktion bootstrap_application wird mit den initialisierten Abhängigkeiten (self.repository, self.auth, self.docs, self.workspace) aufgerufen, um den Bootstrap-Prozess zu starten.
  1. Validierung (Assertions):
  • Existenzprüfung: Es wird sichergestellt, dass die erstellten Benutzer (teacher, student) und die Gruppe (class-1a) im Repository existieren (assertIsNotNone).
  • Statusprüfung: Der Status des Lehrers wird auf "active" überprüft. Der Status des Studenten wird auf "active" überprüft.
  • Display-Name-Prüfung: Die Display-Namen werden auf die erwarteten Werte ("Lehrkraft", "Schueler Demo") überprüft.
  • Berechtigungsprüfung (Benutzer): Die Berechtigungssets der Benutzer werden auf leere Sets ({}) überprüft.
  • Berechtigungsprüfung (Gruppe): Die Berechtigungsset der Gruppe wird auf das erwartete Set ({"workspace.group": True, "chat.use": True, "docs.read": True}) überprüft.

4. Zusammenfassung und Implikationen

Dieser Test stellt sicher, dass der Bootstrap-Mechanismus eine konsistente und definierte Ausgangsbasis für Benutzer- und Gruppenkonfigurationen schafft. Er validiert die Robustheit des Systems, indem er prüft, ob ältere oder fehlerhafte Konfigurationen durch den Bootstrap-Prozess überschrieben werden, um auf die definierten Standardwerte zurückzukehren. Schlüsselkomponenten, die getestet werden:

  • Datenpersistenz (Erstellung/Abruf von Benutzern und Gruppen).
  • Status-Management (Übersetzung von initialen Statuscodes).
  • Berechtigungs-Reset-Logik (Wiederherstellung von Standardberechtigungen).
  • Konfigurations-Injection (Überprüfung, ob die Bootstrap-Funktion die korrekten Werte in die Datenbank schreibt).
