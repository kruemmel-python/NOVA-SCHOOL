Technische Dokumentation: test_nova_bridge.py

Version: 1.0 Datum: 2023-10-27 Autor: [Ihr Name/Team] Beschreibung: Diese Datei dient als Test- und Setup-Skript für die Interaktion mit der nova_bridge-Komponente des Nova School Servers. Sie beinhaltet die notwendigen Mechanismen zur temporären Erstellung von Dateien und die Durchführung von Unit-Tests, um die Funktionalität der Bridge zu validieren.

1. Überblick

Das Hauptziel dieser Datei ist es, die Integration und die korrekte Initialisierung der nova_bridge zu testen. Sie nutzt das unittest-Framework und das tempfile-Modul, um sicherzustellen, dass die Bridge in einem isolierten und temporären Kontext korrekt funktioniert.

2. Abhängigkeiten

  • nova_school_server.nova_bridge: Die zentrale Schnittstelle, die die Logik für die Kommunikation mit dem Nova School Server bereitstellt.
  • tempfile: Für die Erstellung temporärer Dateien.
  • unittest: Für die Durchführung von automatisierten Tests.
  • pathlib.Path: Für die moderne und objektorientierte Dateimanagement-Manipulation.

Basierend auf dem Import von load_nova_bridge lässt sich ableiten, dass die Teststrategie folgende Schritte umfassen wird:

  1. Initialisierung: Die Funktion load_nova_bridge wird aufgerufen, um eine Instanz oder einen Kontext der Bridge zu laden.
  2. Isolation: Verwendung von tempfile zur Sicherstellung, dass Testdaten oder temporäre Ressourcen sauber nach dem Test entfernt werden.
  3. Validierung: Durchführung von Unit-Tests, um zu prüfen, ob die Bridge die erwarteten Operationen (z.B. Datenübertragung, Konfiguration) korrekt ausführt.

Technische Dokumentation: Testsuite für die Embedded Nova Bridge

1. Überblick

Diese Dokumentation beschreibt die Testsuite EmbeddedNovaBridgeTests, die zur Validierung der Funktionalität der Embedded Nova Bridge dient. Der Fokus liegt auf der Überprüfung, dass die Bridge ohne externe Abhängigkeiten korrekt Token-Flows und Tool-Sandbox-Autorisierungen handhabt.

2. Zielsetzung der Tests

Die Hauptziele dieser Tests sind:

  1. Token-Flow-Validierung: Sicherstellen, dass die Bridge Token-Generierung und Authentifizierung korrekt durchführt, selbst wenn keine externen Abhängigkeiten vorhanden sind.
  2. Tool-Sandbox-Autorisierung: Verifizieren, dass die Tool-Sandbox korrekt die Berechtigungen (Autorisierung) für spezifische Werkzeuge (Tools) basierend auf Benutzerrollen und Projektkontext gewährt.
  3. Architektur-Verifizierung: Bestätigen, dass die Testumgebung korrekt als "embedded" (eingebettet) konfiguriert ist.

3. Testfälle im Detail

Die Testsuite besteht aus zwei Hauptmethoden, die jeweils spezifische Aspekte der Embedded Nova Bridge abdecken.

3.1. Testfall: test_embedded_bridge_handles_token_flow_without_external_dependency

Beschreibung: Dieser Test validiert den vollständigen Token-Generierungs- und Authentifizierungszyklus der Bridge, wobei explizit sichergestellt wird, dass keine externen Bibliotheken oder Dienste benötigt werden. Schritte:

  1. Setup: Ein temporäres Verzeichnis wird erstellt (tempfile.TemporaryDirectory()).
  2. Bridge Initialisierung: Die Funktion load_nova_bridge() wird verwendet, um eine Instanz der Bridge zu laden.
  3. Security Plane Initialisierung: Eine SecurityPlane wird mit dem temporären Verzeichnis initialisiert.
  4. Tenant Registrierung: Ein Tenant (nova-school) wird registriert.
  5. Token-Generierung: Ein Token wird mithilfe von security.issue_token() für den Benutzer "teacher" mit den Rollen {"teacher"} und einer Lebensdauer von 3600 Sekunden generiert.
  6. Authentifizierung: Der generierte Token wird über security.authenticate() validiert.

    7. Erwartete Ergebnisse (Assertions):

  • Erfolg: Der Authentifizierungsaufruf muss erfolgreich einen principal (Benutzerobjekt) zurückgeben (self.assertIsNotNone(principal)).
  • Rollenprüfung: Das subject des erstellten Principals muss "teacher" sein (self.assertEqual(principal.subject, "teacher")).
  • Token-ID-Konsistenz: Die token_id des Principals muss exakt mit der token_id des generierten Tokens übereinstimmen.
  • Backend-Verifizierung: Die Snapshot-Information der Sandbox muss bestätigen, dass das Backend als "embedded" konfiguriert ist (self.assertEqual(security.snapshot()["backend"], "embedded")).

3.2. Testfall: test_embedded_tool_sandbox_authorizes_requests

Beschreibung: Dieser Test prüft die Logik der ToolSandbox zur Entscheidungsfindung, ob ein Benutzer für die Ausführung spezifischer Werkzeuge autorisiert ist. Schritte:

  1. Bridge Initialisierung: Eine Instanz der Bridge wird geladen.
  2. Sandbox Initialisierung: Die ToolSandbox wird instanziiert.
  3. Autorisierungsanfrage: Eine authorize()-Methode wird aufgerufen mit folgenden Parametern:
  • Benutzer: "user:teacher"
  • Erlaubte Tools: {"run.python", "ai.use"}
  • Angefragte Tools: {"run.python"}
  • Metadaten: {"project_id": "abc"}

    • Erwartete Ergebnisse (Assertions):

  • Autorisierung: Die Methode muss True zurückgeben, was bedeutet, dass die Anfrage autorisiert wurde (self.assertTrue(session["authorized"])).
  • Angefragte Tools: Das angefragte Tool ("run.python") muss in der Liste der angefragten Tools enthalten sein (self.assertIn("run.python", session["requested_tools"])).
  • Backend-Verifizierung: Die Snapshot-Information der Sandbox muss bestätigen, dass das Backend als "embedded" konfiguriert ist (self.assertEqual(sandbox.snapshot()["backend"], "embedded")).

4. Voraussetzungen

Um diese Tests ausführen zu können, sind folgende Komponenten und Bibliotheken erforderlich:

  • Die Implementierung der NovaBridge und deren Unterklassen (z.B. SecurityPlane, ToolSandbox).
  • Die Funktion load_nova_bridge(), die die Bridge-Instanz initialisiert.
  • Standard-Python-Bibliotheken (unittest, tempfile, pathlib).

5. Zusammenfassung

Die Testsuite stellt sicher, dass die Embedded Nova Bridge sowohl die grundlegende Authentifizierungslogik (Token-Flow) als auch die granulare Zugriffskontrolle (Tool-Sandbox) korrekt implementiert und konfiguriert, insbesondere im Kontext einer "embedded"-Umgebung.
