Technische Dokumentation: release_notes.py

1. Überblick

Diese Datei (release_notes.py) definiert die Datenstrukturen (Dataclasses) und die Logik zur Verwaltung von Versions- und Commit-Historien für Software-Releases. Sie dient dazu, Metadaten über Commits und veröffentlichte Versionen zu strukturieren und zu speichern.

2. Struktur und Komponenten

Die Datei enthält drei Haupt-Dataclasses, die die verschiedenen Entitäten der Versionsverwaltung modellieren, sowie eine globale Konfiguration für Kategorien.

2.1. Dataclasses

2.1.1. ReleaseCommit

Diese Klasse repräsentiert einen einzelnen Git-Commit und speichert relevante Metadaten. Attribute:

  • full_hash: str: Der vollständige Git-Commit-Hash.
  • short_hash: str: Eine verkürzte Version des Commit-Hashes.
  • subject: str: Die Beschreibung des Commits.
  • date: str: Das Datum des Commits (im String-Format).
  • category: str: Die Klassifizierung des Commits (z.B. "Release", "Fixed", "Changed").

    • Merkmale:

  • Verwendet slots=True für eine effizientere Speicherung.

2.1.2. ReleaseVersion

Diese Klasse fasst eine Reihe von Commits zusammen, die zu einer spezifischen veröffentlichten Version gehören. Attribute:

  • tag: str: Der Versions-Tag (z.B. "v1.0.0").
  • date: str: Das Datum der Veröffentlichung dieser Version.
  • commits: list[ReleaseCommit]: Eine Liste aller ReleaseCommit-Instanzen, die zu dieser Version gehören.

    • Merkmale:

  • Verwendet slots=True.

2.1.3. ReleaseHistory

Diese Klasse dient als Container für die gesamte Historie des Projekts, trennt zwischen unpublizierten Änderungen und veröffentlichten Versionen. Attribute:

  • unreleased: list[ReleaseCommit]: Eine Liste aller Commits, die noch nicht in einer offiziellen Version verpackt wurden.
  • releases: list[ReleaseVersion]: Eine Liste aller offiziell veröffentlichten Versionsobjekte.

    • Merkmale:

  • Verwendet slots=True.

2.2. Konfiguration

_CATEGORY_RULES

Dies ist eine globale Konstante, die die Regeln definiert, wie Commits automatisch in Kategorien eingeordnet werden können. Typ: tuple[tuple[str, tuple[str, ...]], ...] Definition: Eine Tupel-Liste von Tupeln. Jedes Tupel definiert eine Kategorie und eine Liste von Schlüsselwörtern (Strings), die als Indikatoren für diese Kategorie dienen. Beispiel:

_CATEGORY_RULES = (
("Release", ("release", "publish")),
("Added", ("add", "create", "introduce", "implement", "build")),
("Fixed", ("fix", "correct", "repair", "harden", "normalize", "resolve")),
("Changed", ("update", "change", "adjust", "refactor", "rename", "extend")),
)

Funktion: Diese Struktur wird implizit für die Klassifizierung von ReleaseCommit.category verwendet (obwohl die eigentliche Klassifizierungslogik in diesem Ausschnitt nicht vollständig implementiert ist, deutet die Struktur darauf hin).

3. Funktionen

Der bereitgestellte Code enthält keine explizit definierten Funktionen, außer der Definition der Datenstrukturen und der Konfiguration. Hinweis: Die Zeile def <|end|> deutet darauf hin, dass hier die Implementierung der Hauptlogik (z.B. Funktionen zur Verarbeitung von Commits, zur Erstellung von Releases oder zur Klassifizierung) folgen wird.

4. Zusammenfassung der Nutzung

Die Datei bietet eine robuste, datenorientierte Methode, um die Entwicklungshistorie (Commits) und die daraus resultierenden Versionen (Releases) strukturiert zu speichern und zu verwalten. Die Verwendung von dataclasses sorgt für eine klare, wartbare und typisierte Datenstruktur.

Technische Dokumentation: release_notes.py

Übersicht

Die Datei release_notes.py enthält Funktionen zur Generierung einer Versionshistorie basierend auf Git-Tags und Commits in einem angegebenen Pfad. Sie dient dazu, eine strukturierte Übersicht über frühere Releases und den aktuellen unreleaseden Zustand zu erstellen.

Funktionen

1. build_release_history(base_path: Path) -> ReleaseHistory

Diese Hauptfunktion konstruiert die vollständige Versionshistorie, einschließlich vergangener Releases und des unreleaseden Bereichs. Beschreibung: Sie nimmt einen Pfad zu einem Git-Repository (base_path) entgegen und generiert eine ReleaseHistory-Instanz. Diese Instanz enthält eine Liste der offiziellen Releases (basierend auf Git-Tags) und eine Liste der Commits im unreleaseden Bereich. Parameter:

  • base_path: Path: Der absolute oder relativ zu dem Skript definierte Pfad zum Git-Repository.

    • Rückgabe:

  • ReleaseHistory: Eine Klasse (nicht definiert in diesem Ausschnitt, aber impliziert), die die Versionshistorie enthält.

    • Schritte:

  1. Pfad auflösen: Der bereitgestellte base_path wird aufgelöst (resolve(strict=False)).
  2. Tags extrahieren: Ruft die Funktion list_git_tags auf, um alle Git-Tags im Repository zu finden.
  3. Releases erstellen: Iteriert über die gefundenen Tags, um für jedes Tag eine ReleaseVersion zu erstellen.
  • Für jedes Tag wird der Revisionenbereich (previous_tag..tag oder nur tag für das erste Tag) berechnet.
  • Die Commits innerhalb dieses Bereichs werden über list_git_commits abgerufen.
  • Eine ReleaseVersion wird mit dem Tag, dem Datum und den zugehörigen Commits erstellt und in die Liste releases hinzugefügt.
  1. Unreleased-Bereich bestimmen: Identifiziert das neueste Tag und definiert den unreleaseden Bereich als latest_tag..HEAD.
  2. Ergebnis zurückgeben: Gibt eine ReleaseHistory zurück, die den unreleaseden Bereich und die Liste der Releases enthält.

    3. ---

2. list_git_tags(base_path: Path) -> list[dict[str, str]]

Diese Funktion führt einen Git-Befehl aus, um alle Git-Tags im Repository zu extrahieren. Beschreibung: Sie führt den Git-Befehl git for-each-ref --sort=-creatordate --format=%(refname:strip=2)|%(creatordate:short) refs/tags aus, um eine Liste von Tag-Namen und deren Erstellungsdaten zu erhalten. Parameter:

  • base_path: Path: Der Pfad zum Git-Repository.

    • Rückgabe:

  • list[dict[str, str]]: Eine Liste von Dictionaries, wobei jedes Dictionary den Schlüssel "tag" (der Tag-Name) und den Schlüssel "date" (das Datum der Erstellung) enthält.

    • Schritte:

  1. Git-Befehl ausführen: Führt den spezifischen Git-Befehl aus, um die Tag-Informationen zu generieren.
  2. Ergebnis verarbeiten: Parset die Ausgabe des Git-Befehls Zeile für Zeile.
  3. Daten extrahieren: Trennt den Tag-Namen und das Datum anhand des Trennzeichens (|) und speichert sie als Dictionary in der tags-Liste.
  4. Rückgabe: Gibt die final bereinigte Liste der Tag-Informationen zurück.

    5. ---

Abhängigkeiten und Konzepte

  • **Path (von pathlib):** Wird verwendet, um Dateipfade im Betriebssystem-unabhängig zu verwalten.
  • **ReleaseHistory und ReleaseVersion:** Diese Klassen sind impliziert und dienen der Strukturierung der Ergebnisse.
  • **_run_git(...):** Eine interne oder externe Funktion, die den eigentlichen Git-Befehl ausführt und die Rohausgabe (String) zurückgibt.
  • **list_git_commits(...):** Eine interne Funktion, die Commits innerhalb eines angegebenen Revisionenbereichs abruft.

Zusammenfassung des Datenflusses

  1. **build_release_history** startet den Prozess.
  2. Sie ruft **list_git_tags** auf, um die Struktur der Releases zu ermitteln.
  3. Basierend auf den Tags erstellt sie detaillierte **ReleaseVersion**-Objekte, indem sie Commits im jeweiligen Revisionenbereich abruft.
  4. Sie definiert den unreleaseden Bereich (latest_tag..HEAD).
  5. Sie fasst alle Ergebnisse in die finale **ReleaseHistory** zusammen.

Technische Dokumentation: release_notes.py

Übersicht

Diese Datei (release_notes.py) enthält Funktionen zur automatisierten Extraktion und Kategorisierung von Git-Commits, um diese in ein strukturiertes Format für Release Notes zu konvertieren. Die Hauptfunktionen umfassen das Abrufen von Git-Historie und die Klassifizierung der Commit-Themen.

Funktionen

1. list_git_commits(base_path: Path, revision_range: str) -> list[ReleaseCommit]

Diese Funktion führt den Git-Log aus einem angegebenen Pfad mit einer bestimmten Revisionspanage aus und parst die Ausgabe, um eine Liste von ReleaseCommit-Objekten zu erstellen. Parameter:

  • **base_path: Path**: Der lokale Pfad zum Git-Repository, aus dem die Commits abgerufen werden sollen.
  • **revision_range: str**: Der Git-Revisionspanage (z.B. HEAD~1, v1.0..HEAD), die definiert, welche Commits abgerufen werden sollen.

    • Beschreibung:

  1. Rufen die interne Funktion _run_git auf, um die Git-Historie basierend auf base_path und revision_range abzurufen.
  2. Die Git-Ausgabe wird in eine Liste von Zeilen aufgeteilt.
  3. Jede Zeile wird analysiert, um den vollständigen Hash, den kurzen Hash, das Datum und den Commit-Titel zu extrahieren.
  4. Ein ReleaseCommit-Objekt wird für jeden Commit erstellt, wobei der Commit-Titel mithilfe der Funktion categorize_commit_subject in eine Kategorie klassifiziert wird.
  5. Die Liste der erstellten ReleaseCommit-Objekte wird zurückgegeben.

    6. Rückgabe:

  • Eine Liste von Objekten vom Typ list[ReleaseCommit].

    • Abhängigkeiten:

  • _run_git(): Eine interne Funktion, die die eigentliche Git-Befehlsausführung übernimmt.
  • ReleaseCommit: Eine Klasse, die die Struktur für ein einzelnes Commit darstellt.
  • categorize_commit_subject(): Die Funktion zur Themenklassifizierung.

    • ---

2. categorize_commit_subject(subject: str) -> str

Diese Funktion analysiert den Titel eines Git-Commits und ordnet ihn einer vordefinierten Kategorie zu. Parameter:

  • **subject: str**: Der Titel des Git-Commits, der analysiert werden soll.

    • Beschreibung:

  1. Normalisierung: Der eingehende subject wird bereinigt:
  • Alle Zeichen, die nicht alphanumerisch sind, werden durch Leerzeichen ersetzt.
  • Der String wird kleingeschrieben (.lower()).
  • Überflüssige Leerzeichen werden entfernt.
  1. Erste Wort-Prüfung: Das erste Wort der normalisierten Zeichenkette wird extrahiert.
  2. Kategorie-Prüfung (Erste Iteration): Die Funktion durchläuft die globale Konfiguration _CATEGORY_RULES. Wenn das erste Wort mit einem definierten Schlüsselwort in einer Kategorie übereinstimmt, wird diese Kategorie zurückgegeben.
  3. Kategorie-Prüfung (Zweite Iteration): Falls die erste Prüfung fehlschlägt, wird erneut durch die _CATEGORY_RULES iteriert. Wenn irgendein Schlüsselwort aus einer Kategorie im gesamten normalisierten String vorkommt, wird diese Kategorie zurückgegeben.
  4. Standardfall: Wenn keine der definierten Regeln zutrifft, wird die Kategorie "Changed" zurückgegeben.

    5. Rückgabe:

  • Ein String, der die zugewiesene Kategorie darstellt (z.B. "Feature", "Bugfix", oder "Changed").

    • Abhängigkeiten:

  • _CATEGORY_RULES: Eine globale Struktur (wahrscheinlich ein Dictionary oder eine Liste von Tupeln), die die Regeln für die Commit-Kategorisierung definiert.
  • re: Das Python-RegExp-Modul zur Zeichenkettenmanipulation.

    • ---

Strukturierte Datenmodelle (Impliziert)

  • **ReleaseCommit**: Eine Klasse, die die Metadaten eines Git-Commits speichert (z.B. full_hash, short_hash, subject, date, category).
  • **_CATEGORY_RULES**: Eine globale Konfiguration, die die Mapping-Regeln zwischen Commit-Themen und Kategorien definiert.

Zusammenfassung des Workflows

Der Prozess beginnt mit der Definition eines Git-Bereichs (revision_range) und der Suche nach den relevanten Commits (list_git_commits). Anschließend wird jeder identifizierte Commit analysiert, um seine Beschreibung zu extrahieren und ihn mithilfe von categorize_commit_subject in eine sinnvolle Kategorie einzuordnen, was die Grundlage für die Erstellung der Release Notes bildet.

Technische Dokumentation: release_notes.py

Übersicht

Die Datei release_notes.py enthält Funktionen zur automatischen Erstellung und Generierung von Versions-Changelogs basierend auf der Git-Historie eines Projekts. Sie nutzt eine Klasse ReleaseHistory (impliziert), um die notwendigen Daten zu verarbeiten und formatierte Markdown-Texte zu erzeugen.

Funktionen

1. render_changelog(history: ReleaseHistory) -> str

Beschreibung: Diese Funktion generiert das vollständige Changelog-Dokument. Sie strukturiert das Dokument in Abschnitte für "Unreleased" und für jede veröffentlichte Version. Parameter:

  • history: ReleaseHistory: Eine Instanz der Klasse ReleaseHistory, die die gesamte Git-Historie enthält.

    • Rückgabe:

  • str: Der formatierte Markdown-String des gesamten Changelogs.

    • Logik:

  1. Initialisiert eine Liste von Zeilen für den Header und den "Unreleased"-Abschnitt.
  2. Fügt die Ergebnisse der Funktion _render_commit_groups für die "Unreleased"-Phase hinzu.
  3. Iteriert über alle releases in history.releases.
  4. Für jede Veröffentlichung fügt sie einen Abschnitt für den Tag und das Datum hinzu.
  5. Fügt die Ergebnisse der Funktion _render_commit_groups für die Commits dieser spezifischen Veröffentlichung hinzu.
  6. Fügt alle generierten Zeilen zusammen und gibt sie als zusammenhängenden String zurück.

2. render_release_notes(history: ReleaseHistory, tag: str) -> str

Beschreibung: Diese Funktion generiert die spezifischen Notizen für eine einzelne, gegebene Versions-Tag. Parameter:

  • history: ReleaseHistory: Eine Instanz der Klasse ReleaseHistory.
  • tag: str: Der spezifische Tag (z.B. v1.0.0), für den die Notizen generiert werden sollen.

    • Rückgabe:

  • str: Der formatierte Markdown-String für die Release-Notizen.

    • Logik:

  1. Sucht in history.releases nach einem Eintrag, dessen tag mit dem übergebenen tag übereinstimmt.
  2. Fehlerbehandlung: Wenn kein passender Tag gefunden wird, wirft sie eine ValueError mit der Fehlermeldung "unknown tag: {tag}".
  3. Initialisiert die Zeilen für den Release-Header (Tag und Datum).
  4. Fügt die Ergebnisse der Funktion _render_commit_groups für die Commits dieser spezifischen Veröffentlichung hinzu.
  5. Gibt den zusammengefassten String zurück.

3. write_changelog(base_path: Path, target_path: Path | None = None) -> Path

Beschreibung: Diese Hauptfunktion orchestriert den gesamten Prozess: baut die Historie, rendert das Changelog und schreibt das Ergebnis in eine Datei. Parameter:

  • base_path: Path: Der Pfad zum Git-Repository oder zum Verzeichnis, aus dem die Historie abgeleitet werden soll.
  • target_path: Path | None = None: Optionaler Zielpfad für die Ausgabe. Wenn None, wird automatisch CHANGELOG.md im base_path erstellt.

    • Rückgabe:

  • Path: Der vollständige Pfad zur erstellten oder überschriebenen Changelog-Datei.

    • Logik:

  1. Pfadauflösung: Stellt sicher, dass base_path aufgelöst wird (resolve(strict=False)).
  2. Historien-Erstellung: Ruft build_release_history(base_path) auf, um die ReleaseHistory-Instanz zu erstellen.
  3. Ausgabe-Pfad-Bestimmung: Bestimmt den output_path. Wenn target_path angegeben ist, wird dieser verwendet; andernfalls wird base_path / "CHANGELOG.md" verwendet.
  4. Rendern: Ruft render_changelog(history) auf, um den gesamten Inhalt zu generieren.
  5. Schreiben: Schreibt den generierten String in die output_path mit der Kodierung "utf-8".
  6. Gibt den finalen output_path zurück.

    7. ---

Abhängigkeiten und Annahmen

  • **ReleaseHistory:** Eine Klasse, die die Struktur der Git-Historie (Unreleased, Releases, Commits) hält.
  • **_render_commit_groups(commits: list) -> list[str]:** Eine interne, nicht gezeigte Funktion, die die einzelnen Commits einer Version in eine formatierte Liste von Markdown-Zeilen umwandelt.
  • **build_release_history(base_path: Path) -> ReleaseHistory:** Eine externe Funktion, die die Git-Historie basierend auf dem gegebenen Pfad konstruiert.
  • **Path:** Ein Typ aus der pathlib-Bibliothek, der für Pfadmanipulation verwendet wird.

Nutzungsszenario

Um das Changelog zu erstellen, wird write_changelog aufgerufen:

from pathlib import Path
# Angenommen, ReleaseHistory und build_release_history sind definiert
from release_notes import write_changelog
repo_root = Path("/path/to/your/project")
output_file = Path("/path/to/output/CHANGELOG.md")
final_path = write_changelog(repo_root, target_path=output_file)
print(f"Changelog erfolgreich erstellt unter: {final_path}")

Technische Dokumentation: release_notes.py

Übersicht

Die Datei release_notes.py enthält Funktionen, die für die automatische Erstellung von Versionshinweisen (Release Notes) basierend auf Git-Commit-Historien zuständig sind. Die Hauptfunktion write_release_notes orchestriert den gesamten Prozess: das Sammeln der Git-Historie, das Rendern der Commit-Gruppen und das Schreiben der finalen Notizen in eine Zieldatei.

Funktionen

write_release_notes(base_path: Path, tag: str, target_path: Path) -> Path

Diese Hauptfunktion ist der zentrale Einstiegspunkt für die Erstellung der Release Notes. Zweck: Löst den gesamten Workflow aus:

  1. Bereinigt den Basis-Pfad.
  2. Ruft die Funktion build_release_history auf, um die notwendige Git-Historie zu erstellen.
  3. Bereinigt den Zielpfad.
  4. Ruft die Funktion render_release_notes auf, um die generierten Daten in einen Textstring umzuwandeln.
  5. Schreibt den resultierenden Text in die Zieldatei.

    6. Parameter:

  • **base_path: Path**: Der Pfad zum Git-Repository (oder zum relevanten Verzeichnis), aus dem die Commit-Historie abgerufen werden soll.
  • **tag: str**: Der Git-Tag, der für die aktuelle Release-Version verwendet wird (wird in den Release Notes verwendet).
  • **target_path: Path**: Der vollständige Pfad, an dem die generierten Release Notes gespeichert werden sollen.

    • Rückgabe:

  • **Path**: Der vollständige Pfad zur erfolgreich erstellten Release Notes Datei.

    • Details:

  1. Pfad-Resolvierung: base_path wird auf seinen absoluten Pfad normalisiert (resolve(strict=False)).
  2. Historie-Erstellung: build_release_history(base_path) wird aufgerufen, um die notwendigen Commit-Daten zu sammeln.
  3. Zielpfad-Resolvierung: target_path wird auf seinen absoluten Pfad normalisiert.
  4. Rendern und Schreiben:
  • render_release_notes(history, tag) wird aufgerufen, um den finalen Text zu generieren.
  • Der Text wird in die Zieldatei geschrieben, wobei die Zeichenkodierung utf-8 verwendet wird.

    • ---

_render_commit_groups(commits: Iterable[ReleaseCommit]) -> list[str]

Diese interne Funktion ist für die Formatierung der Commit-Daten verantwortlich. Zweck: Gruppiert die eingehenden ReleaseCommit-Objekte nach ihrer Kategorie (z. B. "Release", "Added", "Fixed", "Changed") und formatiert diese Gruppen in eine lesbare Textliste. Parameter:

  • **commits: Iterable[ReleaseCommit]**: Eine iterable Sammlung von Commit-Objekten, die die Historie darstellen.

    • Rückgabe:

  • **list[str]**: Eine Liste von Strings, die den finalen, formatierten Inhalt der Release Notes darstellen.

    • Details:

  1. Gruppierung: Es wird ein Dictionary (grouped) verwendet, um Commits basierend auf ihrem category-Attribut zu speichern.
  2. Fehlfall-Handling: Wenn keine Commits vorhanden sind, wird eine Meldung "- Keine Eintraege." zurückgegeben.
  3. Formatierung: Es werden die Kategorien ("Release", "Added", "Fixed", "Changed") sequenziell durchlaufen.
  • Für jede Kategorie wird eine Überschrift (### Kategorie) und eine Liste der Commits mit ihrem Subject und Short Hash (- Subject (hash)) erstellt.
  1. Sauberkeit: Am Ende wird sichergestellt, dass ein unnötiger leeres Ende (lines[-1] == "") entfernt wird, falls die letzte Kategorie leer war.

    2. ---

_run_git(base_path: Path, args: list[str]) -> str

Diese interne Funktion dient zur sicheren Ausführung von Git-Befehlen. Zweck: Führt einen spezifischen Git-Befehl (z. B. git log, git rev-parse etc.) im angegebenen Arbeitsverzeichnis aus und gibt die Standardausgabe zurück. Parameter:

  • **base_path: Path**: Das Verzeichnis, in dem der Git-Befehl ausgeführt werden soll.
  • **args: list[str]**: Die Argumente, die an den Git-Befehl übergeben werden (z. B. ["log", "--pretty=format:%s"]).

    • Rückgabe:

  • **str**: Die Standardausgabe (stdout) des Git-Befehls, bereinigt um unnötigen Whitespace.

    • Fehlerbehandlung:

  • **RuntimeError:** Wenn der Git-Befehl fehlschlägt (Return Code $\neq 0$), wird eine RuntimeError geworfen, die entweder die Fehlermeldung aus stderr oder eine generische Fehlermeldung enthält.

    • Details:

  • Ausführung: Nutzt subprocess.run zur Synchronous-Ausführung.
  • Sicherheit: check=False wird gesetzt, damit die Funktion selbst nicht sofort abstürzt, sondern den Fehler über eine RuntimeError an den Aufrufer weitergibt.
  • Text-Handling: capture_output=True und text=True stellen sicher, dass die Ausgabe als Text (String) zurückgegeben wird.

    • ---

Abhängigkeiten

  • **pathlib.Path**: Für die Verwaltung von Dateipfaden.
  • **subprocess**: Für die Ausführung externer Befehle (Git).
  • **typing**: Für die Typisierung von Parametern und Rückgabewerten (Path, Iterable, list).
  • **ReleaseCommit**: Eine definierte Klasse (nicht in dieser Datei sichtbar), die die Struktur der Git-Commits abbildet.
  • **render_release_notes**: Eine externe Funktion (nicht definiert), die die Logik zur eigentlichen Textgenerierung übernimmt.
  • **build_release_history**: Eine externe Funktion (nicht definiert), die die Git-Historie sammelt.

Zusammenfassung des Workflows

Der Prozess folgt dieser Kette: $$\text{Input (Path, Tag)} \rightarrow \text{build\_release\_history} \rightarrow \text{_run\_git (für Historie)} \rightarrow \text{render\_release\_notes} \rightarrow \text{write\_release\_notes (Schreiben)}$$

Technische Dokumentation: release_notes.py

Übersicht

Die Python-Skriptdatei release_notes.py dient dazu, Versionshinweise (Changelog) und optionalen Release Notes basierend auf der Git-Historie eines angegebenen Projektverzeichnisses zu generieren. Das Skript verwendet das argparse-Modul, um eine flexible Kommandozeilen-Schnittstelle (CLI) zu ermöglichen, die den Benutzer bei der Auswahl des Projektwurzelpfads, der Zielpfade für die Ausgaben und optionaler Tags für die Release Notes unterstützt.

Funktionen

main()

Die Hauptfunktion des Skripts, die die gesamte Logik der Argumentenverarbeitung und der Generierung der Dokumentationen koordiniert. Argumentenverarbeitung: Die Funktion definiert und parst die folgenden Kommandozeilen-Argumente:

Argument Typ Pflicht Beschreibung Standardwert
base_path String Nein Der Pfad zum Projektwurzelverzeichnis, aus dem die Git-Historie analysiert wird. . (Aktueller Verzeichnis)
--changelog-path String Nein Der gewünschte Pfad für die Ausgabe des Versionshinweises (CHANGELOG). CHANGELOG.md
