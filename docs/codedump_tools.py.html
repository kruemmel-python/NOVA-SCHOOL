Diese Datei enthält eine Reihe von Funktionen, die zur Filterung und Erkennung von Dateien dienen, insbesondere im Kontext eines "Dump"- oder Analyseprozesses (wie durch den Namen codedump_tools.py und die Verwendung von DumpConfig angedeutet). Die Funktionen prüfen, ob eine Datei ignoriert werden soll, ob sie als Code-Datei gilt oder welche Programmiersprache sie verwendet.

Funktionen

1. is_ignored(path: str, config: DumpConfig, *, output_path: str | None = None) -> bool

Diese Funktion entscheidet, ob ein gegebener Pfad ignoriert werden soll, basierend auf verschiedenen Kriterien aus der Konfiguration (DumpConfig). Parameter:

  • path: str: Der vollständige Pfad zur Datei, die überprüft werden soll.
  • config: DumpConfig: Ein Objekt, das die Konfigurationsparameter für die Ignorierung enthält (z.B. ignore_names, ignore_prefixes).
  • output_path: str | None = None: Optionaler Pfad, der zur Normalisierung des Zielpfads verwendet wird, falls dieser als Ignorierungskriterium relevant ist.

    • Logik:

  1. Normalisierung des Pfades:
  • Der Eingabepfad wird auf Unix-ähnliche Pfadzeichen (/) umgewandelt und führende/ende Schrägstriche entfernt.
  • Der Pfad wird in Kleinbuchstaben umgewandelt (casefold()).
  • Falls output_path angegeben ist, wird dieser ebenfalls normalisiert und in Kleinbuchstaben umgewandelt.
  1. Direkte Übereinstimmung mit Ausgabe:
  • Wenn der normalisierte Pfad und der normalisierte output_path (falls vorhanden) identisch sind, wird die Datei ignoriert (True).
  1. Erkennung von Dump-Artefakten:
  • Es wird die interne Funktion _is_dump_artifact(normalized) aufgerufen. Wenn diese Funktion True zurückgibt, wird die Datei ignoriert.
  1. Ignorierung nach Dateinamen/Teilen:
  • Der normalisierte Pfad wird in seine Komponenten (Teile) zerlegt.
  • Jede Teilkomponente wird in Kleinbuchstaben umgewandelt.
  • Es wird geprüft, ob irgendein Teilkomponente mit den Namen in config.ignore_names übereinstimmt. Wenn ja, wird die Datei ignoriert.
  1. Ignorierung nach Präfixen:
  • Für jedes Präfix in config.ignore_prefixes wird geprüft:
  • Ist der gesamte normalisierte Pfad gleich dem normalisierten Präfix?
  • Fängt der normalisierte Pfad mit dem Präfix gefolgt von einem Schrägstrich an (z.B. prefix/rest_of_path)?
  • Wenn eine dieser Bedingungen erfüllt ist, wird die Datei ignoriert.
  1. Rückgabe:
  • Wenn keine der oben genannten Ignorierungsbedingungen zutrifft, wird False zurückgegeben.

2. def is_code_file(path: str, config: DumpConfig) -> bool

Diese Funktion prüft, ob eine Datei als Code-Datei klassifiziert werden soll, basierend auf den erlaubten Dateiendungen und Dateinamen in der Konfiguration. Parameter:

  • path: str: Der Pfad zur Datei, die überprüft werden soll.

    • Logik:

  1. Erweiterung extrahieren: Die Dateiendung (suffix) des Pfades wird extrahiert und in Kleinbuchstaben umgewandelt.

  3. Alternative Prüfung: Es wird geprüft, ob der Dateiname (candidate.name) in der Liste der erlaubten Dateinamen (config.allowed_filenames) enthalten ist.
  4. Rückgabe: Die Funktion gibt True zurück, wenn entweder die Erweiterung oder der Dateiname erlaubt ist, ansonsten False.

  3. Rückgabe:

  • Wenn keine Übereinstimmung gefunden wird, wird standardmäßig "text" zurückgegeben.

Die Funktion dump_target_to_markdown ist eine zentrale Schnittstelle zur Generierung eines Markdown-Dokuments aus einem gegebenen Ziel (Datei oder Verzeichnis). Sie übernimmt die Logik, um den Inhalt des Zielobjekts zu sammeln (entweder als Verzeichnis-Dump oder als ZIP-Datei-Dump) und das Ergebnis in ein formatiertes Markdown-Format zu rendern, bevor es in die angegebene Ausgabe geschrieben wird.

2. Signatur

def dump_target_to_markdown(target: str | Path, output_md: str | Path, config: DumpConfig = CONFIG) -> Path:
# ... Funktionskörper ...

3. Parameter

Parameter Typ Beschreibung Standardwert
target str oder Path Der Pfad zur zu dumpenden Ressource. Kann ein Dateipfad (str oder Path) oder ein Verzeichnis (Path) sein. Nicht angegeben
output_md str oder Path Der Pfad oder die Zeichenkette, an die das resultierende Markdown-Dokument geschrieben werden soll. Nicht angegeben
config DumpConfig Eine Konfigurationsstruktur, die für den Dump-Prozess verwendet wird (z.B. für spezifische Dump-Einstellungen). CONFIG (Globaler Standardwert)

4. Rückgabe (Return Value)

Typ Beschreibung
Path Der Pfad zur erfolgreich erstellten Markdown-Datei (output_path).

5. Logik und Verhalten

Die Funktion führt die folgenden Schritte durch:

  1. Pfad-Konvertierung:
  • Der eingegebene target wird in ein Path-Objekt umgewandelt (target_path).
  • Der output_md wird in ein Path-Objekt umgewandelt (output_path).
  1. Ausgabeverzeichnis Sicherstellung:
  • Das Verzeichnis, das dem output_path zugeordnet ist, wird erstellt, falls es nicht existiert (output_path.parent.mkdir(parents=True, exist_ok=True)).
  1. Input-Typ-Prüfung und Dump-Methode Auswahl:

    2. Die Funktion prüft den Typ des target_path, um die korrekte Dump-Methode auszuwählen:

  • **Fall 1: Verzeichnis (target_path.is_dir()):**
  • Es wird die Funktion collect_directory_dump aufgerufen, um den Inhalt des Verzeichnisses zu sammeln.
  • **Fall 2: ZIP-Datei (target_path.is_file() and target_path.suffix.lower() == ".zip"):**
  • Es wird die Funktion collect_zip_dump aufgerufen, um den Inhalt der ZIP-Datei zu sammeln.
  • Fall 3: Ungültiger Input:
  • Wenn weder ein Verzeichnis noch eine .zip-Datei erkannt wird, wird eine FileNotFoundError mit einer spezifischen Fehlermeldung ausgelöst.
  1. Markdown-Rendern und Speichern:
  • Das Ergebnis (result) der jeweiligen Dump-Funktion wird durch die Funktion render_dump_markdown in ein Markdown-String umgewandelt.
  • Dieser String wird in die Datei unter Verwendung der UTF-8-Kodierung geschrieben (output_path.write_text(...)).
  1. Rückgabe:
  • Der Pfad zur erstellten Markdown-Datei wird zurückgegeben.

6. Abhängigkeiten

  • Path (von pathlib)
  • collect_directory_dump (Externe Funktion)
  • collect_zip_dump (Externe Funktion)
  • render_dump_markdown (Externe Funktion)
  • DumpConfig (Definierte Konfigurationsklasse/Struktur)

7. Fehlerbehandlung

  • **FileNotFoundError:** Wird explizit geworfen, wenn das target weder ein Verzeichnis noch eine .zip-Datei ist.

8. Anwendungsfälle

Diese Funktion ist ideal für Szenarien, in denen man strukturierten, lesbaren Dokumentationen (Markdown) aus komplexen Datenquellen (Dateisysteme oder komprimierten Archiven) generieren muss. Beispiel-Workflow: Ein Benutzer möchte den gesamten Inhalt eines Projektverzeichnisses (/home/user/project_data) in ein Markdown-Dokument speichern, das in den Ordner /output/reports/ abgelegt wird.

Technische Dokumentation: collect_directory_dump Funktion

1. Funktion: collect_directory_dump

Zweck: Diese Funktion durchläuft einen gegebenen Projektordner (project_root) rekursiv, identifiziert Dateien, die als Code gelten, und erstellt daraus einen Dump-Ergebnis (DumpResult). Sie filtert gleichzeitig Dateien, die gemäß einer Konfiguration ignoriert werden, und sammelt Pfade zu Dateien, die nicht als Code betrachtet werden. Signature:

Rückgabe:

Typ Beschreibung
DumpResult Ein Objekt, das die Ergebnisse des Dump-Prozesses enthält, einschließlich der gefundenen Einträge, ignorierter Pfade und Pfade zu nicht-Code-Dateien.

2. Funktionsbeschreibung und Logik

Die Funktion führt die folgenden Schritte durch:

  1. Initialisierung:
  • Eine leere Liste entries wird initialisiert, um die gefundenen Code-Dateien zu speichern.
  • Eine leere Liste ignored_paths wird initialisiert, um Pfade zu speichern, die ignoriert wurden.
  • Eine leere Liste non_code_paths wird initialisiert, um Pfade zu speichern, die nicht als Code-Dateien klassifiziert wurden.
  • Der Zielpfad (output_path) wird normalisiert (output_resolved) und gegebenenfalls relativ zum Projektwurzelpfad (output_relative) berechnet.
  1. Rekursive Durchsicht des Projekts:
  • Die Funktion iteriert über alle Dateien und Verzeichnisse im project_root rekursiv (project_root.rglob("*")).
  • Die Iteration erfolgt in alphabetischer Reihenfolge (basierend auf der POSIX-Sortierung des Pfades), um eine konsistente Ausgabe zu gewährleisten.
  • Filterung 1: Nicht-Dateien: Dateien, die keine Dateien sind (z.B. Verzeichnisse), werden übersprungen.
  1. Filterung 2: Spezifische Ignorierung (Output-Pfad):
  • Wenn output_path angegeben ist, wird geprüft, ob der aktuelle Pfad exakt dem aufgelösten Zielpfad entspricht. Wenn ja, wird der relative Pfad des aktuellen Pfades zum Projektwurzelpfad in ignored_paths aufgenommen, und die Iteration wird fortgesetzt (continue).
  1. Filterung 3: Allgemeine Ignorierung:
  • Der relative Pfad der aktuellen Datei wird berechnet.
  • Die Funktion prüft, ob is_ignored(rel_path, config, output_path=output_relative) True zurückgibt. Wenn ja, wird der relative Pfad in ignored_paths aufgenommen, und die Iteration wird fortgesetzt.
  1. Filterung 4: Code-Dateien:
  • Die Funktion prüft, ob is_code_file(rel_path, config) True zurückgibt.
  • Wenn nicht Code-Datei (False), wird der relative Pfad in non_code_paths aufgenommen, und die Iteration wird fortgesetzt.
  • Wenn Code-Datei (True), wird die Datei als Eintrag in entries hinzugefügt mithilfe der internen Funktion _entry_from_path.
  1. Rückgabe:
  • Am Ende wird ein DumpResult Objekt erstellt, das alle gesammelten Daten (source_name, source_path, entries, ignored_paths, non_code_paths) enthält und zurückgegeben wird.

  • **Path (von pathlib):** Wird verwendet, um Dateisystempfade manipulationssicher zu verwalten.
  • **DumpConfig:** Eine externe Struktur, die die Geschäftslogik für die Klassifizierung von Dateien (z.B. welche Dateiendungen oder Verzeichnisse ignoriert werden) definiert.
  • **is_ignored(rel_path, config, output_path):** Eine interne Helper-Funktion, die die spezifischen Ignorierungsregeln prüft.
  • **is_code_file(rel_path, config):** Eine interne Helper-Funktion, die entscheidet, ob eine Datei als Code-Datei betrachtet werden soll.
  • **_entry_from_path(path, rel_path, config):** Eine interne Helper-Funktion, die die spezifische Struktur des Dump-Eintrags für eine identifizierte Code-Datei erstellt.

Diese Funktion ist ideal für folgende Szenarien:

  • Code-Analyse-Vorbereitung: Erstellung einer Übersicht aller relevanten Code-Dateien in einem Projekt, während gleichzeitig nicht-Code-Dateien und konfigurierte Ignorierungen separat erfasst werden.
  • Build-System-Integration: Nutzung der Ergebnisse, um zu entscheiden, welche Dateien in einen bestimmten Build-Prozess einbezogen werden sollen.

Die Funktion collect_zip_dump ist eine zentrale Methode zur Verarbeitung eines ZIP-Archivs. Sie durchläuft alle Dateien im Archiv, filtert diese basierend auf Konfigurationsregeln (Ignorieren von Pfaden, Identifizierung von Code-Dateien) und erstellt eine strukturierte DumpResult-Instanz, die die extrahierten Informationen enthält.

3. Beschreibung

Diese Funktion liest ein ZIP-Archiv an der angegebenen zip_path und extrahiert die relevanten Dateien. Sie wendet spezifische Filter an, um nur Code-Dateien zu extrahieren und andere Dateien (wie Dokumentationen oder Konfigurationsdateien) zu kategorisieren.

4. Parameter

Parameter Typ Pflicht Beschreibung

Typ Beschreibung
DumpResult Eine Instanz der Klasse DumpResult, die die Ergebnisse der Analyse enthält.

**Struktur von DumpResult:**

  • source_name: Der Stammname des ZIP-Archivs (ohne die .zip-Erweiterung).
  • source_kind: Der Typ der Quelle ("zip").
  • source_path: Der vollständige Pfad zum ursprünglichen ZIP-Archiv (als String).
  • entries: Eine Liste von DumpEntry-Objekten, die die erfolgreich identifizierten Code-Dateien enthalten.
  • ignored_paths: Eine Liste von relativen Pfaden, die aufgrund der Konfiguration ignoriert wurden.
  • non_code_paths: Eine Liste von relativen Pfaden, die als nicht-Code-Dateien klassifiziert wurden.

6. Funktionsweise (Algorithmus)

  1. Initialisierung:
  • Eine leere Liste entries wird initialisiert, um die gefundenen Code-Dateien zu speichern.
  • Zwei Listen, ignored_paths und non_code_paths, werden initialisiert, um die Kategorisierung zu speichern.
  • Der Name des Ausgabe-Pfads (output_name) wird basierend auf output_path ermittelt.
  1. Archiv-Lesevorgang:
  • Das ZIP-Archiv wird im Modus r (read) geöffnet.
  1. Dateien-Durchlauf:
  • Alle Elemente (infolist()) des Archivs werden durchlaufen.
  • Filter 1 (Nicht-Verzeichnisse): Nur Dateien (not info.is_dir()) werden in die Liste files aufgenommen.
  • Sortierung: Die Dateien werden nach ihrem relativen Pfad (case-insensitive) sortiert, um eine konsistente Verarbeitung zu gewährleisten.
  1. Einzelne Datei-Verarbeitung:
  • Für jede Datei wird der relative Pfad (rel_path) extrahiert (Windows-Backslashes werden auf Unix-Forward Slashes umgewandelt und Leerzeichen entfernt).
  • Filter 2 (Ignorieren): Es wird geprüft, ob is_ignored(rel_path, config, output_path=output_name) wahr ist. Wenn ja, wird der Pfad zu ignored_paths hinzugefügt, und die Verarbeitung dieser Datei wird beendet (continue).
  • Filter 3 (Code-Identifikation): Es wird geprüft, ob is_code_file(rel_path, config) wahr ist. Wenn nein, wird der Pfad zu non_code_paths hinzugefügt, und die Verarbeitung dieser Datei wird beendet (continue).
  • Erfassung: Wenn die Datei weder ignoriert noch als nicht-Code-Datei identifiziert wurde, wird ein _entry_from_zip-Objekt erstellt und zur Liste entries hinzugefügt.
  1. Rückgabe:
  • Die Funktion gibt die finalisierte DumpResult-Instanz zurück.

7. Abhängigkeiten

  • pathlib.Path: Für die Handhabung von Dateipfaden.
  • zipfile.ZipFile: Für die Arbeit mit ZIP-Archiven.
  • DumpConfig: Eine definierte Konfigurationsklasse/Struktur.
  • DumpEntry: Eine definierte Klasse/Struktur für die erstellten Einträge.
  • is_ignored(rel_path, config, output_path): Eine externe Helper-Funktion zur Ignorierregeln.

  • _entry_from_zip(archive, info, config): Eine interne Helper-Funktion zur Konvertierung der ZIP-Info in ein DumpEntry-Objekt.

  • Code-Audit: Systematische Extraktion aller Quellcode-Dateien aus einem Projekt-ZIP-Archiv zur schnellen Analyse.
  • Archiv-Cleanup: Identifizierung und Trennung von Code-Dateien von Metadaten oder Dokumentationsdateien innerhalb eines Archivs.
  • Konfigurationsanalyse: Spezifische Filterung von Dateien, die für die Analyse irrelevant sind (z.B. Build-Artefakte oder temporäre Dateien).

Technische Dokumentation: Funktion render_dump_markdown

1. Übersicht

Die Funktion render_dump_markdown ist eine Methode, die das Ergebnis einer Code-Dump-Analyse (DumpResult) in ein formatiertes Markdown-String-Format umwandelt. Sie dient dazu, die Ergebnisse der Analyse lesbar und strukturiert darzustellen, ideal für Berichte oder Dokumentationen.

2. Signatur

def render_dump_markdown(result: DumpResult, *, config: DumpConfig = CONFIG) -> str:

Parameter:

  • **result: DumpResult**: Das Ergebnis der Code-Dump-Analyse. Dieses Objekt enthält Metadaten über die Quelle, die gefundenen Dateien, ignorierte Pfade und andere relevante Informationen.
  • **config: DumpConfig = CONFIG (Optional)**: Eine Konfigurationsstruktur, die spezifische Formatierungsdetails steuert (z.B. Profilname, maximale Dateigröße, Limitierungen für die Zusammenfassung ignorierter Pfade). Standardmäßig wird die globale Konfiguration CONFIG verwendet.

    • Rückgabe:

  • **str**: Ein zusammenhängender String im Markdown-Format, der die strukturierte Darstellung des Code-Dumps enthält.

3. Funktionsbeschreibung

Die Funktion konstruiert einen Markdown-String, der folgende Hauptabschnitte enthält:

  1. Titel und Metadaten: Eine Überschrift mit dem Namen der Quelle und eine Zusammenfassung der Analyseparameter (Quelle, Typ, Profil, Anzahl der Dateien, Ignorierte Pfade etc.).
  2. Ignorierte Bereiche (Optional): Wenn result.ignored_paths nicht leer ist, wird ein Abschnitt für ignorierte Pfade erstellt, basierend auf der Konfiguration (config.skipped_preview_limit).
  3. Projektstruktur: Eine Darstellung des Dateisystems (Baumstruktur) der gefundenen Dateien, generiert mithilfe der Funktion generate_tree.
  4. Dateien: Eine detaillierte Auflistung aller gefundenen Dateien. Für jede Datei werden folgende Informationen angezeigt:
  • Pfad (entry.path)
  • Sprache (entry.language)
  • Größe (entry.size)
  • Status (entry.status)
  • Der eigentliche Code-Inhalt, formatiert in einem Code-Block, der der Sprache (entry.language) entspricht.

4. Abhängigkeiten und interne Logik

Die Funktion verlässt sich auf folgende interne Komponenten:

  • **_summarize_paths(result.ignored_paths, config=config)**: Eine interne Funktion, die die ignorierten Pfade zusammenfasst, um die Anzahl der Dateien pro Label zu ermitteln.
  • **generate_tree(...)**: Eine interne Funktion, die eine Baumstruktur (Tree View) aus einer Liste von Pfaden generiert.

5. Datenfluss und Logikschritte

  1. Initialisierung: Eine Liste (lines) wird mit den statischen Header-Informationen (Titel, Übersicht) gefüllt.
  2. Ignorierte Pfade Verarbeitung:
  • Prüft, ob result.ignored_paths existiert.
  • Wenn ja, fügt die Überschriften für "Ignorierte Bereiche" hinzu.
  • Iteriert über die zusammengefassten ignorierten Pfade und fügt eine Liste von Einträgen hinzu.
  1. Projektstruktur Generierung:
  • Fügt die Überschrift "Projektstruktur" hinzu.
  • Fügt einen Text-Code-Block ( `text ) ein.
  • Fügt das Ergebnis von generate_tree ein (oder eine Standardnachricht, falls keine Dateien gefunden wurden).
  • Schließt den Text-Code-Block.
  1. Dateien Auflisten:
  • Iteriert über alle Einträge in result.entries.
  • Für jeden Eintrag wird ein Abschnitt (### {entry.path}) erstellt.
  • Die Metadaten (Sprache, Größe, Status) werden aufgelistet.
  • Der Code-Inhalt wird in einem entsprechenden Code-Block ( `language ) eingefügt.
  1. Finalisierung: Alle erstellten Zeilen werden durch einen Zeilenumbruch (\n) verbunden und die resultierende Zeichenkette wird bereinigt (.rstrip()) und mit einem abschließenden Zeilenumbruch beendet.

6. Randbedingungen und Besonderheiten

  • Konfigurationsabhängigkeit: Die Darstellung von ignorierten Pfaden wird durch config.skipped_preview_limit begrenzt.
  • Code-Block-Formatierung: Die Sprache des Code-Blocks wird dynamisch aus entry.language bestimmt, was eine sprachspezifische Syntaxhervorhebung ermöglicht.
  • Fehlende Dateien: Wenn result.entries leer ist, wird stattdessen eine Platzhalternachricht ([keine passenden Dateien gefunden]) im Baum-Abschnitt angezeigt.

7. Nutzungsszenario

Diese Funktion wird typischerweise aufgerufen, wenn eine Analyse abgeschlossen ist und die Rohdaten (die Dump-Ergebnisse) in ein benutzerfreundliches, lesbares Dokument umgewandelt werden sollen, beispielsweise für die Anzeige in einer Web-Oberfläche oder die Erstellung eines finalen Berichts.

Funktion: _entry_from_path

Beschreibung: Diese interne Funktion liest den Inhalt einer Datei, prüft deren Größe gegen eine Konfiguration und erstellt ein DumpEntry Objekt, das den Prozessstatus (erfolgreich, übersprungen wegen Größe, oder Fehler beim Lesen) dokumentiert. Signatur:

def _entry_from_path(path: Path, relative_path: str, config: DumpConfig) -> DumpEntry:

Parameter:

Parameter Typ Beschreibung
path Path Der vollständige Pfad zur Datei, die verarbeitet werden soll.
relative_path str Der relative Pfad, der in der Dump-Ergebnisseite angezeigt werden soll.
config DumpConfig Eine Konfigurationsstruktur, die Parameter wie max_file_size enthält.

Rückgabe:

Typ Beschreibung
DumpEntry Ein Objekt, das die Ergebnisse der Verarbeitung (Pfad, Sprache, Größe, Status und Inhalt) speichert.

Logik und Verhalten:

  1. Größenprüfung:
  • Die aktuelle Dateigröße (size) wird mithilfe von path.stat().st_size ermittelt.
  • Bedingung: Wenn die size größer ist als der Wert von config.max_file_size, wird die Funktion sofort einen DumpEntry mit dem Status "skipped-too-large" zurückgeben.
  1. Dateilese-Versuch:
  • Falls die Datei die Größenbeschränkung nicht überschreitet, wird versucht, den gesamten Inhalt der Datei in Bytes zu lesen (path.read_bytes()).
  • Fehlerbehandlung (Lesefehler): Falls ein Fehler beim Lesen der Datei auftritt (z.B. Berechtigsprobleme, Datei nicht erreichbar), wird ein DumpEntry mit dem Status "read-error" zurückgegeben, der den spezifischen Fehler (exc) dokumentiert.
  1. Erfolgreiche Verarbeitung:
  • Wenn das Lesen der Datei erfolgreich ist, wird die Funktion zur internen Hilfsfunktion _entry_from_bytes delegiert, um den eigentlichen Dump-Eintrag zu erstellen.

    • Abhängigkeiten:

  • Path (von pathlib)
  • DumpConfig (eine definierte Konfigurationsklasse/Struktur)
  • detect_language(relative_path) (eine externe Funktion zur Sprachdetektion)
  • _entry_from_bytes(...) (eine interne Funktion, die den eigentlichen Inhalt verarbeitet)

  1. Input: Ein Pfad zu einer Datei, eine Konfiguration mit max_file_size=102400 Bytes.
  2. Schritt 1: Die Größe der Datei wird ermittelt.
  3. Prüfung: Ist die Größe $> 102400$ Bytes?
  • JA: Rückgabe: status="skipped-too-large".
  • NEIN: Versuche, die Datei zu lesen.
  1. Lesevorgang: Wenn das Lesen fehlschlägt, Rückgabe: status="read-error".
  2. Erfolg: Wenn das Lesen erfolgreich ist, wird der Inhalt verarbeitet und ein DumpEntry mit dem korrekten Status zurückgegeben.

Zweck: Diese interne Funktion ist dafür verantwortlich, eine einzelne Datei innerhalb eines zipfile.ZipFile-Archivs zu verarbeiten. Sie prüft die Dateigröße, behandelt mögliche Lesefehler und leitet die Rohdaten der Datei an eine nachfolgende Funktion (_entry_from_bytes) weiter, falls alle Prüfungen erfolgreich sind. Signaturen:

def _entry_from_zip(archive: zipfile.ZipFile, info: zipfile.ZipInfo, config: DumpConfig) -> DumpEntry:
# ... Implementierung ...

Parameter:

Parameter Typ Beschreibung
archive zipfile.ZipFile Das offene ZIP-Archiv-Objekt, aus dem die Datei gelesen werden soll.
info zipfile.ZipInfo Informationen über die spezifische Datei innerhalb des ZIP-Archivs (z.B. Name, Größe).

  • Wenn die Größe akzeptabel ist: Es wird versucht, die Datei im Archiv zu öffnen und den gesamten Inhalt in eine Bytes-Variable (raw) zu lesen (archive.open(info)).
  • Fehlerbehandlung (Exception Handling):
  • Falls ein Fehler beim Öffnen oder Lesen der Datei auftritt (z.B. Korruptes Archiv, Zugriffsprobleme), wird eine DumpEntry zurückgegeben mit dem Status "read-error" und einer detaillierten Fehlermeldung, die den verursachten Exception enthält.
  1. Weiterleitung an nachfolgende Funktion:
  • Bei erfolgreichem Lesen: Die Rohdaten (raw) und die Metadaten werden an die Funktion _entry_from_bytes weitergegeben, zusammen mit dem bereinigten Pfad, der ursprüngliche Dateigröße und die Konfiguration.
  1. Rückgabe (Erfolg):
  • Wenn alle Schritte erfolgreich sind, wird das Ergebnis der Verarbeitung durch _entry_from_bytes zurückgegeben.

Abhängigkeiten

  • zipfile (Standardbibliothek)
  • DumpConfig (Custom Klasse)
  • DumpEntry (Custom Klasse)
  • detect_language (Externe Funktion, die die Sprache aus dem Pfad erkennt)
  • _entry_from_bytes (Interne Funktion, die die eigentliche Datenverarbeitung durchführt)

Anwendungsfälle

Diese Funktion wird verwendet, um sicherzustellen, dass nur Dateien mit akzeptabler Größe und die erfolgreich lesbaren Dateien aus einem ZIP-Archiv verarbeitet werden, bevor die eigentliche Analyse oder Extraktion der Datei beginnt.

Verbesserungsvorschläge (Optional)

  1. Typ-Checking: Obwohl die Typen angegeben sind, könnte eine stärkere Typisierung (z.B. durch typing.Annotated oder spezifischere Interfaces) die Wartbarkeit verbessern.
  2. Konfigurationsprüfung: Es könnte eine zusätzliche Prüfung erfolgen, um sicherzustellen, dass config.max_file_size eine gültige positive Zahl ist, bevor die Größenprüfung durchgeführt wird.
  3. Fehler-Logging: Für eine produktive Anwendung wäre es ratsam, die zurückgegebenen DumpEntry-Objekte nicht nur zurückzugeben, sondern sie an ein zentrales Logging-System zu senden, um Fehler und Übersprünge zu protokollieren.

Technische Dokumentation: codedump_tools.py

Übersicht

Diese Datei enthält Funktionen zur Verarbeitung von Rohdaten (Bytes) aus einem Dump-Prozess, um diese in strukturierte DumpEntry-Objekte zu konvertieren und Pfade zu aggregieren. Die Hauptfunktionen behandeln die Erkennung von binären Daten und die Dekodierung von Textdaten.

Funktionen

1. _entry_from_bytes(relative_path: str, raw: bytes, size: int, config: DumpConfig) -> DumpEntry

Diese interne Funktion ist verantwortlich für die Verarbeitung eines einzelnen Datenblocks (raw Bytes) basierend auf dem angegebenen Pfad und den Konfigurationsparametern. Zweck: Entscheidet, ob die Daten als binär oder als textbasiert behandelt werden sollen, und erstellt entsprechend eine DumpEntry. Parameter:

  • relative_path: str: Der relative Pfad des analysierten Datenblocks.
  • raw: bytes: Die rohen Daten, die analysiert werden sollen.
  • size: int: Die Größe der analysierten Daten (wahrscheinlich in Bytes).
  • config: DumpConfig: Eine Konfigurationsstruktur, die die notwendigen Informationen für die Dekodierung (Encoding, Fallback-Encoding) bereitstellt.

    • Logik:

  1. Binäre Datenprüfung: Prüft, ob das Byte b"\x00" in den raw-Daten vorkommt.
  • Wenn ja: Die Datei wird als binär markiert (status="skipped-binary") und der Inhalt wird als Platzhalter ("[SKIPPED: binary content detected]") gespeichert.
  1. Text-Dekodierung: Wenn keine Null-Byte-Sequenz gefunden wird, versucht die Dekodierung der Bytes:
  • Zuerst wird die Dekodierung mit config.encoding versucht.
  • Falls eine UnicodeDecodeError auftritt, wird versucht, die Dekodierung mit config.fallback_encoding unter Verwendung von errors="replace" durchzuführen.
  1. Ergebnis: Ein DumpEntry wird zurückgegeben, wobei der Status entweder "included" (für erfolgreich dekodierten Text) oder "skipped-binary" (für binäre Daten) ist.

    2. Rückgabe:

  • DumpEntry: Ein Objekt, das den Pfad, die Sprache (erkannt durch detect_language), die Größe, den Status und den dekodierten Inhalt enthält.

    • ---

2. _summarize_paths(paths: Iterable[str], *, config: DumpConfig) -> list[tuple[str, int]]

Diese Funktion aggregiert Pfade basierend auf einer Labeling-Strategie, um eine Zusammenfassung der Pfadverteilung zu erstellen. Zweck: Zählt die Häufigkeit verschiedener Labels für eine Liste von Pfaden und sortiert das Ergebnis nach Häufigkeit (absteigend) und dann alphabetisch (kleingeschrieben). Parameter:

  • paths: Iterable[str]: Eine iterable Liste von Pfaden, die zusammengefasst werden sollen.
  • config: DumpConfig: Die Konfigurationsstruktur, die für die Bestimmung des Labels verwendet wird.

    • Logik:

  1. Initialisierung: Ein Dictionary buckets wird erstellt, um die Zählung der Labels zu speichern (label: count).
  2. Labeling und Zählung: Iteriert über jeden path in paths:
  • Rückt das Label (label) für den Pfad mithilfe von _summary_label_for_path(path, config=config) heraus.
  • Incrementiert den Zähler für dieses Label in buckets.
  1. Sortierung: Das resultierende Dictionary wird in eine Liste von Tupeln ((label, count)) umgewandelt und sortiert:
  • Primärer Sortierkriterium: Die Zählung (item[1]) in absteigender Reihenfolge ($-item[1]$).
  • Sekundärer Sortierkriterium: Der Label (item[0]) in alphabetischer Reihenfolge (kleingeschrieben, item[0].casefold()).

    • Rückgabe:

  • list[tuple[str, int]]: Eine sortierte Liste von Tupeln, wobei jedes Tupel ein Label und die Anzahl der Pfade darstellt, die dieses Label aufweisen.

Abhängigkeiten und Notizen

  • **DumpConfig:** Diese Klasse muss definiert sein und muss die notwendigen Attribute (encoding, fallback_encoding) für die Dekodierung bereitstellen.
  • **detect_language(relative_path: str) -> str:** Eine externe Funktion, die die Sprache eines Pfades erkennt.
  • **_summary_label_for_path(path: str, config: DumpConfig) -> str:** Eine externe Funktion, die entscheidet, welches Label für einen bestimmten Pfad verwendet werden soll.
  • **DumpEntry:** Eine Klasse, die die Ergebnisse der Verarbeitung (Pfad, Inhalt, Status etc.) strukturiert.

Verwendungshinweise

  • Die Funktionen _entry_from_bytes und _summarize_paths sind interne Hilfsfunktionen und sollten primär über die Hauptlogik des Dump-Prozesses aufgerufen werden.
  • Stellen Sie sicher, dass config korrekt initialisiert ist, um fehlerhafte Dekodierungen zu vermeiden.

Übersicht

Diese Dokumentation beschreibt die Funktionen der Python-Datei codedump_tools.py, die spezifische Logik für die Verarbeitung und Benennung von Pfaden im Kontext eines "Code Dump"-Prozesses implementiert. Die Funktionen dienen dazu, Pfade zu normalisieren, zu ignorieren und den korrekten Ausgabe-/Referenznamen für einen Code Dump zu bestimmen.

Funktionen

1. _summary_label_for_path(path: str, *, config: DumpConfig) -> str

Diese interne Funktion ist verantwortlich für die Generierung eines zusammenfassenden Labels oder Namens für einen gegebenen Pfad basierend auf einer Konfiguration (DumpConfig). Beschreibung: Sie normalisiert den Eingabepfad, prüft, ob er durch bestimmte Präfixe oder ignorierte Namen gekennzeichnet ist, und entscheidet basierend auf diesen Regeln, welchen Label-Namen zurückgegeben werden soll. Parameter:

  • path: str: Der Pfad, für den ein Label generiert werden soll.
  • config: DumpConfig: Eine Konfigurationsstruktur, die Regeln für die Pfadverarbeitung enthält (z.B. ignore_prefixes, ignore_names).

    • Rückgabe:

  • str: Das generierte Label oder der korrigierte Pfadbestandteil.

    • Logikdetails:

  1. Normalisierung: Der Pfad wird auf Unix-ähnliche Pfadzeichen (/) umgewandelt und führende/endende Leerzeichen entfernt.
  2. Präfix-Prüfung: Es wird geprüft, ob der normalisierte Pfad mit den Präfixen in config.ignore_prefixes übereinstimmt oder diesen beginnt. Wenn ja, wird das entsprechende Präfix (normalisiert) als Label zurückgegeben.
  3. Ignorierte Namen: Der Pfad wird in seine Teile zerlegt. Es wird geprüft, ob irgendein Teil des Pfades mit den Namen in config.ignore_names übereinstimmt. Wenn ja, wird der entsprechende Name aus der Konfiguration zurückgegeben.
  4. Artefakt-Prüfung: Es wird geprüft, ob der normalisierte Pfad ein "Dump-Artefakt" ist (siehe _is_dump_artifact). Falls ja, wird "codedump-artifacts" zurückgegeben.
  5. Standardfall: Wenn keine der oben genannten Bedingungen zutrifft, wird der erste Teil des normalisierten Pfades (parts[0]) oder, falls der Pfad leer ist, "." zurückgegeben.

    6. ---

2. _is_dump_artifact(path: str) -> bool

Diese interne Funktion prüft, ob ein gegebenen Pfad als ein generiertes Code-Dump-Artefakt identifiziert werden kann. Beschreibung: Sie prüft, ob der Name des Pfades bestimmte Muster aufweist, die auf ein Artefakt hindeuten. Parameter:

  • path: str: Der Pfad, der überprüft werden soll.

    • Rückgabe:

  • bool: True, wenn der Pfad ein Dump-Artefakt ist, False andernfalls.

    • Logikdetails:

  • Sie extrahiert den Namen des Pfades, konvertiert ihn in Kleinbuchstaben und prüft, ob er entweder mit "codedump" beginnt und mit ".md" endet, oder mit ".codedump.md" endet.

    • ---

3. _is_directory_like_target(target: Path) -> bool

Diese interne Funktion prüft, ob ein gegebenes Path-Objekt wie ein Verzeichnis aussieht. Beschreibung: Sie entscheidet, ob ein Zielpfad entweder ein tatsächliches Verzeichnis ist oder ob er als "verzeichnisähnlich" betrachtet werden soll (z.B. wenn er keine Endung hat). Parameter:

  • target: Path: Das Path-Objekt zur Prüfung.

    • Rückgabe:

  • bool: True, wenn das Ziel ein Verzeichnis ist oder keine Endung hat.

    • Logikdetails:

  • Wenn target.exists() wahr ist, wird geprüft, ob target.is_dir() zurückgibt.
  • Andernfalls wird geprüft, ob target.suffix leer ist (was oft auf ein Verzeichnis hindeutet, das noch keine Datei hat).

    • ---

4. default_output_path(target: str | Path) -> Path

Diese Funktion bestimmt den Standard-Ausgabepfad für einen gegebenen Zielpfad. Beschreibung: Sie konfiguriert die Endung des Pfades, basierend darauf, ob das Ziel ein Verzeichnis ist oder nicht. Parameter:

  • target: str | Path: Der ursprüngliche Pfad, für den der Ausgabepfad bestimmt werden soll.

    • Rückgabe:

  • Path: Der berechnete, standardmäßige Ausgabepfad.

    • Logikdetails:

  1. Der Eingabepfad wird zu einem Path-Objekt konvertiert.
  2. Verzeichnis-Logik: Wenn _is_directory_like_target zurückgibt (True), wird .CODEDUMP.md am Ende des Pfades angehängt.
  3. Datei-Logik: Andernfalls (wenn es eine Datei oder ein spezifischer Pfad ist), wird die Endung durch .codedump.md ersetzt (target.with_suffix(".codedump.md")).

Abhängigkeiten

  • pathlib.Path: Für die Manipulation von Pfaden.

Zusammenfassung der Nutzungsszenarien

Diese Tools werden typischerweise verwendet, um:

  1. Pfad-Mapping: Unstrukturierte oder komplexe Pfade in konsistente, lesbare Labels umzuwandeln (_summary_label_for_path).
  2. Artefakt-Erkennung: Spezifische Dateien, die als Code-Dump-Artefakte markiert sind, zu erkennen (_is_dump_artifact).
  3. Standardisierung: Sicherzustellen, dass alle Ausgaben (Dateien oder Verzeichnisse) einen konformen Namen erhalten (default_output_path).

Funktionen

1. config_for_profile(profile: str, *, max_file_size: int | None = None) -> DumpConfig

Diese Funktion erstellt eine DumpConfig-Instanz basierend auf dem angegebenen Dump-Profil und optionalen Optionen für die maximale Dateigröße. Parameter:

  • profile: str: Der gewünschte Dump-Profilname (z.B. "compact", "deep", oder ein anderer Name). Optional, wird standardmäßig "standard" verwendet.
  • max_file_size: int | None = None: Optionaler Wert für die maximale Dateigröße in Bytes. Wenn angegeben, wird die Konfiguration entsprechend angepasst.

    • Rückgabe:

  • DumpConfig: Eine Konfigurationsstruktur für die Dump-Operation.

    • Beschreibung:

  3. Konfiguration: Basierend auf dem normalisierten Profil werden spezifische Einstellungen für ignore_prefixes, max_file_size und skipped_preview_limit gesetzt:
  • **compact:**
  • ignore_prefixes: ("data", "docs", "wiki", "nova_school_server/tests")
  • max_file_size: 80_000 Bytes
  • skipped_preview_limit: 12
  • **deep:**
  • ignore_prefixes: ("data",)
  • max_file_size: 350_000 Bytes
  • skipped_preview_limit: 60
  • Standard: Keine spezifischen Änderungen (außer der Profilzuweisung).
  1. Maximale Dateigröße: Wenn max_file_size angegeben wird, wird der Wert auf mindestens 1 gesetzt und in eine Integer-Konvertierung umgewandelt, um config.max_file_size zu setzen.

2. def default_output_path_for_profile(target: str | Path, profile: str) -> Path

Diese Funktion berechnet den Standard-Ausgabepfad für einen Code-Dump basierend auf dem Zielpfad und dem gewählten Profil. Parameter:

  • target: str | Path: Der ursprüngliche Pfad, an dem der Dump erstellt werden soll.
  • profile: str: Das gewählte Dump-Profil (z.B. "standard").

    • Rückgabe:

  • Path: Der berechnete, vollständige Ausgabe-Pfad.

    • Beschreibung:

  1. Profil-Normalisierung: Der profile wird normalisiert (Entfernen von Leerzeichen, Kleinbuchstaben).
  2. Zielpfad-Analyse: Der target wird als Path-Objekt konvertiert und überprüft, ob es sich um ein Verzeichnis handelt (_is_directory_like_target(target_path)).
  • Wenn es ein Verzeichnis ist:
  • Wenn das Profil "standard" ist, wird der Pfad mit CODEDUMP.md am Ende hinzugefügt.
  • Andere Profile fügen den Profilnamen als Suffix hinzu (z.B. CODEDUMP.compact.md).
  • **Wenn das Profil "standard" ist (und es kein Verzeichnis ist):**
  • Die Dateiendung wird auf .codedump.md geändert.
  • Andere Profile:
  • Der Pfad erhält den Namen des ursprünglichen Pfades (target.stem) und den Profilnamen als Suffix, gefolgt von .codedump.md (z.B. mein_projekt.compact.codedump.md).

Abhängigkeiten

  • Path: Ein Objekt aus dem pathlib-Modul (impliziert).
  • PROFILE_NAMES: Eine globale Struktur (impliziert), die die validierten Dump-Profile speichert.
