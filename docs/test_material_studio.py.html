Technische Dokumentation: test_material_studio.py

Version: 1.0 Datum: 2026-04-06 Autor: Ralf Krümmel

1. Überblick

Diese Datei (test_material_studio.py) enthält Unit-Tests für Komponenten, die mit dem Material Studio in Verbindung stehen, wahrscheinlich im Kontext eines Schulservers (nova_school_server). Die Datei implementiert Mocking- oder Fake-Runner- und KI-Klassen, um die Funktionalität von Hauptkomponenten zu isolieren und zu testen.

2. Struktur und Komponenten

Die Datei definiert drei Hauptklassen, die als Test-Skelette oder Mock-Implementierungen dienen:

  1. **_FakeRunner**: Simuliert einen Prozess-Runner, der Ergebnisse verarbeitet.
  2. **_FakeAI**: Simuliert eine Künstliche Intelligenz (KI)-Komponente, die Antworten liefert.
  3. Importierte Abhängigkeiten: Importiert notwendige Klassen und Konstanten aus anderen Modulen (SchoolRepository, TeacherMaterialStudioService, verschiedene Catalog-Funktionen und Resolver).

3. Detaillierte Analyse der Klassen

3.1. Klasse _FakeRunner

Diese Klasse dient dazu, die Logik eines Runner-Prozesses zu simulieren. Attribute:

  • self.results: Eine Liste von Ergebnissen (list[dict[str, object]]), die als Ergebnisse des Runners gespeichert werden.
  • self.payloads: Eine Liste von übergebenen Daten (list[dict[str, object]]), die während des Runners verarbeitet wurden.

    • Methoden:

  • **__init__(self, results: list[dict[str, object]]) -> None**:
  • Initialisiert die Klasse, indem die bereitgestellten Ergebnisse in self.results kopiert werden.
  • Initialisiert self.payloads als leere Liste.
  • **run_bundle(self, _session, payload: dict[str, object]) -> dict[str, object]**:
  • Fügt das übergebene payload zur Liste self.payloads hinzu.
  • Validierung: Prüft, ob self.results leer ist. Wenn ja, wirft eine AssertionError, da keine weiteren Ergebnisse verfügbar sind.
  • Rückgabe: Entfernt das erste Element aus self.results (self.results.pop(0)) und gibt es als Ergebnis zurück.

    • Zweck: Testet die korrekte Verarbeitung von Eingabedaten (payload) und die Extraktion von Ergebnissen aus einer Ergebnisquelle (results).

3.2. Klasse _FakeAI

Diese Klasse simuliert eine Schnittstelle zu einem KI-Modell, das Antworten generiert. Attribute:

  • self.responses: Eine Liste von erwarteten Antworten (list[object]), die als Antworten des KI-Modells gespeichert werden.
  • self.calls: Eine Liste von Aufrufen (list[dict[str, object]]), die die tatsächlichen Anfragen an das KI-Modell darstellen.

    • Methoden:

  • **__init__(self, responses: list[object]) -> None**:
  • Initialisiert die Klasse mit einer Liste von erwarteten Antworten.
  • Initialisiert self.calls als leere Liste.
  • **complete(self, **kwargs) -> tuple[str, str]**:
  • Fügt die übergebenen Schlüssel-Wert-Paare (kwargs) zur Liste self.calls hinzu.
  • Validierung: Prüft, ob self.responses leer ist. Wenn ja, wirft eine AssertionError.
  • Antwort-Extraktion: Nimmt die erste Antwort aus self.responses (self.responses.pop(0)).
  • Fehlerbehandlung (Exception): Wenn die Antwort eine Exception ist, wird diese direkt zurückgeworfen.
  • Formatvalidierung: Prüft, ob die Antwort entweder ein Tupel von zwei Elementen ((text, model)) ist oder eine Exception ist. Fällt dies nicht, wird eine AssertionError geworfen.
  • Rückgabe: Gibt die extrahierte Antwort zurück.

    • Zweck: Testet die korrekte Protokollierung von KI-Anfragen und die korrekte Rückgabe von KI-Antworten, einschließlich der Behandlung von Fehlern.

4. Abhängigkeiten und Kontext

Die Klasse ist stark von folgenden externen Modulen abhängig:

  • **nova_school_server.database.SchoolRepository**: Wahrscheinlich für den Zugriff auf Schuldaten.
  • **nova_school_server.material_studio**: Enthält die Kernlogik des Material Studio, einschließlich:
  • SUPPORTED_LANGUAGES
  • TeacherMaterialStudioService
  • material_studio_instruction_preset_catalog
  • material_studio_profile_catalog
  • resolve_material_studio_instruction_preset (Funktion zum Auflösen von Anweisungen).

5. Zusammenfassung und Testzweck

Die Datei dient primär dazu, Unit-Tests für die Schnittstellenlogik von Material Studio zu erstellen. Durch die Verwendung von _FakeRunner und _FakeAI können Entwickler die Kernfunktionen des Material Studio-Services isoliert testen, ohne auf komplexe Datenbank- oder tatsächliche KI-Infrastrukturen angewiesen zu sein. Schlüsselthemen im Test:

  1. Datenfluss-Validierung: Sicherstellen, dass Daten korrekt von einem Runner zum nächsten fließen.
  2. Fehlerbehandlung: Testen, wie die Klassen auf leere Ergebnislisten oder ungültige KI-Antworten reagieren.
  3. Protokollierung: Überprüfung, ob Aufrufe und Ergebnisse korrekt in die jeweiligen Listen protokolliert werden.

Technische Dokumentation: test_material_studio.py

1. Überblick

Diese Datei enthält eine Reihe von Python-Klassen, die scheinbar Teile einer Simulation oder eines Testframeworks für einen "Material Studio" darstellen. Die Klassen _FakeCurriculumService und _TeacherSession dienen dazu, spezifische Geschäftslogiken oder Zustände zu simulieren, insbesondere im Kontext von Präferenzen/Presets und Benutzerrollen.

2. Klassen-Analyse

2.1. Klasse _FakeCurriculumService

Diese Klasse simuliert einen Dienst, der Anweisungen oder Präferenzen basierend auf einem bereitgestellten Preset auflöst. Attribute:

  • self.preset: dict[str, object] | None: Ein optionales Dictionary, das die Konfiguration (Preset) enthält, das zur Auflösung verwendet werden kann.
  • self.calls: list[dict[str, str]]: Eine Liste, die alle aufgerufenen Anweisungen (Calls) speichert. Jedes Element in der Liste ist ein Dictionary mit den Schlüsseln "preset_key", "profile" und "language".

    • Methoden:

  • **resolve_material_studio_instruction_preset(self, preset_key: str, *, profile: str, language: str) -> dict[str, object] | None**
  • Beschreibung: Diese Methode versucht, ein Material Studio-Instruktions-Preset basierend auf den übergebenen Schlüssel, Profil und Sprache zu finden und zurückzugeben.
  • Parameter:
  • preset_key: str: Der eindeutige Schlüssel des gesuchten Presets.
  • profile: str: Das zugehörige Profil.
  • language: str: Die gewünschte Sprache.
  • Logik:
  1. Fügt die aktuellen Aufrufparameter (preset_key, profile, language) zur internen Liste self.calls hinzu.
  2. Prüft, ob das gespeicherte self.preset existiert und ob die Schlüssel ("key", "profile", "language") des gespeicherten Presets exakt mit den übergebenen Werten übereinstimmen.
  3. Erfolg: Wenn die Übereinstimmung zutrifft, wird das gesamte gespeicherte self.preset als Dictionary zurückgegeben.
  4. Fehler/Nicht-Treffer: Andernfalls wird None zurückgegeben.

2.2. Klasse _TeacherSession

Diese Klasse definiert die grundlegenden Eigenschaften und Berechtigungen einer simulierten Lehrer-Sitzung. Attribute:

  • username: str: Der Standard-Benutzername, gesetzt auf "teacher".
  • role: str: Die Standardrolle, gesetzt auf "teacher".
  • is_teacher: bool: Der Status, gesetzt auf True, um die Rolle als Lehrer zu bestätigen.
  • permissions: dict[str, bool]: Ein Dictionary, das die Berechtigungen des Benutzers definiert.
  • "ai.use": True: Berechtigung zur Nutzung von KI-Funktionen.
  • "teacher.materials.use": True: Berechtigung zur Nutzung von Lehrer-Materialien.
  • "run.python": True: Berechtigung zur Ausführung von Python-Code.
  • "web.access": False: Berechtigung zum Webzugriff ist deaktiviert.

3. Zusammenfassung und Verwendungshinweise

Die Datei implementiert zwei isolierte Komponenten:

  1. **Dienst-Simulation (_FakeCurriculumService):** Dient der Abbildung eines Dienstes, der Konfigurationen (Presets) nach spezifischen Parametern abfragt.
  2. **Zustandsmodell (_TeacherSession):** Dient der Definition eines Benutzerprofils mit festen Rollen und Berechtigungen.

    3. Anwendungsfall: Diese Klassen sind typisch für Unit-Tests oder Mocking-Setups, um zu verifizieren, wie ein System mit Benutzerkonfigurationen und Dienstaufrufen interagiert.

4. Abhängigkeiten

Die Klasse _FakeCurriculumService verwendet Standard-Python-Typen (dict, list, str, object). Es sind keine externen Bibliotheken erforderlich.

Technische Dokumentation: test_material_studio.py

1. Überblick

Diese Datei enthält eine Reihe von Unit-Tests für die Klasse MaterialStudioTests aus dem material_studio-Modul. Die Tests validieren die Funktionalität verschiedener Komponenten des Material-Studios, darunter:

  1. Profilkatalog: Überprüfung, ob spezifische Lehrerprofile (z.B. example-code, worksheet) und deren Schlüssel vorhanden sind.
  2. Anweisungs-Preset-Katalog: Validierung der Struktur und des Inhalts des Katalogs für verschiedene Sprachen und Profile.
  3. Curriculum-Service-Interaktion: Testen, wie der TeacherMaterialStudioService Anweisungs-Presets über den CurriculumService überschreiben kann.
  4. JSON-Parsing: Überprüfung der Fähigkeit der Methode _extract_json_object, um verschachtelte JSON-Strukturen aus Markdown-Formatierungen zu extrahieren.
  5. Code-Block-Extraktion: Validierung der Robustheit der Methode _extract_code_block bei verschiedenen Formaten (eingeschlossene Fences, ungeschlossene Fences, reine Code-Strings).
  6. Code-Response-Parsing: Testen, wie die Methode _parse_code_response Code-Antworten verarbeitet, einschließlich der Ablehnung von selbstevaluierendem Prosa.

2. Test-Setup und -Teardown

2.1. setUp(self) -> None

  • Zweck: Initialisiert die Testumgebung.
  • Aktionen:
  • Erstellt ein temporäres Verzeichnis (self.tmp) für die Datenbankdatei.
  • Initialisiert eine SchoolRepository mit der temporären Datenbankdatei (self.tmp.name / "school.db").

2.2. tearDown(self) -> None

  • Zweck: Reinigt die Ressourcen nach jedem Test.
  • Aktionen:
  • Schließt die SchoolRepository.
  • Führt die Bereinigung des temporären Verzeichnisses (self.tmp.cleanup()) durch.

3. Testfälle (Test Cases)

3.1. test_profile_catalog_contains_specialized_teacher_profiles

  • Ziel: Sicherstellen, dass der Katalog aller verfügbaren Lehrerprofile die erwarteten Schlüssel enthält.
  • Erwartungen: Der Schlüssel für example-code, worksheet, assessment, board-lesson und differentiation muss im resultierenden Schlüssel-Set enthalten sein.

3.2. test_instruction_preset_catalog_contains_python_course_entries_for_worksheet_and_example_code

  • Ziel: Validiert die spezifische Struktur und die korrekte Resolution von Anweisungs-Presets für die Sprache Python.
  • Erwartungen:
  • Der Katalog muss für worksheet und example-code Einträge enthalten.
  • Alle unterstützten Sprachen (SUPPORTED_LANGUAGES) müssen im Katalog vorhanden sein.
  • Resolutionstests:
  • Die Resolution eines worksheet-Presets für Python muss erfolgreich sein.
  • Die Resolution eines example-code-Presets für Python muss erfolgreich sein und den erwarteten Prompt ("kompaktes, direkt ausfuehrbares Python-Beispiel") enthalten.
  • Die Resolution eines worksheet-Presets für Python als example-code muss None zurückgeben.

3.3. test_instruction_preset_catalog_covers_every_supported_language_with_both_profiles

  • Ziel: Überprüfen, dass für jede unterstützte Sprache sowohl worksheet- als auch example-code-Presets existieren.
  • Erwartungen:
  • Für jede Sprache muss mindestens ein Preset für worksheet und ein Preset für example-code existieren.
  • Spezifische Inhaltserwartungen:
  • Ein example-code-Preset für rust muss den Prompt "Rust-Grundkurs" und "println!" enthalten.
  • Ein worksheet-Preset für node muss den Prompt "Node.js-Grundkurs" und "keine DOM- oder Browser-APIs" enthalten.

3.4. test_curriculum_service_can_override_instruction_preset_resolution

  • Ziel: Testen, ob der TeacherMaterialStudioService die Resolution von Anweisungs-Presets durch eine externe CurriculumService-Konfiguration überschreiben kann.
  • Erwartungen:
  • Ein spezifisches Preset (bundle-python) wird über den CurriculumService bereitgestellt.
  • Die Methode _instruction_preset_from_state muss den Wert des label (z.B. "Bundle Python") korrekt aus dem Curriculum-Objekt zurückgeben.

3.5. test_extract_json_object_merges_multiple_fenced_objects

  • Ziel: Validiert die Fähigkeit von _extract_json_object, mehrere JSON-Objekte, getrennt durch Fences, zu verarbeiten und zu fusionieren.
  • Erwartungen: Die extrahierte Payload muss sowohl teacher_notes_markdown als auch student_material_markdown korrekt aus den beiden separaten JSON-Blöcken extrahieren.

3.6. test_extract_json_object_accepts_jsonish_string_concatenation

  • Ziel: Testet die Robustheit des Parsers bei JSON-Strukturen, die über Zeilenumbrüche oder Kommentare (wie in einem Markdown-Block) verbunden sind.
  • Erwartungen: Die extrahierte Liste (student_material_markdown) muss die Inhalte der beiden JSON-Objekte korrekt als eine einzige Liste zusammenfügen.

3.7. test_extract_code_block_prefers_nested_language_block_over_outer_text_wrapper

  • Ziel: Testet die Priorität der Code-Block-Extraktion.
  • Erwartungen: Wenn ein Code-Block durch äußere Text-Wrapper (z.B. ```text ... ``) umschlossen ist, muss der Parser den Inhalt des inneren, spezifischen Code-Fences (z.B. ``python ... ```) bevorzugen und nur den reinen Code zurückgeben.

3.8. test_extract_code_block_recovers_trailing_code_from_unclosed_fence

  • Ziel: Testet die Fehlerbehandlung bei ungeschlossenen Code-Fences.
  • Erwartungen: Bei einem ungeschlossenen ```python Block muss der Parser den gesamten nachfolgenden Text, der als Code interpretiert werden kann, korrekt extrahieren.

3.9. test_parse_code_response_accepts_plain_python_code_without_fence

  • Ziel: Testet die Fähigkeit von _parse_code_response, reinen Python-Code, der keinen Fence enthält, zu verarbeiten.
  • Erwartungen: Der zurückgegebene Code muss exakt den ursprünglichen String (nach Entfernen von Whitespace) wiedergeben.

3.10. test_parse_code_response_rejects_self_evaluation_prose

  • Ziel: Testet die Sicherheits-/Validierungsfunktion, die verhindert, dass selbstevaluierende Prosa als Code interpretiert wird.
  • Erwartungen: Wenn die Eingabe Text ist, der selbstbewertend ist, muss eine ValueError ausgelöst werden.

3.11. test_parse_code_response_rejects_python_main_without_invocation

  • Ziel: Testet die Validierung, dass Code-Antworten eine ausführbare Struktur (z.B. eine Funktion oder einen Aufruf) benötigen.
  • Erwartungen: Wenn der Code nur eine einfache main-Funktion ohne tatsächlichen Aufruf enthält, muss eine ValueError ausgelöst werden.

4. Zusammenfassung und Fazit

Die Testsuite deckt die Kernfunktionen des MaterialStudioService umfassend ab, insbesondere in den Bereichen Datenmodellierung (Kataloge), Datenmanipulation (JSON-Parsing) und Text-/Code-Verarbeitung (Code-Block-Extraktion und Parsing). Die Tests sind gut strukturiert, um sowohl die korrekte Datenintegrität als auch die Robustheit gegenüber unvorhergesehenen Markdown-Formaten zu gewährleisten.
