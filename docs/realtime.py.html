Technische Dokumentation: realtime.py

1. Überblick

Die Datei realtime.py dient als zentraler Knotenpunkt für die Implementierung von Echtzeit-Kommunikations- und Prozessmanagement-Funktionen innerhalb des Systems. Sie orchestriert wahrscheinlich die Kommunikation mit externen Prozessen (z.B. durch PTYs), die Verwaltung von Sitzungszuständen (SessionContext), die Interaktion mit einer Datenbank (SchoolRepository) und die Durchführung von Code-Ausführungen (CodeRunner). Die Klasse, die in diesem Ausschnitt definiert wird, scheint die Hauptlogik für die Echtzeit-Verarbeitung und die Verbindung zwischen verschiedenen Komponenten zu beherbergen.

2. Importierte Module und Abhängigkeiten

Die Datei importiert eine Vielzahl von Modulen, die auf spezifische Funktionen im System zugreifen:

  • **__future__**: Für zukünftige Python-Syntax-Funktionen.
  • **base64, codecs, hashlib, json**: Standardmodulen für Datenkodierung, Hashing und JSON-Serialisierung.
  • **socket**: Für Netzwerkkommunikation.
  • **struct**: Für die Serialisierung/Deserialisierung von binären Daten (wahrscheinlich für Protokoll- oder Prozessdaten).
  • **subprocess**: Für das Ausführen externer Befehle oder Prozesse.
  • **threading, time**: Für Multithreading und Zeitsteuerung.
  • **uuid**: Zur Generierung eindeutiger Identifikatoren.
  • **dataclasses**: Für die Definition von Datenstrukturen (dataclass).
  • **http.HTTPStatus**: Für die Darstellung von HTTP-Statuscodes.
  • Interne Module:
  • **.auth.SessionContext**: Wahrscheinlich für die Verwaltung von Benutzer- oder Sitzungsdaten.
  • **.code_runner.CodeRunner, LivePreparedRun**: Für die Ausführung und Überwachung von Code.
  • **.database.SchoolRepository**: Für die Abfrage und Speicherung von Schuldaten.
  • **.pty_host.PtyProcess, create_pty_process, normalize_terminal_size**: Für die Verwaltung von Pseudo-Terminal-Prozessen (PTY) und Terminal-Größenanpassungen.

Technische Dokumentation: WebSocketConnection Klasse

1. Überblick

Die Klasse WebSocketConnection ist eine Klasse, die die Kommunikation über eine einzelne WebSocket-Verbindung (repräsentiert durch eine socket.socket) abstrahiert. Sie implementiert die notwendigen Protokollfunktionen, um Daten (JSON-Payloads und Text) zwischen dem Client und dem WebSocket-Server zu senden und zu empfangen, basierend auf dem WebSocket-Protokoll. Die Klasse beinhaltet Logik für das Parsen von WebSocket-Frames (Header, Opcode, Payload-Größe) und die korrekte Formatierung von Frames für Senden.

2. Klasse: WebSocketConnection

2.1. Konstruktor (__init__)

Initialisiert die Verbindung und stellt die notwendigen internen Zustände und Locks bereit. Parameter:

  • sock: socket.socket: Die zugrundeliegende Socket-Objekt, über die die WebSocket-Verbindung läuft.

    • Aktionen:

  1. Speichert die zugrundeliegende Socket.
  2. Setzt den Timeout der Socket auf None, um eine unbegrenzte Lesezeit zu ermöglichen.
  3. Initialisiert einen threading.Lock (_send_lock) zur Synchronisation bei Schreibvorgängen.
  4. Initialisiert einen threading.Event (_closed), um den geschlossenen Zustand der Verbindung zu signalisieren.

    5. Attribute:

  • self.sock: Die aktive Socket-Objekt.
  • self._send_lock: Ein Lock zur Sicherstellung der atomaren Sendeoperationen.
  • self._closed: Ein Event, das den geschlossenen Zustand der Verbindung steuert.

2.2. Methoden

2.2.1. send_json(self, payload: dict[str, Any]) -> None

Sendet einen JSON-Payload über die WebSocket-Verbindung. Parameter:

  • payload: dict[str, Any]: Das zu sendende Python-Dictionary.

    • Beschreibung: Konvertiert das Python-Dictionary in eine JSON-String-Repräsentation und sendet diesen String über die interne send_text-Methode.

2.2.2. send_text(self, text: str) -> None

Sendet eine reine Textnachricht über das WebSocket. Parameter:

  • text: str: Der Textstring, der gesendet werden soll.

    • Beschreibung: Verpackt den Text in ein WebSocket-Frame mit Opcode 0x1 (Text Frame) und sendet ihn.

2.2.3. recv_json(self) -> dict[str, Any]

Empfängt eine Nachricht und versucht, diese als JSON-Dictionary zu parsen. Beschreibung: Rufen recv_text() auf, um die Rohdaten zu erhalten, und versucht anschließend, diese Daten zu einem Python-Dictionary zu deserialisieren.

2.2.4. recv_text(self) -> str

Empfängt eine Rohdaten-Nachricht (Text Frame) und dekodiert sie. Beschreibung: Wartet auf das nächste WebSocket-Frame, parst den Opcode und den Payload, und dekodiert den Payload in einen UTF-8-String. Fehlerbehandlung:

  • **UnicodeDecodeError:** Wird ausgelöst, wenn der empfangene Payload nicht korrekt als UTF-8 dekodiert werden kann.
  • **Opcode 0x8 (Close Frame):** Führt die Verbindung sofort ab und wirft eine ConnectionError.
  • Ungültiger Opcode: Wirft eine ConnectionError, falls ein nicht unterstützter Opcode empfangen wird.

2.2.5. close(self, code: int = 1000, reason: str = "") -> None

Schließt die WebSocket-Verbindung sauber. Parameter:

  • code: int = 1000: Der WebSocket-Abstimmungs-Code (z.B. 1000 für Normalgeschlossen).
  • reason: str = "": Der Begründungstext für den Abschluss.

    • Aktionen:

  1. Setzt den internen Zustand _closed auf True.
  2. Packt den Abschluss-Code und den Begründungstext in ein WebSocket-Control-Frame (Opcode 0x8) und sendet ihn.
  3. Schaltet die Socket auf SHUT_RDWR (Read/Write) um.
  4. Schließt die Socket (sock.close()).

2.2.6. accept_key(self, key: str) -> str (Statische Methode)

Generiert einen kryptografisch gesicherten Schlüssel basierend auf einem Eingabeschlüssel. Parameter:

  • key: str: Der Eingabeschlüssel.

    • Beschreibung: Berechnet einen SHA1-Hash der Kombination aus key und einer definierten Magic-String (WS_MAGIC) und kodiert das Ergebnis als Base64-String.

2.2.7. _recv_frame(self) -> tuple[int, bytes] (Private Methode)

Die Kernlogik zum Parsen eines einzelnen WebSocket-Frames. Beschreibung:

  1. Ruft _recv_exact(2) auf, um den Header zu lesen.
  2. Extrahiert Opcode, FIN-Flag, Reserved-Flag und Mask-Flag aus dem Header.
  3. Validierungsprüfungen: Überprüft, ob Extensions-Bits gesetzt sind, ob das Frame fragmentiert ist, ob es maskiert ist und ob die Payload-Größe korrekt interpretiert wird (inkl. Handling von 126-Byte und 127-Byte-Payload-Längen).
  4. Masking: Wenn das Frame maskiert ist, wird der Mask-Byte-Block (_recv_exact(4)) verwendet, um die Payload zu entschlüsseln (XOR-Operation).
  5. Gibt den Opcode und die entschlüsselte Payload zurück.

2.2.8. _send_frame(self, opcode: int, payload: bytes) -> None (Private Methode)

Formatiert und sendet ein WebSocket-Frame. Parameter:

  • opcode: int: Der WebSocket-Opcode (z.B. 0x1 für Text).
  • payload: bytes: Die zu sendende Nutzlast.

    • Beschreibung:

  1. Berechnet die notwendige Header-Struktur basierend auf der Länge der Payload (126, 127 oder eine feste Länge).
  2. Fügt den Header und die Payload zusammen.
  3. Sendet die kombinierten Daten über self.sock.sendall().
  4. Nutzt einen threading.Lock (self._send_lock), um die Sendeoperation zu synchronisieren.

2.2.9. _recv_exact(self, size: int) -> bytes (Private Methode)

Stellt sicher, dass exakt size Bytes von der Socket gelesen werden. Parameter:

  • size: int: Die gewünschte Anzahl der Bytes.

    • Beschreibung: Wartet, bis die gesuchte Anzahl an Bytes empfangen wurde. Nutzt self.sock.recv() und behandelt TimeoutError oder leere Rücksendungen als Verbindungsschluss.

3. Protokoll-Details (WebSocket Frame Types)

Die Klasse unterstützt die Verarbeitung der folgenden grundlegenden Frame-Typen:

Opcode (Hex) Typ Beschreibung
0x1 Text Frame Für den Austausch von Textdaten (nutzt send_json/recv_json).
0x8 Close Frame Signalisiert den Abschluss der Verbindung (nutzt close).
0xA Continuation Frame Wird verwendet, um eine Nachricht in mehrere Frames zu zerlegen.
0x9 Write Binary Frame Wird verwendet, um binäre Daten zu senden (nutzt _send_frame).
0xA Ping Frame Wird verwendet, um die Verbindung zu überprüfen (impliziert in _recv_frame).

4. Abhängigkeiten

Die Klasse benötigt die folgenden Standard-Python-Module:

  • socket: Für die Netzwerkkommunikation.
  • threading: Für die Synchronisation (Lock, Event).
  • json: Für die Serialisierung und Deserialisierung von JSON-Daten.
  • struct: Für die Packung und Unpackung von binären Daten (Header-Größen).
  • hashlib: Für die SHA1-Hashing-Funktion.
  • base64: Für die Base64-Kodierung/Dekodierung.
  • dataclasses: Für die Verwendung von @dataclass (in der Definition).

5. Sicherheits- und Robustheitsaspekte

  1. Timeouts: Die Socket-Leseoperationen sind auf None gesetzt, was eine kontinuierliche Erwartung der Daten ermöglicht, aber die Anwendung muss Mechanismen zur Timeout-Handhabung bereitstellen (was durch _recv_exact abgedeckt wird).
  2. Data Integrity: Die Protokollprüfung in _recv_frame stellt sicher, dass nur gültige WebSocket-Header und Frame-Größen verarbeitet werden.
  3. Encoding Safety: Es wird explizit auf utf-8 dekodiert, und Fehler werden bei der Dekodierung von Text-Payloads explizit abgefangen (ConnectionError).
  4. Concurrency Safety: Der _send_lock stellt sicher, dass mehrere Threads nicht gleichzeitig versuchen, Daten über die Socket zu senden, was zu Datenkorruption führen würde.
  5. Resource Management: Die close-Methode stellt sicher, dass die Socket korrekt geschlossen wird, selbst bei Fehlern während des Sendevorgangs.

Technische Dokumentation: RealtimeService Klasse

1. Überblick

Die Klasse RealtimeService ist verantwortlich für die Verwaltung der Echtzeit-Verbindungen (Sockets) zu Benutzern, die mit einer Anwendung interagieren. Sie orchestriert die Kommunikation zwischen den Echtzeit-Clients (RealtimeClient), der Anwendung selbst (application) und dem Live-Run-Manager (LiveRunManager). Die Klasse bietet Mechanismen zur Registrierung, Deaktivierung von Verbindungen, dem Handling von Echtzeit-Nachrichten und dem gleichzeitigen Senden von Updates an andere Clients innerhalb desselben Projekts.

2. Klasse: RealtimeService

2.1. Konstruktor (__init__)

Initialisiert die Service-Instanz und stellt die notwendigen Abhängigkeiten bereit. Parameter:

  • application: Any: Eine Instanz der Hauptanwendung, die Zugriff auf Funktionen wie runner, repository und collaboration bietet.

    • Attribute:

  • self.application: Referenz auf die übergegebene Anwendung.
  • self.live_runs: Eine Instanz von LiveRunManager, initialisiert mit dem Runner und dem Repository der Anwendung.
  • self._lock: Ein threading.RLock (Reentrant Lock), der für den sicheren Zugriff auf den internen Zustand (_project_clients) verwendet wird.
  • self._project_clients: Ein Dictionary, das die Echtzeit-Clients organisiert.
  • Struktur: dict[str, dict[str, RealtimeClient]]
  • Schlüssel: Der eindeutige ID des Projekts (project_id als String).
  • Wert: Ein weiteres Dictionary, das die Clients für dieses Projekt enthält.
  • Schlüssel: Die eindeutige Client-ID (client_id).
  • Wert: Der RealtimeClient-Objekt.

2.2. Methoden

2.2.1. close(self) -> None

Stellt einen sauberen Abschluss aller laufenden Echtzeit-Verbindungen und Live-Runs sicher. Funktion:

  1. Ruft self.live_runs.close() auf, um alle laufenden Live-Runs zu beenden.
  2. Sperrt den Zugriff auf _project_clients (self._lock).
  3. Sammelt alle RealtimeClient-Instanzen aus allen Projekten.
  4. Leert das gesamte self._project_clients Dictionary.
  5. Schließt die Verbindung (connection.close()) jedes gesammelten Clients.

2.2.2. handle_project_socket(self, connection: WebSocketConnection, session: SessionContext, project: dict[str, Any]) -> None

Die Hauptmethode, die eine neue WebSocket-Verbindung verarbeitet und den Client initialisiert. Parameter:

  • connection: WebSocketConnection: Die aktive WebSocket-Verbindung.
  • session: SessionContext: Kontextinformationen über die Sitzung des Benutzers (z.B. Benutzername, Berechtigungen).
  • project: dict[str, Any]: Die spezifischen Daten des Projekts, zu dem der Client gehört.

    • Ablauf:

  1. Client-Initialisierung: Erstellt einen neuen RealtimeClient mit einer zufälligen ID, der Verbindung und den Session-Kontext.
  2. Registrierung: Ruft self._register(client) auf, um den Client zu speichern.
  3. Initialisierung der Verbindung: Sendet eine Willkommensnachricht ({"type": "hello", ...}) an den Client.
  4. Initialer State (Optional): Wenn die Berechtigung "notebook.collaborate" vorhanden ist, holt er den aktuellen Zustand des Projekts (snapshot) und sendet ihn an den Client ({"type": "collab.state", ...}).
  5. Haupt-Loop: Startet eine Schleife, die kontinuierlich Nachrichten von der Verbindung empfängt (connection.recv_json()).
  • Jede empfangene Nachricht wird an self._handle_message(client, message) weitergeleitet.
  • Fehlerbehandlung: Fängt Ausnahmen während des Nachrichtenverarbeitens ab und sendet eine Fehlermeldung zurück.
  1. **Aufräumphase (finally Block):** Unabhängig vom Ende der Schleife (sei es durch einen Fehler oder einen manuellen Abbruch):
  • Ruft self.live_runs.stop_for_client(client.client_id) auf, um den Live-Run für diesen Client zu beenden.
  • Ruft self._unregister(client) auf, um den Client aus der Registry zu entfernen.
  • Schließt die WebSocket-Verbindung.

2.2.3. _handle_message(self, client: RealtimeClient, message: dict[str, Any]) -> None

Verarbeitet die vom Client empfangenen Echtzeit-Aktionen basierend auf dem action-Feld der Nachricht. Logik (Aktionen):

Aktion (action) Beschreibung Abhängigkeiten/Aktionen
"ping" Antwortet auf eine Ping-Anfrage. Sendet "pong" mit aktuellen Zeitstempel.
"collab.presence" Aktualisiert die Präsenz/Heartbeat-Informationen. Prüft Berechtigung (notebook.collaborate). Ruft application.collaboration.heartbeat() auf und sendet das Ergebnis an alle Clients des Projekts (_broadcast_project).
"collab.sync" Synchronisiert den Zustand (z.B. Code-Zellen). Prüft Berechtigung. Ruft application.collaboration.sync() auf. Sendet den synchronisierten Zustand an alle Clients (_broadcast_project).
"run.start" Startet einen Live-Run. Ruft self.live_runs.start() auf mit den Payload-Daten.
"run.stdin" Sendet Eingabe in einen laufenden Live-Run. Überprüft session_id. Ruft self.live_runs.send_input() auf.
"run.resize" Ändert die Größe des Terminalfensters. Überprüft session_id. Ruft self.live_runs.resize() auf.
"run.stop" Beendet einen laufenden Live-Run. Überprüft session_id. Ruft self.live_runs.stop() auf.
Unbekannt Fehlerfall. Löst eine ValueError aus.

2.2.4. _broadcast_project(self, project_id: str, payload: dict[str, Any]) -> None

Sendet eine Nachricht (payload) an alle Clients, die diesem spezifischen Projekt zugeordnet sind. Sicherheit: Der Zugriff auf die Clients erfolgt unter Verwendung des internen Locks (self._lock), um Race Conditions zu verhindern.

2.2.5. _register(self, client: RealtimeClient) -> None

Fügt einen neuen RealtimeClient zur internen Registry hinzu. Logik:

  1. Sperrt den Zugriff.
  2. Setzt den client in das Dictionary unter dem project_id des Clients.
  3. Falls das Projekt-Dictionary für diesen project_id neu ist, wird es erstellt.

2.2.6. _unregister(self, client: RealtimeClient) -> None

Entfernt einen RealtimeClient aus der Registry. Logik:

  1. Bestimmt den project_id des Clients.
  2. Sperrt den Zugriff.
  3. Entfernt den Eintrag des Clients aus dem Projekt-Bucket.
  4. Optional: Wenn das Projekt-Bucket leer wird, wird der gesamte Eintrag für diesen project_id aus self._project_clients entfernt.

2.2.7. _require_permission(self, session: SessionContext, key: str) -> None

Eine Hilfsmethode zur Überprüfung der Benutzerberechtigungen. Parameter:

  • session: SessionContext: Der Kontext des Benutzers.
  • key: str: Der Schlüssel der benötigten Berechtigung (z.B. "notebook.collaborate").

    • Verhalten:

  • Prüft, ob session.permissions den Schlüssel key mit True oder einem entsprechenden Wert enthält.
  • Löst bei fehlender Berechtigung eine PermissionError aus.

3. Abhängigkeiten und Schlüsselkonzepte

  • **WebSocketConnection**: Die Schnittstelle zur physischen Netzwerkverbindung.
  • **SessionContext**: Metadaten über den aktuell verbundenen Benutzer (z.B. Benutzername, Berechtigungen).
  • **RealtimeClient**: Repräsentiert einen einzelnen, aktiven Client, der eine Verbindung hält und Nachrichten empfängt/sendet.
  • **LiveRunManager**: Verantwortlich für das Management von Live-Sessions (Starten, Stopper, Eingabe/Resize).
  • **application**: Der zentrale Dienst, der die Geschäftslogik (Kollaboration, Runner) bereitstellt.
  • Threading: Nutzt threading.RLock zur Sicherstellung der Datenintegrität bei gleichzeitigen Zugriffsoperationen auf _project_clients.

4. Sicherheit und Fehlerbehandlung

  • Zugriffskontrolle: Berechtigungen (_require_permission) werden vor sensiblen Aktionen (collab.presence, collab.sync) immer geprüft.
  • Robustheit: Die Hauptverarbeitung in handle_project_socket verwendet umfassende try...except...finally-Blöcke, um sicherzustellen, dass Verbindungen und Ressourcen (Live-Runs) immer ordnungsgemäß beendet werden, selbst bei Netzwerkfehlern oder Verarbeitungsproblemen.
  • Validierung: Spezifische Validierungen werden für Befehle wie "run.stdin" und "run.resize" durchgeführt (z.B. Prüfung auf fehlende session_id).
  • Fehlermeldungen: Fehler, die während der Verarbeitung auftreten, werden entweder an den Client zurückgesendet oder im internen Log (durch die try/except Blöcke) behandelt.

Technische Dokumentation: def upgrade_websocket(handler: Any) -> WebSocketConnection

1. Übersicht

Die Funktion upgrade_websocket ist eine Methode, die dazu dient, eine HTTP-Anfrage (repräsentiert durch ein handler-Objekt) auf eine WebSocket-Verbindung umzuwandeln. Sie implementiert den Standardprozess für den WebSocket-Upgrade-Header-Austausch, wie er in HTTP-Protokollen definiert ist.

2. Signatur

def upgrade_websocket(handler: Any) -> WebSocketConnection:
# ... Implementierung ...
  • Name: upgrade_websocket
  • Parameter:
  • handler: Any: Ein Objekt, das die HTTP-Anfrage-Informationen enthält (z.B. ein HTTP-Request-Objekt). Es muss Header-Zugriff und Methoden wie send_response, send_header, end_headers, und close_connection besitzen.
  • Rückgabe:
  • WebSocketConnection: Ein Objekt, das die neu etablierte WebSocket-Verbindung repräsentiert.

3. Funktionsbeschreibung

Die Funktion führt die notwendigen Schritte aus, um den HTTP-Header-Austausch für eine WebSocket-Verbindung zu initialisieren:

  1. Upgrade-Header-Validierung: Sie prüft, ob der Upgrade-Header in den Anfrage-Headern auf "websocket" (kleingeschrieben) gesetzt ist.
  2. WebSocket-Key-Extraktion: Sie extrahiert den Sec-WebSocket-Key aus den Headern.
  3. Fehlerbehandlung (Validierung): Sie stellt sicher, dass sowohl der Upgrade-Header als auch der Sec-WebSocket-Key vorhanden sind.
  4. HTTP-Response-Setzung: Sie sendet die entsprechenden HTTP-Antwort-Header an den Client:
  • HTTPStatus.SWITCHING_PROTOCOLS: Gibt an, dass der Protokollwechsel initiiert wird.
  • Upgrade: websocket: Gibt den gewünschten Protokollwechsel an.
  • Connection: Upgrade: Gibt an, dass die Verbindung für den Upgrade-Prozess verwendet wird.
  • Sec-WebSocket-Accept: Setzt den Antwort-Key basierend auf dem eingehenden Sec-WebSocket-Key mithilfe der Methode WebSocketConnection.accept_key(key).
  1. Header-Ende: Sie ruft handler.end_headers() auf, um die Header-Senden abzuschließen.
  2. Verbindungseinstellung: Sie setzt die Verbindung des Handlers auf close_connection = True, um die Verbindung nach dem Upgrade zu schließen.
  3. Status-Setzung: Sie setzt ein internes Attribut _websocket_upgraded auf True, um den Status der Verbindung zu markieren.
  4. Rückgabe: Sie erstellt und gibt eine neue Instanz von WebSocketConnection zurück, die die zugrunde liegende Verbindung des Handlers verwaltet.

4. Abhängigkeiten und Voraussetzungen

  • **handler:** Muss ein Objekt sein, das die notwendigen HTTP-Funktionen (Header-Setzung, Response-Senden) bereitstellt.
  • **HTTPStatus:** Muss eine Enum- oder Konstanten-Definition für HTTP-Statuscodes enthalten (z.B. HTTPStatus.SWITCHING_PROTOCOLS).
  • **WebSocketConnection:** Muss eine Klasse sein, die die Logik zur Generierung des Sec-WebSocket-Accept-Keys (accept_key(key)) und die Repräsentation der WebSocket-Verbindung bereitstellt.

5. Fehlerbehandlung (Exceptions)

Die Funktion wirft ValueError bei folgenden Bedingungen, wenn die notwendigen Header fehlen oder falsch sind:

  1. Falscher Upgrade-Header: Wenn der Upgrade-Header nicht exakt "websocket" (nach Entfernen von Leerzeichen und Kleinbuchstaben) ist.
  2. Fehlender Key: Wenn der Sec-WebSocket-Key-Header fehlt.

6. Use Case

Diese Funktion wird typischerweise im Kontext eines Web-Frameworks oder eines HTTP-Server-Handlers verwendet, um eine initiale HTTP-Anfrage, die eine WebSocket-Verbindung anfordert, in eine aktive WebSocket-Sitzung umzuwandeln.
