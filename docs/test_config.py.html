Technische Dokumentation: test_config.py

1. Überblick

Diese Datei enthält eine Reihe von Unit-Tests, die die Funktionalität der Konfigurationsverwaltung und -initialisierung in der Anwendung (vermutlich im Kontext von ServerConfig und runtime_config) validieren. Die Tests konzentrieren sich auf die korrekte Handhabung von Konfigurationsquellen (Standardwerte, Umgebungsvariablen, gespeicherte Payloads und Laufzeitkonfigurationen).

2. Struktur und Komponenten

Die Datei definiert eine Testklasse ConfigTests, die spezifische Szenarien für die Konfigurationslogik abdeckt. Wichtige Abhängigkeiten (impliziert):

  • ServerConfig: Eine Klasse, die die Konfiguration lädt und verwaltet.
  • tempfile: Für das Erstellen temporärer Verzeichnisse zur Simulation von Dateisystempfaden.
  • os: Für die Manipulation von Umgebungsvariablen.
  • Path: Für die Arbeit mit Dateipfaden.
  • save_server_config_payload und load_server_config_payload: Funktionen zur Speicherung und Wiederherstellung von Konfigurationsdaten.
  • stored_runtime_config und active_runtime_config: Funktionen zur Abfrage von gespeicherten und aktiven Laufzeitkonfigurationen.
  • runtime_config_requires_restart: Eine Funktion, die prüft, ob ein Neustart erforderlich ist.

3. Detaillierte Testfälle

Die folgenden Tests werden im Detail beschrieben:

3.1. test_default_host_binds_for_lan

Ziel: Überprüfen, dass die Standardkonfiguration den erwarteten Hostwert für lokale Netzwerke zurückgibt. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. ServerConfig.from_base_path() wird mit diesem Verzeichnis aufgerufen.
  3. Es wird verifiziert, dass das zurückgegebene config.host den Wert "0.0.0.0" hat.

3.2. test_env_host_override_is_respected

Ziel: Validieren, dass Umgebungsvariablen, die die Konfiguration überschreiben sollen (speziell NOVA_SCHOOL_HOST), korrekt angewendet werden. Ablauf:

  1. Der aktuelle Wert von NOVA_SCHOOL_HOST wird gespeichert.
  2. Die Umgebungsvariable wird temporär auf "127.0.0.1" gesetzt.
  3. ServerConfig.from_base_path() wird aufgerufen.
  4. Es wird verifiziert, dass config.host den Wert "127.0.0.1" hat.
  5. Cleanup: Die Umgebungsvariable wird auf ihren ursprünglichen Wert zurückgesetzt.

3.3. test_server_config_payload_is_saved_and_loaded

Ziel: Testen, dass die Speicherung und Wiederherstellung eines spezifischen Konfigurations-Payloads erfolgreich sind und die Daten korrekt übernommen werden. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Ein spezifisches Payload ({"host": "127.0.0.1", "port": 9999, "tenant_id": "school-x"}) wird mithilfe von save_server_config_payload gespeichert.
  3. Das gespeicherte Payload wird direkt verifiziert.
  4. Die Konfiguration wird mithilfe von load_server_config_payload aus dem Verzeichnis geladen.
  5. Es wird verifiziert, dass das tenant_id korrekt "school-x" ist.

3.4. test_stored_runtime_config_falls_back_to_active_values

Ziel: Sicherstellen, dass die gespeicherte Laufzeitkonfiguration (stored) mit der aktuell aktiven Laufzeitkonfiguration (active) konsistent ist, wenn keine Änderungen vorgenommen wurden. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Die ServerConfig wird initialisiert.
  3. Die Laufzeitkonfiguration wird gespeichert (stored_runtime_config).
  4. Die aktive Laufzeitkonfiguration wird abgerufen (active_runtime_config).
  5. Es wird verifiziert, dass stored["host"] == active["host"] und stored["port"] == active["port"].
  6. Es wird verifiziert, dass runtime_config_requires_restart(active, stored) falsch zurückgibt (kein Neustart nötig).

3.5. test_runtime_config_requires_restart_when_stored_values_differ

Ziel: Validieren, dass ein Neustart erforderlich ist, wenn die gespeicherten Laufzeitkonfigurationen von den aktiven abweichen. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Die ServerConfig wird initialisiert.
  3. Ein spezifisches Payload ({"port": 9988, "run_timeout_seconds": 45}) wird gespeichert.
  4. Die aktive Laufzeitkonfiguration wird abgerufen (active_runtime_config).
  5. Die gespeicherte Laufzeitkonfiguration wird abgerufen (stored_runtime_config).
  6. Es wird verifiziert, dass runtime_config_requires_restart(active, stored) wahr zurückgibt (Neustart erforderlich).

3.6. test_static_path_falls_back_to_real_package_root_for_flat_repo_layout

Ziel: Überprüfen, dass der Pfad zur statischen Ressource (static_path) korrekt berechnet wird, wenn die Repository-Struktur flach ist. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Die ServerConfig wird initialisiert.
  3. Der erwartete Pfad wird berechnet: Path(config_module.__file__).resolve().parent / "static".
  4. Es wird verifiziert, dass config.static_path exakt diesem berechneten Pfad entspricht.

4. Zusammenfassung und Implikationen

Diese Testsuite stellt sicher, dass die Konfigurationsmechanismen der Anwendung robust sind:

  • Flexibilität: Die Konfiguration kann über verschiedene Quellen (Standard, Umgebung, Payload) gesteuert werden.
  • Konsistenz: Es wird sichergestellt, dass die Unterscheidung zwischen aktiven und gespeicherten Laufzeitkonfigurationen korrekt gemanagt wird, insbesondere im Hinblick auf die Notwendigkeit eines Neustarts.
  • Pfadmanagement: Die relative Pfadlogik für statische Dateien ist korrekt implementiert, um Abhängigkeiten von der Repository-Struktur zu lösen.

    • Empfehlung: Die Tests sind gut strukturiert und decken die wichtigsten Schnittstellen der Konfigurationsverwaltung ab. Sie sind für die Validierung der Kernlogik der Konfigurations-Loader sehr nützlich.
