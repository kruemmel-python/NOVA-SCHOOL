Diese Klasse dient dazu, den Kontext eines Benutzers (insbesondere seine Rolle) zu modellieren. Attribute:

  • **self.role: str**: Speichert die Rolle des Benutzers (z.B. "teacher", "admin").

  • Parameter: role (String) – Die Rolle des Benutzers.
  • Rückgabe: None.
  • **@property def is_teacher(self) -> bool**:
  • Beschreibung: Eine Read-Only Property, die prüft, ob der Benutzer eine Rolle besitzt, die als Lehrer oder Administrator definiert ist.
  • Logik: Gibt True zurück, wenn self.role in der Menge {"teacher", "admin"} enthalten ist, andernfalls False.

Die Datei ist primär darauf ausgelegt, die Funktionalität von WikiManualService unter verschiedenen Szenarien zu validieren. Technische Dokumentation: test_wiki_manual.py

1. Überblick

Diese Datei (test_wiki_manual.py) enthält eine Sammlung von Unit-Tests, die die Funktionalität der Klasse WikiManualService (oder einer ähnlichen Service-Klasse) im Zusammenhang mit der Generierung und Darstellung von Benutzerhandbüchern (Manuals) in einem Wiki-System validieren. Die Tests konzentrieren sich auf folgende Hauptaspekte:

  1. Initialisierung und Dokumentenabruf: Überprüfung, ob das System Dokumente korrekt basierend auf einem gegebenen Scope findet.
  2. Rendering von Dokumenten: Überprüfung, ob Dokumente aus verschiedenen Perspektiven (z.B. Schüler- vs. Lehrer-Scope) korrekt gerendert und formatiert werden.
  3. Scope-Wechsel: Überprüfung, ob der Service korrekt auf unterschiedliche Benutzerrollen (Scopes) umschalten kann und die entsprechenden Inhalte anzeigt.

2. Struktur und Komponenten

2.1. Hauptklasse

  • Name: WikiManualTests
  • Typ: Erbt von unittest.TestCase.
  • Zweck: Enthält spezifische Testmethoden, um die Logik der WikiManualService zu verifizieren.

2.2. Abhängigkeiten (Impliziert)

Die Tests implizieren die Existenz folgender Komponenten:

  • **WikiManualService:** Die zentrale Klasse, die die Logik für das Laden, Verarbeiten und Rendern von Wiki-Handbüchern implementiert.
  • **_Session:** Eine Klasse oder Funktion, die den aktuellen Benutzerkontext (z.B. _Session("student") oder _Session("teacher")) bereitstellt.
  • **Path (von pathlib):** Wird verwendet, um Dateipfade im temporären Verzeichnis zu verwalten.
  • **tempfile:** Wird verwendet, um temporäre Verzeichnisse für die Testumgebung zu erstellen.

3. Detaillierte Analyse der Testfälle

Testfall 1: test_service_seeds_manuals_when_scope_folders_are_missing

Ziel: Verifizierung, dass das System Dokumente korrekt findet, selbst wenn die erwarteten Scope-Ordner nicht explizit vorhanden sind (was auf eine flexible Such- oder Initialisierungslogik hindeutet). Ablauf:

  1. Ein temporäres Verzeichnis (tmp) wird erstellt.
  2. Eine Instanz von WikiManualService wird mit diesem Verzeichnis initialisiert.
  3. Dokumente für den Scope "student-user" und "teacher-admin" werden über die Methode service.documents() abgerufen.

    4. Erwartete Ergebnisse:

  • Beide Scope-Abrufe (student_docs, teacher_docs) müssen erfolgreich sein (assertTrue).
  • Die ersten gefundenen Dokumente in beiden Scopes müssen den erwarteten Slug "README" haben.

    • Bedeutung: Testet die Robustheit des Dokumenten-Indexierungs- oder -Abrufmechanismus.

Testfall 2: test_student_gets_student_manual_and_markdown_is_rendered

Ziel: Überprüfung des vollständigen Rendering-Prozesses für einen spezifischen Benutzer-Scope (Schüler) und die korrekte Darstellung von Markdown-Inhalten, einschließlich Code-Blöcken und Tabellen. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt und zwei spezifische Ordner (Lehrer_Admin, Schüler_User) werden erstellt.
  2. Inhaltserstellung:
  • README.md im Lehrer-Scope enthält interne Notizen.
  • README.md im Schüler-Scope enthält eine strukturierte Anleitung mit einem Markdown-Code-Block (Python).
  • Eine Abhängigkeitsdatei (01_Start.md) wird erstellt.
  1. Die WikiManualService wird initialisiert.
  2. Die Methode service.render_page() wird aufgerufen, um das Handbuch für den Schüler-Scope (requested_scope="student-user") und die Seite "README" zu rendern.

    3. Erwartete Ergebnisse:

  • Der gerenderte HTML-String (html) muss Schlüsselkomponenten enthalten:
  • Den Titel "Schüler-Handbuch".
  • Die URL-Parameter /manual?scope=student-user&page=01_Start.
  • Die HTML-Struktur für die Tabelle (<table class="manual-table">).
  • Die HTML-Struktur für den Code-Block (<pre class="manual-code"><code data-language="python">).
  • Die interne Notiz ("Nur intern.") muss nicht im finalen HTML erscheinen.

    • Bedeutung: Testet die korrekte Interpretation von Markdown, die korrekte Verknüpfung von Dokumenten (z.B. [Kapitel 1](./01_Start.md)), und die korrekte Filterung von Inhalten basierend auf dem Scope.

Testfall 3: test_teacher_can_switch_to_teacher_scope

Ziel: Verifizierung der Fähigkeit des Systems, den Benutzerkontext erfolgreich von einem Scope (Schüler) auf einen anderen (Lehrer) umzuschalten und die entsprechenden Inhalte anzuzeigen. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt und die Struktur mit Lehrer_Admin und Schüler_User wird erstellt.
  2. Inhaltserstellung:
  • README.md im Lehrer-Scope enthält Lehrer-spezifische Informationen.
  • README.md im Schüler-Scope enthält Schüler-spezifische Informationen.
  1. Die WikiManualService wird initialisiert.
  2. Die Methode service.render_page() wird aufgerufen, um das Handbuch für den Lehrer-Scope (requested_scope="teacher-admin") zu rendern.

    3. Erwartete Ergebnisse:

  • Der gerenderte HTML-String muss Lehrer-spezifische Inhalte enthalten (z.B. "Lehrkraft", "Lehrkräfte und Administration").
  • Die URL-Parameter müssen den Lehrer-Scope widerspiegeln (/manual?scope=student-user – *Anmerkung: Hier scheint ein potenzieller Fehler in der Testlogik zu sein, da der Scope im Test auf teacher-admin gesetzt wird, aber die URL-Parameter auf student-user verweist. Dies sollte überprüft werden.*).
  • Eine interne Verlinkung (href="#start") muss vorhanden sein.

    • Bedeutung: Testet die korrekte Kontextwechsel-Logik und die korrekte Abbildung der Inhalte, die für die jeweilige Benutzerrolle relevant sind.

4. Zusammenfassung und Fazit

Die Testsuite ist umfassend und deckt die kritischen Pfade der Wiki-Funktionalität ab:

  1. Datenzugriff (Indexing/Retrieval)
  2. Content-Rendering (Markdown Parsing & Formatting)
  3. Role-Based Access Control (Scope Switching)

    4. Die Tests sind gut strukturiert, verwenden temporäre Dateien für Isolation und prüfen sowohl die Form (HTML-Struktur, Code-Blöcke) als auch den Inhalt (spezifische Texte, Verweise). Sie stellen sicher, dass die WikiManualService die unterschiedlichen Anforderungen verschiedener Benutzerrollen korrekt abbildet.
