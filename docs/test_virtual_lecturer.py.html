Technische Dokumentation: test_virtual_lecturer.py

Version: 1.0 Datum: 2026-04-06 Autor: Ralf Krümmel

1. Überblick

Diese Datei (test_virtual_lecturer.py) enthält Unit-Tests für die Komponenten des nova_school_server-Systems, insbesondere für die VirtualLecturerService. Die Tests zielen darauf ab, die korrekte Initialisierung und das Verhalten der _Session-Klasse sowie die Interaktion mit den Abhängigkeiten (CurriculumService, SchoolRepository, VirtualLecturerService) zu validieren.

2. Struktur und Inhalt

Die Datei definiert eine Klasse _Session, die wahrscheinlich zur Verwaltung von Benutzer- oder Session-Kontexten dient. Die Tests werden implizit durch die Importe und die Struktur der Datei (obwohl die eigentlichen Testmethoden fehlen, deutet die Struktur auf eine Test-Suite hin).

2.1. Klasse _Session

Die Klasse _Session dient als Datencontainer für die Informationen einer aktiven Sitzung oder eines Benutzers. Code-Auszug:

class _Session:
def __init__(self, username: str, *, is_teacher: bool = False, permissions: dict | None = None) -> None:
self.username = username
self.is_teacher = is_teacher
self.permissions = permissions or {}
self.group_ids: list[str] = []
self.user = {"display_name": username.title()}

Attribute und Methoden:

Attribut Typ Beschreibung
username str Der eindeutige Benutzername der Sitzung.
is_teacher bool Flag, der angibt, ob der Benutzer ein Lehrer ist (Standardwert: False).
permissions dict Eine Optionale Dictionary, die spezifische Berechtigungen für den Benutzer speichert. Standardmäßig leer ({}).
group_ids list[str] Eine Liste von IDs von Gruppen, zu denen der Benutzer gehört. Standardmäßig leer ([]).
user dict Ein Dictionary, das Benutzerinformationen enthält, primär {"display_name": "..."}.

**Konstruktor (__init__) Details:**

  • **username: str**: Muss angegeben werden.
  • **is_teacher: bool = False**: Optional. Standardmäßig auf False gesetzt.
  • **permissions: dict | None = None**: Optional. Wenn None übergeben wird, wird ein leeres Dictionary ({}) verwendet.
  • **self.user Initialisierung**: Der display_name wird durch das Aufrufen von username.title() generiert (z.B. "john_doe" wird zu "John Doe").

  • from nova_school_server.database import SchoolRepository: Importiert den Repository zur Abfrage/Speicherung von Schuldaten.
  • from nova_school_server.virtual_lecturer import VirtualLecturerService: Importiert den Service, der die Logik für virtuelle Dozenten verwaltet.

4. Test-Kontext

Obwohl die eigentlichen Testmethoden nicht im bereitgestellten Codeausschnitt enthalten sind, impliziert die Struktur, dass die Tests folgende Szenarien abdecken sollten:

  1. **Test der _Session Initialisierung:** Überprüfung, ob alle Attribute korrekt gesetzt werden (insbesondere display_name und Standardwerte für is_teacher, permissions, group_ids).
  2. **Test der VirtualLecturerService Interaktion:** Validierung, wie die VirtualLecturerService mit den Abhängigkeiten (CurriculumService, SchoolRepository) interagiert, um virtuelle Dozenten zu verwalten oder zu zuweisen.
  3. Test der Session-Verwaltung: Überprüfung, wie Sessions erstellt, aktualisiert oder zerstört werden.

Technische Dokumentation: test_virtual_lecturer.py

1. Überblick

Diese Datei enthält eine Reihe von Unit-Tests (VirtualLecturerTests), die die Funktionalität einer Klasse namens VirtualLecturerService sowie deren Interaktion mit einem SchoolRepository und einem CurriculumService validieren. Die Tests konzentrieren sich auf den Lebenszyklus von Lernaktivitäten, insbesondere das Starten von Kursen, das Vorbereiten von Aufgaben und das Speichern von Rückmeldungen.

2. Zielsetzung der Tests

Die Hauptziele dieser Tests sind:

  1. **Validierung des Startprozesses (test_start_creates_session_and_opening_message):** Sicherstellen, dass das Starten einer Aufgabe erfolgreich eine Session erstellt, die korrekte Metadaten (Kurs-ID, Modul-ID, Aufgaben-Titel) enthält und die erste Interaktion korrekt darstellt.
  2. **Validierung der Aufgabenvorbereitung (test_prepare_includes_task_and_runtime_context):** Überprüfen, dass die prepare-Methode die notwendigen Informationen (Prompt, Laufzeitkontext, Erfolgskriterien) korrekt in die generierte Antwort einbindet.
  3. Validierung der direkten Rückmeldung (Erfolgreiche und Erweiterte Antworten): Testen, wie die Methode auf verschiedene Arten von Code-Inputs (einfache Ausgaben vs. komplexe HTML/Semantik-Code) reagiert und die Qualität der generierten Rückmeldung bewertet.
  4. **Validierung des Benutzerzugriffs (test_student_cannot_start_explicitly_disabled_course):** Sicherstellen, dass Benutzer, die keinen Zugriff auf einen Kurs haben, den Startvorgang verhindern.
  5. **Validierung des Speichervorgangs (test_store_reply_persists_thread_entries):** Überprüfen, dass gespeicherte Antworten korrekt in die Threads des Benutzerverlaufs eingefügt werden.
  6. **Validierung der Antwortbereinigung (test_store_reply_strips_internal_answer_instructions):** Verifizieren, dass interne Anweisungen oder Metadaten aus der eingegebenen reply-String-Eingabe beim Speichern entfernt werden, um eine saubere Darstellung für den Benutzer zu gewährleisten.

3. Testaufbau (setUp und tearDown)

Der setUp-Methoden-Block initialisiert die Testumgebung:

  • Temporäre Umgebung: Erstellt ein temporäres Verzeichnis (self.tmp) für die Datenbankdatei.
  • Repository & Datenbank: Initialisiert einen SchoolRepository mit einer Datenbankdatei (school.db) in der temporären Umgebung.
  • Service-Instanziierung: Initialisiert den CurriculumService und den VirtualLecturerService.
  • Benutzer- und Berechtigungs-Setup:
  • Erstellt Benutzer (teacher und student) mit spezifischen Rollen und Berechtigungen (z.B. curriculum.use, curriculum.manage, mentor.use).
  • Erstellt Session-Objekte (self.teacher, self.student) mit den entsprechenden Berechtigungen.
  • Projektdefinition: Definiert ein Beispielprojekt (self.project).

    • Der tearDown-Methoden-Block sorgt für eine saubere Aufräumung:

  • Schließt die Datenbankverbindung (self.repository.close()).
  • Bereinigt das temporäre Verzeichnis (self.tmp.cleanup()).

4. Detaillierte Testfälle

Testname Beschreibung Erwartetes Ergebnis Schlüssel-Methoden
test_start_creates_session_and_opening_message Testet den erfolgreichen Start einer Aufgabe und die korrekte Struktur der zurückgegebenen Session-Daten. Session enthält korrekte course_id, module_id, task_title und eine erste Nachricht, die print() erwähnt. service.start()
test_prepare_includes_task_and_runtime_context Testet, ob die prepare-Methode den Kontext (Prompt, Laufzeitstatus, Erfolgskriterien) korrekt in die generierte Antwort einbindet. Die generierte Antwort enthält spezifische Phrasen wie "Aktive Aufgabe", "Letzter Returncode" und "Erfolgskriterien". service.prepare()
test_prepare_direct_feedback_accepts_valid_extended_python_intro_run Testet die Verarbeitung einer ausführlichen Python-Code-Antwort, um zu bestätigen, dass die Antwort inhaltlich vollständig und konstruktiv ist. Die Antwort wird als "erfüllt" bewertet, spezifische Code-Nutzungen (4 print()-Ausgaben) werden positiv hervorgehoben, und negative Hinweise fehlen. service.prepare()
test_prepare_direct_feedback_accepts_web_intro_with_semantics_lists_and_extensions Testet die Verarbeitung von HTML/Web-Code, um zu bestätigen, dass die Methode semantische Strukturen (Listen, CSS, JavaScript) korrekt interpretiert und positiv bewertet. Die Antwort bestätigt die Korrektheit des Code-Inputs und bewertet CSS/JavaScript als sinnvolle Erweiterungen, während spezifische, nicht gewünschte Begriffe vermieden werden. service.prepare()
test_student_cannot_start_explicitly_disabled_course Testet die Zugriffskontrolle: Ein Student, der keinen Zugriff auf einen Kurs hat, darf den Kurs nicht starten. Es wird eine PermissionError geworfen, wenn service.start() auf einen nicht zugänglichen Kurs aufgerufen wird. service.start()
test_store_reply_persists_thread_entries Testet, ob eine gespeicherte Antwort korrekt in den Thread des Benutzers mit dem richtigen Autor und der korrekten Anzahl an Einträgen gespeichert wird. Die gespeicherte Antwort wird korrekt in den Thread hinzugefügt, und der Autor wird als "Nova Dozent" identifiziert. service.store_reply()
test_store_reply_strips_internal_answer_instructions Testet die Bereinigung von Metadaten: Überprüft, dass interne Anweisungen (z.B. "Antwortauftrag:") aus der eingegebenen reply-String-Eingabe entfernt werden.
