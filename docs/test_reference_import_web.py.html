Technische Dokumentation: ReferenceWebImportTests

Diese Dokumentation beschreibt die Testklasse ReferenceWebImportTests, die die Funktionalität der Klasse ReferenceWebMirrorBuilder im Zusammenhang mit dem Import von Webseiten (Mirroring) testet.

1. Überblick

Die Testklasse verwendet das Python-Standardmodul unittest und dient dazu, sicherzustellen, dass die ReferenceWebMirrorBuilder korrekt:

  1. Lokale Ziele auflöst (z.B. für HTML-Dateien und statische Assets).
  2. HTML-Rewrites durchführt, um absolute Pfade in lokale, relative Pfade umzuwandeln.
  3. Landing Pages korrekt generiert, die auf die importierten Quellen verweisen.

    4. Die Tests werden in temporären Verzeichnissen (tempfile.TemporaryDirectory) ausgeführt, um sicherzustellen, dass die Tests isoliert und ohne Abhängigkeiten von externen Dateien oder Systemstrukturen laufen.

2. Testfälle

2.1. test_resolve_local_target_finds_html_and_assets

Ziel: Überprüfen, ob die Methode resolve_local_target korrekt lokale Pfade für URLs auflöst, sowohl für Hauptseiten als auch für statische Assets. Setup:

  • Ein temporäres Verzeichnis wird erstellt.
  • Ziel 1 (HTML): Ein Pfad zu https://developer.mozilla.org/en-US/docs/Web/JavaScript wird simuliert, wobei die lokale Datei site_root/JavaScript.html erstellt wird.
  • Ziel 2 (Asset): Ein Pfad zu einem statischen Asset /static/client/runtime.js wird simuliert, wobei die lokale Datei static_root/runtime.js erstellt wird.

    • Erwartete Ergebnisse:

  • builder.resolve_local_target("https://developer.mozilla.org/en-US/docs/Web/JavaScript") sollte auf site_root / "JavaScript.html" verweisen.
  • builder.resolve_local_target("/static/client/runtime.js") sollte auf static_root / "runtime.js" verweisen.

    • Zweck: Validierung der korrekten Mapping-Logik für verschiedene Arten von Ressourcen (HTML vs. JavaScript/CSS).

2.2. test_html_rewrite_localizes_absolute_pack_urls

Ziel: Überprüfen, ob die Methode _rewrite_mirror_html erfolgreich absolute Pfade in relative Pfade innerhalb der generierten HTML-Struktur umwandelt. Setup:

  • Ein temporäres Verzeichnis wird erstellt.
  • Quellstruktur:
  • Eine Hauptseite (current/index.html) enthält Links zu externen Ressourcen (z.B. /static.files/rustdoc.css, /static.files/logo.svg).
  • Eine Quellseite (std/index.html) existiert.
  • Zielstruktur:
  • Die statischen Dateien (rustdoc.css, logo.svg) werden in static_root/ platziert.
  • Die Quellseite (std/index.html) wird in std_root/ platziert.

    • Ablauf:

  1. builder._rewrite_mirror_html() wird aufgerufen.
  2. Der Inhalt der umgeschriebenen current/index.html wird gelesen.

    3. Erwartete Ergebnisse:

  • Der umgeschriebene HTML-Code muss relative Pfade enthalten, die auf die lokalen Strukturen verweisen (z.B. href="../std/index.html").
  • Die Pfade zu den statischen Dateien müssen ebenfalls relativ sein (z.B. src="../static.files/logo.svg").

    • Zweck: Validierung der Logik zur Umwandlung von absoluten URL-Referenzen in lokale, relativ verknüpfte Pfade.

2.3. test_landing_page_links_to_mirrored_sources

Ziel: Überprüfen, ob die Methode _write_landing_page korrekt die notwendigen Links zu den importierten Quellen in der Landing Page generiert. Setup:

  • Ein temporäres Verzeichnis wird erstellt.
  • Quellstruktur:
  • Eine Landing Page (site_root/index.html) existiert.
  • Eine externe API-Dokumentation (api_root) wird simuliert.

    • Ablauf:

  1. builder._write_landing_page() wird aufgerufen.
  2. Der Inhalt der generierten Landing Page wird gelesen.

    3. Erwartete Ergebnisse:

  • Die Anzahl der generierten Einträge (len(entries)) muss korrekt sein (erwartet: 2).
  • Die Landing Page muss den Namen der Hauptseite enthalten ("Dev.java Learn").
  • Die Landing Page muss einen Link zur API-Dokumentation enthalten ("docs.oracle.com/en/java/javase/21/docs/api/index.html").

    • Zweck: Validierung der Funktion, die die Hauptseite mit wichtigen Referenzen zu den importierten Dokumentationsquellen verknüpft.

3. Voraussetzungen

  • Klasse: ReferenceWebMirrorBuilder (muss definiert sein).
  • Konstante: MIRROR_PACKS (muss definiert sein und die notwendigen Pakete für die Tests enthalten, z.B. "javascript", "rust", "java").
  • Modelle: Path (von pathlib), tempfile, unittest.

4. Zusammenfassung

Diese Testsuite stellt sicher, dass die Kernfunktionen des ReferenceWebMirrorBuilder – Zielauflösung, HTML-Rewriting und Landing Page Generierung – gemäß den Spezifikationen funktionieren, indem sie mit kontrollierten, temporären Dateisystemstrukturen getestet werden.
