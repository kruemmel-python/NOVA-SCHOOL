Technische Dokumentation: test_realtime.py

1. Überblick

Diese Datei (test_realtime.py) scheint eine Test- oder Hilfsklasse zu sein, die Komponenten aus dem nova_school_server-Framework simuliert oder testet, insbesondere im Kontext von Echtzeit-Funktionalitäten (WebSocket-Verbindungen, Code-Ausführung, Benutzer-Sitzungen). Die Datei definiert mehrere interne Klassen (_FakeToolSandbox, _Session, _TeacherSession, _RecordingConnection), die wahrscheinlich zur Simulation von Authentifizierungsmechanismen, Benutzerrollen und der Protokollierung von Ereignissen in Echtzeit-Verbindungen dienen.

2. Struktur und Komponenten

Die Datei enthält die folgenden definierten Klassen:

Diese Klasse dient wahrscheinlich als Mock oder Fake für einen Tool-Sandbox-Mechanismus.

  • Rückgabe: Ein Dictionary, das den Status der Autorisierung zurückgibt: {"authorized": True}.

  • Attribute:

  • Beschreibung: Fügt ein gegebenes JSON-Payload zur Ereignisliste hinzu.

Technische Dokumentation: _FakeSocket Klasse

1. Überblick

Die Klasse _FakeSocket ist eine Simulationsklasse, die die Funktionalität eines Netzwerk-Sockets (Socket) für Testzwecke bereitstellt. Sie dient dazu, die Kommunikation zwischen verschiedenen Komponenten zu simulieren, insbesondere das Lesen (recv) und Schreiben (sendall) von Daten in Chunks.

2. Klasse: _FakeSocket

Diese Klasse implementiert eine vereinfachte Version eines Sockets, die Daten in diskreten Teilen (Chunks) verarbeitet und die Möglichkeit bietet, Timeout-Verhalten zu simulieren.

2.1. Konstruktor (__init__)

Der Konstruktor initialisiert den Zustand des Fake-Sockets. Parameter:

  • chunks: list[bytes] = None: Eine Liste von Datenfragmenten (bytes), die als Puffer für zukünftige Empfangsvorgänge dienen. Wenn None angegeben wird, wird eine leere Liste erstellt.
  • raise_timeout: bool = False: Ein boolescher Flag, das bestimmt, ob der recv-Aufruf bei Timeout-Bedingungen eine TimeoutError auslösen soll.

    • Attribute:

  • self.timeout: str = "unset": Speichert den aktuellen Timeout-Status. Standardwert ist "unset".
  • self.chunks: list[bytes]: Eine Liste von Datenfragmenten, die noch empfangen werden müssen.
  • self.raise_timeout: bool: Flag, das die Timeout-Verhalten steuert.
  • self.sent_frames: list[bytes]: Eine Liste von Datenblöcken (bytes), die über die Methode sendall gesendet wurden.

    • Spezielle Methoden:

  • settimeout(value: str) -> None: Setzt den Timeout-Status des Sockets.
  • recv(size: int) -> bytes: Simuliert den Empfang von Daten.
  • sendall(data: bytes) -> None: Simuliert das Senden von Daten.
  • shutdown(_how: None) -> None: Simuliert den Shutdown-Prozess (aktuell leer).
  • close() -> None: Simuliert das Schließen des Sockets (aktuell leer).

2.2. Methoden

2.2.1. settimeout(self, value: str) -> None

Setzt den Timeout-Status des Sockets auf den angegebenen Wert.

  • **value**: Der neue Timeout-Wert (String).

2.2.2. recv(self, size: int) -> bytes

Simuliert den Empfang von size Bytes von der verbundenen Quelle. Funktionsweise:

  1. Timeout-Prüfung: Prüft, ob self.raise_timeout auf True gesetzt ist. Wenn ja, wird eine TimeoutError("timed out") geworfen.
  2. Chunk-Verarbeitung: Wenn self.chunks nicht leer ist:
  • Der erste Chunk (self.chunks[0]) wird entfernt und als chunk extrahiert.
  • Die ersten size Bytes dieses Chunks werden als Ergebnis zurückgegeben (current).
  • Der Rest des Chunks (remainder) wird geprüft. Wenn dieser Rest existiert, wird er wieder an den Anfang der self.chunks-Liste eingefügt.
  1. Ende der Daten: Wenn self.chunks leer ist, wird eine leere Byte-Sequenz (b"") zurückgegeben.

    2. Parameter:

  • size: int: Die maximale Anzahl von Bytes, die vom Socket empfangen werden sollen.

    • Rückgabetyp:

  • bytes: Die empfangenen Bytes (maximal size Bytes).

2.2.3. sendall(self, data: bytes) -> None

Simuliert das Senden von Daten an den verbundenen Partner. Funktionsweise:

  • Die gesamte übermittelte data-Sequenz wird zur Liste self.sent_frames hinzugefügt.

    • Parameter:

  • data: bytes: Die Daten, die gesendet werden sollen.

2.2.4. shutdown(self, _how: None) -> None

Simuliert den Shutdown-Prozess des Sockets.

  • Parameter:
  • _how: Optionaler Parameter, der den Art des Shutdowns angibt (aktuell nicht implementiert).

2.2.5. close(self) -> None

Simuliert das Schließen des Sockets.

  • Funktionsweise: Keine spezifische Aktion implementiert (aktuell leer).

Überblick

Diese Dokumentation beschreibt die Funktionen, die in der Datei test_realtime.py definiert sind. Die Funktionen scheinen Teil eines Systems zu sein, das mit der Verarbeitung und Maskierung von Daten (wahrscheinlich in einem Netzwerk- oder Protokollkontext) arbeitet, sowie mit der Chunking-Operation von Daten.

Funktionen

1. _build_masked_frame

Diese interne Funktion ist dafür verantwortlich, einen verschlüsselten oder maskierten Datenrahmen (Frame) zu konstruieren. Sie berechnet den Header und den optional maskierten Payload basierend auf verschiedenen Parametern. Signature:

def _build_masked_frame(
*,
opcode: int = 0x1,
payload: bytes = b"",
fin: bool = True,
masked: bool = True,
rsv: int = 0,
mask_key: bytes = b"\x01\x02\x03\x04",
declared_length: int | None = None,
) -> bytes:
"""
Baut einen Datenrahmen (Frame) zusammen, der optional maskiert wird.
Args:
opcode: Der Operationscode (standardmäßig 0x1).
payload: Die eigentlichen Daten, die verschraubt werden sollen.
fin: Flag, ob der Frame als Ende (Finish) gekennzeichnet ist (standardmäßig True).
masked: Flag, ob der Payload maskiert werden soll (standardmäßig True).
rsv: Reserved-Bit-Wert für den Header.
mask_key: Eine Schlüsselsequenz (4 Bytes), die für die Maskierung verwendet wird.
declared_length: Optionaler deklarierter Länge des Payloads. Wenn None, wird die tatsächliche Länge verwendet.
Returns:
Die vollständige Byte-Sequenz des Frames (Header + optional Maskierung + Payload).
"""
# 1. Header-Berechnung
first = (0x80 if fin else 0x00) | (rsv & 0x70) | (opcode & 0x0F)
actual_length = len(payload)
wire_length = actual_length if declared_length is None else max(0, int(declared_length))
second_base = 0x80 if masked else 0x00
# Header-Typisierung basierend auf der Wire Length
if wire_length < 126:
# Header: [first, second_base | wire_length]
header = bytes([first, second_base | wire_length])
elif wire_length < 65536:
# Header: [first, second_base | 126], gefolgt von der 2-Byte-Länge (H)
header = bytes([first, second_base | 126]) + struct.pack("!H", wire_length)
else:
# Header: [first, second_base | 127], gefolgt von der 8-Byte-Länge (Q)
header = bytes([first, second_base | 127]) + struct.pack("!Q", wire_length)
# 2. Payload-Verarbeitung
if not masked:
# Wenn nicht maskiert, Header + Payload
return header + payload
# Wenn maskiert: Byte-für-Byte-XOR-Maskierung
masked_payload = bytes(byte ^ mask_key[index % 4] for index, byte in enumerate(payload))
# Endergebnis: Header + Maskenschlüssel + Maskierter Payload
return header + mask_key + masked_payload
### 2. `_chunk_bytes`
Diese Funktion ist dafür verantwortlich, eine Byte-Sequenz in eine Liste von kleineren, zufällig definierten Byte-Chunks zu zerlegen.
**Signature:**

Args: data: Die zu zerlegende Byte-Sequenz. seed: Der Zufallsgenerator-Seed, um die Chunk-Größen zu bestimmen. Returns: Eine Liste von Byte-Objekten (Chunks). """ randomizer = random.Random(seed) index = 0 chunks: list[bytes] = [] while index < len(data):

Bestimmt die Größe des nächsten Chunks zwischen 1 und 7 Bytes

step = randomizer.randint(1, 7) chunks.append(data[index : index + step]) index += step return chunks

## Zusammenfassung der Logik
1. **Frame-Konstruktion (`_build_masked_frame`):**
* Berechnet einen Header basierend auf der gewünschten `fin`-Flag, `rsv`-Wert und `opcode`.
* Bestimmt die Länge des Frames (`wire_length`) und formatiert den Header entsprechend der Länge (klein, mittel oder groß).
* Wenn `masked=True`, wird der Payload durch eine XOR-Operation mit einem zyklischen `mask_key` verschlüsselt.
* Das Ergebnis ist `Header + Maskenschlüssel + Maskierter Payload`.
2. **Daten-Chunking (`_chunk_bytes`):**
* Initialisiert einen Zufallsgenerator mit dem bereitgestellten `seed`.
* Iteriert durch die Eingabedaten und teilt sie in Segmente (Chunks) auf, wobei die Größe jedes Segments zufällig zwischen 1 und 7 Bytes liegt.
## Abhängigkeiten
* **`struct`:** Wird für die Serialisierung/Deserialisierung von Integer-Typen (`!H` für 2-Byte-Unsigned Short, `!Q` für 8-Byte-Unsigned Long) verwendet.
* **`random`:** Wird für die Generierung zufälliger Chunk-Größen verwendet.

## Technische Dokumentation: Funktion `_random_frame(seed: int) -> bytes`
**Datei:** `test_realtime.py`
**Funktion:** `_random_frame(seed: int) -> bytes`
---
### 1. Beschreibung
Die Funktion `_random_frame` ist eine interne Hilfsfunktion, die basierend auf einem gegebenen `seed`-Wert ein zufälliges Datenpaket (Frame) generiert. Diese Funktion simuliert die Erstellung von Datenpaketen, die wahrscheinlich für eine Echtzeit- oder Netzwerkkommunikationsanwendung relevant sind, indem sie verschiedene Frame-Typen (basierend auf einem zufälligen Modus) generiert.
Die Funktion verwendet einen `random.Random`-Instanz, initialisiert mit dem bereitgestellten `seed`, um reproduzierbare, aber zufällige Ergebnisse zu gewährleisten.
### 2. Signatur

def _random_frame(seed: int) -> bytes:

... Implementierung ...

return bytes

* **Name:** `_random_frame`
* **Parameter:**
* `seed: int`: Ein Integer-Wert, der als Initialisierungs-Seed für den Zufallsgenerator verwendet wird.
* **Rückgabe:**
* `bytes`: Ein Byte-String, der den generierten Frame darstellt.
### 3. Logik und Funktionsweise
Die Funktion führt eine sequenzielle Logik aus, um den Frame zu bestimmen:
1. **Zufallsgenerator initialisieren:** Ein `random.Random`-Objekt wird mit dem übergebenen `seed` initialisiert.
2. **Modusbestimmung:** Es wird ein zufälliger Modus (`mode`) zwischen 0 und 2 ausgewählt (`randomizer.randrange(4)`).
3. **Modus-spezifische Generierung:** Basierend auf dem `mode` wird eine der folgenden drei Pfade ausgeführt:
#### Modus 0: Maskierter Frame mit variabler Payload
* **Payload-Größe:** Die Größe der Payload wird zufällig zwischen 0 und 47 Bytes gewählt (`randomizer.randrange(0, 48)`).
* **Payload-Inhalt:** Die Payload besteht aus zufälligen Bytes (0 bis 255).
* **Frame-Parameter:**
* `opcode`: Zufällige Wahl zwischen 0 und 15.
* `fin` (Finish Flag): Zufällig binär (0 oder 1).
* `masked`: Zufällig binär (0 oder 1).
* `rsv` (Reserved): Ein Wert, der durch `randomizer.randrange(0, 8)` verschoben wird (`<< 4`).
* `mask_key`: Eine Schlüssel-Byte-Sequenz von 4 Bytes, generiert aus zufälligen Bytes (0 bis 255).
* **Ergebnis:** Der generierte Frame wird über die interne Funktion `_build_masked_frame` konstruiert.
#### Modus 1: Maskierter Frame mit fester Opcode und variabler Länge
* **Wire Length (Übertragungsgröße):** Die Länge des Frames wird aus einer Liste von Werten gewählt:
* 126, 127, 200, 1024, oder `MAX_CLIENT_WS_FRAME_BYTES` plus ein zufälliges Byte (0 bis 255).
* **Payload-Größe:** Die Payload besteht aus zufälligen Bytes, wobei die Größe zwischen 0 und 31 Bytes gewählt wird (`randomizer.randrange(0, 32)`).
* **Frame-Parameter:**
* `opcode`: Zufällige Wahl aus $\{0x1, 0x8, 0x9, 0xA\}$.
* `masked`: Wird explizit auf `True` gesetzt.
* `declared_length`: Der zuvor gewählte `wire_length`.
* `mask_key`: Eine Schlüssel-Byte-Sequenz von 4 Bytes, generiert aus zufälligen Bytes (0 bis 255).
* **Ergebnis:** Der generierte Frame wird über die interne Funktion `_build_masked_frame` konstruiert.
#### Modus 2: Einfacher, unmaskierter Frame
* **Payload-Größe:** Die Payload besteht aus zufälligen Bytes, wobei die Größe zwischen 0 und 63 Bytes gewählt wird (`randomizer.randrange(0, 64)`).
* **Payload-Inhalt:** Zufällige Bytes (0 bis 255).
* **Frame-Parameter:** Keine spezifischen Opcode- oder Maskierungs-Parameter werden gesetzt.
* **Ergebnis:** Der generierte Frame besteht ausschließlich aus der generierten Payload.
#### Standardfall (Fallback)
* **Payload-Inhalt:** Die Payload besteht aus dem String `"ok-"` gefolgt vom String-Repräsentation des `seed` (kodiert in UTF-8).
* **Frame-Parameter:**
* `opcode`: Fest auf `0x1` gesetzt.
* `masked`: Wird auf `True` gesetzt.
* **Ergebnis:** Der generierte Frame wird über die interne Funktion `_build_masked_frame` konstruiert.
### 4. Abhängigkeiten und Konstanten
* **Abhängigkeiten:**
* `random` (Standardbibliothek)
* `_build_masked_frame` (Eine externe, nicht gezeigte Hilfsfunktion, die die eigentliche Frame-Konstruktion übernimmt.)
* **Konstanten (impliziert):**
* `MAX_CLIENT_WS_FRAME_BYTES`: Eine definierte Konstante, die die maximale Größe für Modus 1 definiert.
### 5. Nutzungsszenario
Diese Funktion wird typischerweise verwendet, um:
1. Testdaten für Netzwerkprotokolle zu generieren.
2. Simulierter Datenverkehr in einer Echtzeit-Anwendung zu erstellen.
3. Reproduzierbare Testfälle zu erstellen, indem der gleichen `seed` verwendet wird.
### 6. Sicherheit und Robustheit
* **Zufallsgenerierung:** Die Verwendung von `random.Random(seed)` stellt sicher, dass die Ergebnisse reproduzierbar sind, was für Debugging und Unit-Tests wichtig ist.
* **Datenintegrität:** Die Funktion generiert Byte-Strings, was die Eignung für Netzwerkprotokolle impliziert.
* **Fehlermeldungen:** Es wird angenommen, dass die interne Funktion `_build_masked_frame` alle notwendigen Validierungen und Formatierungsprüfungen für die Frame-Struktur durchführt.
---
**Zusammenfassung:** `_random_frame` ist ein generischer Frame-Generator, der durch einen Seed initialisiert wird und zufällig zwischen drei definierten Protokoll- oder Frame-Strukturen wechselt.

# Technische Dokumentation: `test_realtime.py`

## 2. Struktur und Komponenten
Die Datei implementiert eine Testklasse `RealtimeTests`, die alle spezifischen Testfälle kapselt.
### 2.1. Klasse: `RealtimeTests(unittest.TestCase)`

## 3. Detaillierte Analyse der Testfälle
Die Tests lassen sich in mehrere Hauptkategorien einteilen: WebSocket-Protokoll-Validierung, Fehlerbehandlung und End-to-End-Workflow-Simulation.
### 3.1. WebSocket-Protokoll-Validierungstests
Diese Tests prüfen, ob die `WebSocketConnection` korrekt mit verschiedenen WebSocket-Operationen und Protokollregeln interagiert.

| :--- | :--- | :--- |

### 3.2. End-to-End-Workflow-Test
Dieser Test simuliert den gesamten Lebenszyklus eines Live-Code-Runners, um zu bestätigen, dass die Kommunikation zwischen Client, Server und dem Code-Interpreter korrekt funktioniert.
| Test-Methode | Beschreibung | Erwartetes Verhalten |
| :--- | :--- | :--- |
| `test_live_run_manager_streams_prompt_and_accepts_input` | Simuliert die Initialisierung eines Live-Run-Systems, das einen Python-Code ausführt, Benutzerinput entgegennimmt und die Ausgabe zurückgibt. | 1. Der Prozess startet erfolgreich (`run.started`). 2. Der Benutzerinput ("Nova\n") wird gesendet. 3. Die erwartete Ausgabe ("Hallo Nova!") wird im `run.output`-Stream erfasst. 4. Der Prozess beendet sich korrekt (`run.exit` mit Returncode 0). 5. Die Metadaten (z.B. `cell_id`) stimmen überein. |
**Wichtige Details des E2E-Tests:**
* **Setup:** Es wird ein temporäres Verzeichnis erstellt, Konfigurationen (`ServerConfig`), ein Repository (`SchoolRepository`), ein Workspace (`WorkspaceManager`) und ein `CodeRunner` initialisiert.
* **Client-Setup:** Ein `RealtimeClient` wird mit einer `_RecordingConnection` und einem `_TeacherSession` erstellt.
* **Code-Ausführung:** Ein einfacher Python-Code (`name = input('Name: ')\nprint(f'Hallo {name}!')\n`) wird ausgeführt.
* **Interaktion:** Der Test simuliert das Eingeben des Namens "Nova" und wartet auf die Ausgabe.
* **Validierung:** Es werden spezifische String-Ergebnisse (`"Name:"`, `"Hallo Nova!"`) und die korrekte Exit-Information (`returncode=0`) sowie die korrekte Zuweisung der Zelle (`cell_id="cell-1"`) überprüft.
## 4. Technische Zusammenfassung und Implikationen
1. **Robustheit des WebSocket-Parsers:** Die Tests deuten auf eine sehr robuste Implementierung des WebSocket-Parsers hin, der explizit auf Protokollfehler (ungültige UTF-8, falsche Frame-Größen, reservierte Bits) reagiert, indem er spezifische `ConnectionError`-Ausnahmen wirft.
2. **Fehlerbehandlung:** Die Unterscheidung zwischen einem Netzwerk-/Socket-Fehler (`ConnectionError`) und einem allgemeinen Python-Fehler ist wichtig, insbesondere im Fuzzing-Test.
3. **Workflow-Integration:** Der E2E-Test bestätigt, dass die gesamte Architektur – von der Projektinitialisierung über die Verbindungseröffnung bis hin zur Ausführung eines Code-Blocks und der Erfassung der Ergebnisse – kohärent funktioniert.
4. **Abhängigkeiten:** Der Test ist stark abhängig von externen oder internen Klassen und Funktionen (z.B. `_FakeSocket`, `_build_masked_frame`, `_random_frame`, `_chunk_bytes`, `ServerConfig`, `CodeRunner`, `LiveRunManager`), deren genaue Implementierung ist für eine vollständige Analyse erforderlich.
## 5. Empfehlungen
* **Dokumentation der Helper-Funktionen:** Die Funktionen, die zur Frame-Generierung (`_build_masked_frame`, `_random_frame`) und Datenmanipulation (`_chunk_bytes`) verwendet werden, sollten in der Hauptdokumentation oder in ihren jeweiligen Modulen detailliert beschrieben werden, da sie die Grundlage für die Protokoll-Validierung bilden.
* **Klärung der Fehlercodes:** Die spezifischen Fehlermeldungen in den `assertRaisesRegex`-Tests (z.B. `"Extensionsbits"`, `"ungueltiges UTF-8"`) sollten in der Dokumentation erklärt werden, um Entwicklern zu helfen, die Ursachen von Fehlern im Protokoll zu verstehen.
