Technische Dokumentation: WorkspaceTests

Übersicht

Diese Dokumentation beschreibt die Testfälle der Klasse WorkspaceTests aus der Datei test_workspace.py. Diese Tests dienen dazu, die Funktionalität der WorkspaceManager-Klasse zu validieren, insbesondere im Hinblick auf Projektmaterialisierung, Dateisystemoperationen (Erstellen, Löschen, Umbenennen) und die Normalisierung von Daten (z.B. in Notebooks).

Zielsetzung

Die Hauptziele dieser Tests sind:

  1. Projektmaterialisierung: Sicherstellen, dass ein Projekt korrekt in das lokale Dateisystem erstellt wird.
  2. Dateisystem-Sicherheit: Validieren, dass bestimmte Operationen (wie das Ändern von Pfaden) fehlerhaft blockiert werden.
  3. Datenintegrität: Überprüfen, dass Daten (wie Code in Notebooks) korrekt gespeichert und geladen werden.
  4. Metadaten-Management: Testen, wie die Struktur des Projekts (Dateien, Ordner) durch verschiedene Manager-Methoden manipuliert wird.

Testfälle im Detail

1. test_materialize_project_and_block_path_escape

Beschreibung: Dieser Test stellt sicher, dass das Materialisieren eines Projekts erfolgreich ist und dass das System versucht, einen potenziell gefährlichen Pfad (Path-Escape) zu verarbeiten, diesen jedoch fehlschlagen lässt. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Eine ServerConfig wird aus diesem Verzeichnis geladen.
  3. Ein WorkspaceManager wird mit dieser Konfiguration initialisiert.
  4. Ein Projekt-Schema wird definiert.
  5. manager.materialize_project(project) wird aufgerufen.
  6. Erwartung 1: Es wird verifiziert, dass die Hauptdatei (main.py) im erstellten Stammverzeichnis existiert.
  7. Erwartung 2: Ein Aufruf von manager.resolve_project_path(project, "..\\..\\evil.txt") wird versucht.
  8. Erwartung 3: Es wird erwartet, dass eine ValueError ausgelöst wird, was die Sicherheit gegen unzulässige Pfadmanipulation bestätigt.

    9. Schlüsselkonzepte: tempfile.TemporaryDirectory, Path, ValueError.

2. test_load_notebook_normalizes_legacy_starter_cells

Beschreibung: Dieser Test prüft, ob die load_notebook-Funktion Legacy-Starterzellen korrekt parst und die Daten sowohl beim Laden als auch beim Speichern wiederherstellt. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt und die Konfiguration sowie den WorkspaceManager initialisiert.
  2. Ein Projekt wird definiert.
  3. manager.materialize_project(project) wird aufgerufen, um das Projekt zu erstellen.
  4. Ein Testinhalt (JSON-String mit Python-Code) wird in eine Datei (.nova-school/notebook.json) geschrieben.
  5. manager.load_notebook(project) wird aufgerufen, um die Zellen zu laden.
  6. Erwartung 1: Der zurückgegebene Code (cells[0]["code"]) muss exakt mit dem ursprünglichen Code übereinstimmen.
  7. Erwartung 2: Der Inhalt der gespeicherten JSON-Datei wird geladen und der Code wird erneut verglichen, um die Konsistenz der Speicherung zu bestätigen.

    8. Schlüsselkonzepte: JSON-Serialisierung/Deserialisierung, String-Vergleich, Datenkonsistenz.

3. test_delete_file_removes_secondary_file_and_blocks_main_file

Beschreibung: Dieser Test validiert die Logik des Löschvorgangs: Wenn eine Datei gelöscht wird, müssen abhängige oder sekundäre Dateien entfernt werden, und das Löschen der Hauptdatei muss blockiert werden. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Ein Projekt wird definiert, das eine main_file spezifiziert.
  3. Das Projekt wird materialisiert.
  4. Eine zusätzliche Datei (notes/draft.txt) wird erstellt.
  5. manager.delete_file(project, "notes/draft.txt") wird aufgerufen.
  6. Erwartung 1: Die Methode muss den korrekten Pfad zurückgeben.
  7. Erwartung 2: Die Datei (draft.txt) und ihr übergeordneter Ordner (notes/) müssen nicht mehr existieren.
  8. Erwartung 3: Ein Versuch, manager.delete_file(project, "main.py") durchzuführen, muss zu einer ValueError führen.

    9. Schlüsselkonzepte: Abhängigkeitsmanagement beim Löschen, Fehlerbehandlung (ValueError).

4. test_directory_operations_support_create_delete_and_main_path_rename

Beschreibung: Dieser umfassende Test validiert die Fähigkeit des Managers, grundlegende Dateisystemoperationen (Erstellen von Ordnern, Löschen von Ordnern, Umbenennen von Einträgen) zu unterstützen und die daraus resultierenden Metadaten korrekt zu aktualisieren. Ablauf:

  1. Ein temporäres Verzeichnis wird erstellt.
  2. Ein Projekt wird definiert, das eine spezifische main_file angibt.
  3. Das Projekt wird materialisiert.
  4. Erstellen: Ein Ordner (src/components) wird erstellt.
  • Erwartung 1: Die Methode gibt den korrekten Pfad zurück, und der Ordner existiert.
  1. Löschen: Ein Ordner (notes) wird gelöscht.
  • Erwartung 1: Die Methode gibt den Typ "directory" zurück.
  • Erwartung 2: Der Ordner und alle darin enthaltenen Dateien müssen nicht mehr existieren.
  1. Fehlerprüfung: Ein Versuch, einen nicht existierenden Ordner (services) zu löschen, muss zu einer ValueError führen.
  2. Umbenennen: Ein Eintrag (services) wird umbenannt in (core-services).
  • Erwartung 1: Die Methode gibt den alten und neuen Pfad zurück.
  • Erwartung 2: Die Metadaten des Projekts (main_file) müssen korrekt auf den neuen Pfad (core-services/coordinator.py) aktualisiert werden.
  • Erwartung 3: Der alte Eintrag (services) muss nicht mehr existieren, und der neue Eintrag (core-services/coordinator.py) muss existieren.

    • Schlüsselkonzepte: Ordner-Manipulation (mkdir, delete_entry), Pfad-Umbenennung, Metadaten-Synchronisation.

Zusammenfassung und Fazit

Die Testsuite WorkspaceTests deckt die kritischen Bereiche der WorkspaceManager-Funktionalität ab: Initialisierung, Pfdsicherheit, Datenpersistenz (Notebooks) und komplexe Dateisystem-Manipulationen. Die Tests sind gut strukturiert, verwenden temporäre Verzeichnisse für Isolation und prüfen sowohl die erfolgreiche Ausführung als auch die korrekte Fehlerbehandlung (ValueError), was auf eine robuste Implementierung der Workspace-Logik hindeutet.
