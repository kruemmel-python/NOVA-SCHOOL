Technische Dokumentation: auth.py

Übersicht

Diese Datei (auth.py) definiert die Struktur für den Session Context und enthält die notwendigen Komponenten für die Verwaltung von Benutzer- und Sitzungsdaten, einschließlich der Extraktion von Benutzerinformationen und der Überprüfung von Berechtigungen.

Klassen und Datentypen

SessionContext (dataclass)

Die Klasse SessionContext dient als zentrales Objekt, um alle relevanten Informationen zu einem aktiven Benutzer-Sitzung (Session) zu kapseln. Sie verwendet dataclasses mit slots=True für eine effiziente Speicherung. Attribute:

Attribut Typ Beschreibung
token str Der eindeutige Sitzungstoken.
token_id str Eine eindeutige ID für das Token.
principal Any Der Haupt-Principal (kann eine beliebige Art von Objekt sein).
user dict[str, Any] Eine Dictionary, das die grundlegenden Benutzerdaten enthält (z.B. username, role, display_name).
groups list[dict[str, Any]] Eine Liste von Dictionaries, die die Gruppenmitgliedschaften darstellen (z.B. group_id, display_name).
permissions dict[str, bool] Eine Dictionary, das die spezifischen Berechtigungen des Benutzers speichert (Schlüssel sind wahrscheinlich Berechtigennamen, Werte sind True/False).

Properties (Zugriffsschnittstellen): Die Klasse definiert mehrere Read-Only-Properties, um den Zugriff auf spezifische Benutzerinformationen zu vereinfachen:

  • **username (str):** Gibt den Benutzernamen aus dem user-Dictionary zurück.
  • **role (str):** Gibt die Rolle des Benutzers aus dem user-Dictionary zurück.
  • **is_teacher (bool):** Gibt True zurück, wenn die Rolle "teacher" oder "admin" ist.
  • **is_admin (bool):** Gibt True zurück, wenn die Rolle exakt "admin" ist.
  • **group_ids (list[str]):** Gibt eine Liste der group_ids zurück, indem nur die group_id aus jedem Gruppen-Dictionary extrahiert wird.

    • Methoden:

  • **to_dict(self) -> dict[str, Any]:**
  • Konvertiert den gesamten SessionContext in ein einfaches Python-Dictionary.
  • Struktur des zurückgegebenen Dictionaries:
  • "username": Der Benutzername.
  • "display_name": Der angezeigte Name.
  • "role": Die Rolle.
  • "groups": Eine Liste von Dictionaries, die die Gruppeninformationen (group_id und display_name) enthalten.
  • "permissions": Das gesamte Berechtigungs-Dictionary.
  • "token_id": Die Token-ID.

Abhängigkeiten

Die Klasse SessionContext hängt von folgenden externen Komponenten ab:

  1. **SchoolRepository (aus .database):** Wahrscheinlich wird diese Klasse verwendet, um Benutzer- oder Sitzungsdaten aus einer Datenbank abzurufen, um einen SessionContext zu initialisieren.
  2. **resolve_permissions (aus .permissions):** Diese Funktion wird implizit oder explizit verwendet, um Berechtigungen zu verarbeiten oder zu validieren.

Zusammenfassung der Funktionalität

Die Datei dient als Datenmodell für die Sitzungsverwaltung. Sie stellt sicher, dass alle notwendigen Informationen (Identität, Rolle, Gruppenmitgliedschaften, Berechtigungen) in einem konsistenten Objekt (SessionContext) zusammengefasst sind, was die spätere Verarbeitung von Anfragen (z.B. Zugriffskontrolle) vereinfacht.

Technische Dokumentation: AuthService Klasse

1. Übersicht

Die Klasse AuthService ist eine zentrale Komponente für die Verwaltung von Benutzerauthentifizierung, -verwaltung und Sitzungsmanagement innerhalb einer Anwendung. Sie koordiniert die Interaktion mit dem Datenrepository (zur Speicherung von Benutzerdaten) und dem Sicherheitsplan (zur Token-Generierung und -Validierung).

2. Klasse: AuthService

2.1. Konstruktor (__init__)

Der Konstruktor initialisiert die notwendigen Abhängigkeiten und konfiguriert die Tenant-Spezifischen Einstellungen. Parameter:

  • repository: SchoolRepository: Eine Instanz des Repositorys, die für die Speicherung und Abfrage von Benutzerdaten verantwortlich ist.
  • security_plane: Any: Eine Instanz des Sicherheitsplans, die für die Token-Generierung, -Validierung und -Rückzug zuständig ist.
  • tenant_id: str: Die eindeutige Kennung des Tenants (z.B. der Schule), zu dem die Benutzer gehören.
  • session_ttl_seconds: int: Die Zeitdauer (in Sekunden), über die eine Sitzung gültig ist (Time-To-Live).

    • Aktion:

  1. Speichert die Abhängigkeiten (repository, security_plane, tenant_id, session_ttl_seconds).
  2. Rstellt die Tenant-Konfiguration im security_plane, indem security_plane.register_tenant(self.tenant_id, display_name="Nova School") aufgerufen wird.

    3. Rückgabe:

  • None

2.2. Methode: ensure_user

Diese Methode wird verwendet, um die Existenz eines Benutzers zu prüfen und gegebenenfalls die Kontodaten zu aktualisieren oder neu zu erstellen. Parameter:

  • username: str: Der gewünschte Benutzername.
  • password: str: Das Passwort des Benutzers.
  • role: str: Die Rolle des Benutzers ("student", "teacher", "admin").
  • display_name: str: Der angezeigte Name des Benutzers.
  • permissions: dict[str, Any] | None = None: Optionales Dictionary mit spezifischen Berechtigungen.

    • Aktion:

  1. Existenzprüfung: Prüft, ob der Benutzer mit dem username existiert.
  • Wenn Benutzer nicht existiert:
  • Hashet das Passwort (hash_password(password)).
  • Erstellt den Benutzer über self.repository.create_user(...) mit den bereitgestellten Daten.
  • Gibt die erstellte Benutzerinformation zurück.
  • Wenn Benutzer existiert:
  • Prüft, ob die angezeigten Daten (display_name, role, status="active") mit den gespeicherten Daten übereinstimmen.
  • Aktualisierung: Wenn die Daten nicht übereinstimmen, wird die display_name, role und der status auf "active" aktualisiert (self.repository.update_user_account(...)).
  • Berechtigungen: Wenn permissions angegeben sind, werden die Benutzerberechtigungen aktualisiert (self.repository.update_user_permissions(...)).
  • Gibt die aktualisierte Benutzerinformation zurück.

    • Rückgabe:

  • dict[str, Any]: Die vollständigen Benutzerdaten.

2.3. Methode: create_user

Diese Methode wird ausschließlich zur Erstellung eines neuen Benutzers verwendet. Parameter:

  • username: str: Der eindeutige Benutzername.
  • password: str: Das Passwort.
  • role: str: Die Rolle des Benutzers (muss "student", "teacher" oder "admin" sein).
  • display_name: str: Der angezeigte Name.
  • permissions: dict[str, Any] | None = None: Optionales Dictionary mit spezifischen Berechtigungen.

    • Validierung (Pre-Checks):

  1. Prüft, ob username, display_name und password nicht leer sind.
  2. Prüft, ob role eine der erlaubten Werte ist.
  3. Prüft, ob der Benutzer bereits existiert.

    4. Aktion:

  4. Hashet das Passwort.
  5. Erstellt den Benutzer über self.repository.create_user(...).

    6. Rückgabe:

  • dict[str, Any]: Die erstellten Benutzerdaten.

2.4. Methode: login

Diese Methode validiert Anmeldedaten und generiert eine Sitzung (Token) bei erfolgreicher Authentifizierung. Parameter:

  • username: str: Der Benutzername.
  • password: str: Das Passwort.

    • Aktion:

  1. Benutzerprüfung: Holt den Benutzer über das Repository. Prüft, ob der Benutzer existiert und der Status "active" ist. Fällt bei Nichtbefriedigung auf PermissionError.
  2. Passwortvalidierung: Vergleicht das eingegebene Passwort mit dem gespeicherten Hash und Salt (verify_password(...)). Fällt bei Nichtbefriedigung auf PermissionError.
  3. Token-Generierung: Ruft self.security_plane.issue_token(...) auf, um ein Token zu generieren, das den Tenant, den Benutzernamen, die Rollen und die Session-TTL enthält.
  4. Sitzungsabfrage: Ruft self.session_from_token(...) auf, um die Sitzungsdetails aus dem Token zu extrahieren. Fällt bei fehlgeschlagener Sitzungsabfrage auf PermissionError.
  5. Rückgabe: Gibt das generierte Token und die erstellte SessionContext zurück.

    6. Rückgabe:

  • tuple[str, SessionContext]: Ein Tupel, bestehend aus dem Token-String und dem SessionContext.

2.5. Methode: session_from_token

Diese Methode validiert ein eingehendes Token und rekonstruiert die vollständigen Sitzungsdetails. Parameter:

  • token: str: Der zu validierende Token-String.

    • Aktion:

  1. Token-Authentifizierung: Ruft self.security_plane.authenticate(token) auf, um die Authentifizierung durchzuführen. Fällt bei None auf None zurück.
  2. Benutzer-Identifikation: Nimmt die subject des Authentifizierungsprinzips und sucht den entsprechenden Benutzer im Repository.
  • Prüft, ob der Benutzer existiert und "active" ist. Fällt bei Nichtbefriedigung auf None zurück.
  1. Gruppen- und Berechtigungs-Rekonstruktion:
  • Ruft self.repository.list_user_groups(user["username"]) auf, um die Gruppen des Benutzers abzurufen.
  • Führt eine Logik zur Berechtigungsresolution durch (resolve_permissions), die die Berechtigungen aus den Gruppen und die expliziten Berechtigungen des Benutzers kombiniert.
  1. Sitzungs-Objekt Erstellung: Erstellt und gibt ein SessionContext zurück, das alle rekonstruierten Informationen (Token, User, Gruppen, Berechtigungen) enthält.

    2. Rückgabe:

  • SessionContext | None: Ein SessionContext-Objekt, falls die Validierung erfolgreich war, oder None, falls die Validierung fehlschlägt.

2.6. Methode: logout

Diese Methode wird verwendet, um eine Sitzung zu beenden, indem das Token ungültig gemacht wird. Parameter:

  • token_id: str: Die eindeutige ID des Tokens, das ungültig gemacht werden soll.

    • Aktion:

  1. Ruft self.security_plane.revoke_token(token_id) auf, um das Token im System zu widerrufen.

    2. Rückgabe:

  • dict[str, Any]: Eine Antwort, die durch den Sicherheitsplan zurückgegeben wird (Details zum Widerruf).

    • ---

3. Abhängigkeiten und Schlüsselkonzepte

Abhängigkeit Zweck
SchoolRepository Datenpersistenz (CRUD-Operationen für Benutzer).
Any (Security Plane) Verantwortlich für die gesamte Authentifizierungslogik (Token-Generierung, Passwort-Hashing, Token-Validierung).
hash_password Hilfsfunktion zur Erstellung von Passwort-Hashes und Salts.
verify_password Hilfsfunktion zur Überprüfung von Passwörtern.
resolve_permissions Hilfsfunktion zur komplexen Berechtigungslogik (Kombination von Rollen, Gruppen und expliziten Berechtigungen).
SessionContext Datenstruktur, die alle Informationen einer aktiven Sitzung (Token, Benutzer, Gruppen, Berechtigungen) kapselt.

4. Fehlerbehandlung (Exceptions)

Die Klasse wirft spezifische Python-Standardausnahmen, um verschiedene Authentifizierungsfehler abzufangen:

  • **ValueError**: Wird bei ungültigen Eingaben in create_user (z.B. leere Felder) oder bei ungültigen Rollen in create_user geworfen.
  • **PermissionError**: Wird bei fehlgeschlagener Authentifizierung (ungültige Zugangsdaten, fehlende Sitzung) in login und session_from_token geworfen.

5. Sicherheitshinweise

  1. Passwortspeicherung: Passwörter werden nicht im Klartext gespeichert, sondern durch Hashing (hash_password) geschützt.
  2. Sitzungsmanagement: Sitzungen werden über JWT-ähnliche Token verwaltet, die eine begrenzte Gültigkeitsdauer (session_ttl_seconds) haben.
  3. Tenant-Isolation: Die Tenant-ID wird bei der Registrierung im Sicherheitsplan verwendet, um die Datenisolation zu gewährleisten.
  4. Berechtigungsmanagement: Die Berechtigungsprüfung in session_from_token ist komplex und kombiniert Rollen, Gruppen und individuelle Berechtigungen, was eine robuste Zugriffskontrolle ermöglicht.

Technische Dokumentation: Passwort-Hashing- und -Verifizierung

1. Einführung

Diese Dokumentation beschreibt die Funktionen hash_password und verify_password aus der Datei auth.py, die für die sichere Speicherung und Überprüfung von Passwörtern verwendet werden. Die Implementierung basiert auf dem Password-Hashing-Standard PBKDF2 (Password-Based Key Derivation Function 2), was eine robuste Methode zur Sicherung von Passwörtern darstellt, indem es eine bewusste Rechenzeit (durch Iterationen) und einen zufälligen Salt verwendet.

2. Funktionen

2.1. hash_password(password: str) -> tuple[str, str]

Diese Funktion nimmt ein Klartext-Passwort entgegen und generiert einen verschlüsselten Hash sowie einen Salt. Beschreibung: Die Funktion erzeugt einen einzigartigen, zufälligen Salt und verwendet diesen zusammen mit dem Passwort, um einen Hash zu generieren. Der Hash wird mittels PBKDF2 mit dem SHA-256-Hash-Algorithmus berechnet. Parameter:

  • password: str: Das Klartext-Passwort, das gehasht werden soll.

    • Rückgabe:

  • tuple[str, str]: Eine Tupel, das die beiden generierten Werte enthält:
  1. Erster String (Salt): Der Base64 URL-safe kodierte Salt.
  2. Zweiter String (Hash): Der Base64 URL-safe kodierte Hash-Wert.

    3. Details zur Implementierung:

  3. Salt Generierung: Ein zufälliger 16-Byte-Salt wird mithilfe von os.urandom(16) generiert.
  4. Hashing: Die Funktion hashlib.pbkdf2_hmac wird verwendet, um den Hash zu berechnen:
  • Algorithmus: sha256
  • Input: Encodedes Passwort (password.encode("utf-8")), der generierte Salt und eine Iterationszahl von $200\_000$.
  1. Kodierung: Sowohl der Salt als auch der resultierende Hash werden mittels base64.urlsafe_b64encode kodiert und in ASCII-Strings umgewandelt, um sie als Textdaten zu speichern.

    2. Sicherheitshinweis: Die Iterationszahl von $200\_000$ ist eine bewusste Entscheidung, um die Rechenzeit zu erhöhen und Brute-Force-Angriffe zu verlangsamen. ---

2.2. verify_password(password: str, salt_text: str, hash_text: str) -> bool

Diese Funktion prüft, ob ein gegebenes Passwort mit den bereitgestellten Salt- und Hash-Werten übereinstimmt. Beschreibung: Diese Funktion dekodiert den gespeicherten Salt und den Hash, berechnet den Hash des eingegebenen Passworts mit dem dekodierten Salt und vergleicht den neuen Hash mit dem gespeicherten Hash mithilfe eines sicheren Vergleichs (hmac.compare_digest). Parameter:

  • password: str: Das Passwort, das der Benutzer eingibt, um die Identität zu verifizieren.
  • salt_text: str: Der Base64 URL-safe kodierte Salt, der beim Hashing gespeichert wurde.
  • hash_text: str: Der Base64 URL-safe kodierte Hash-Wert, der beim Hashing gespeichert wurde.

    • Rückgabe:

  • bool: True, wenn das Passwort korrekt ist; False, andernfalls.

    • Details zur Implementierung:

  1. Salt Dekodierung: Der salt_text wird mittels base64.urlsafe_b64decode in den ursprünglichen 16-Byte-Salt umgewandelt.
  2. Hash Dekodierung: Der hash_text wird mittels base64.urlsafe_b64decode in den ursprünglichen Hash-Wert umgewandelt.
  3. Actual Hash Berechnung: Der Hash des eingegebenen password wird exakt so berechnet, wie er beim Hashing erstellt wurde:
  • Algorithmus: sha256
  • Input: Encodedes Passwort, der dekodierte Salt und die Iterationszahl von $200\_000$.
  1. Sicherer Vergleich: Die Funktion hmac.compare_digest(actual, expected) wird verwendet, um den berechneten Hash mit dem dekodierten erwarteten Hash zu vergleichen. Dies ist entscheidend, um Timing-Angriffe zu verhindern.

3. Abhängigkeiten

Für die korrekte Ausführung dieser Funktionen sind die folgenden Standard-Python-Module erforderlich:

  • os: Für die Generierung des zufälligen Salts (os.urandom).
  • hashlib: Für die Implementierung des PBKDF2-Hashing-Algorithmus.
  • base64: Für die URL-safe Kodierung und Dekodierung der Salt- und Hash-Strings.
  • hmac: Für den sicheren Vergleich von Hashes (hmac.compare_digest).

4. Zusammenfassung der Sicherheitsmaßnahmen

Die Implementierung folgt modernen Best Practices für Passwort-Management:

  1. Salt-Verwendung: Jeder Hash wird mit einem einzigartigen, zufälligen Salt kombiniert, was die Angriffsanstrengung, die durch Rainbow Tables oder gemeinsame Hashes erleichtert wird, massiv erhöht.
  2. PBKDF2: Die Verwendung von PBKDF2 (anstatt einfacher SHA-256) stellt sicher, dass die Berechnung des Hashes bewusst zeitaufwendig gestaltet wird.
  3. Iterationen: Die hohe Iterationszahl ($200\_000$) verlangsamt gezielt Angriffe.
  4. Sicherer Vergleich: Der Vergleich der Hashes erfolgt asynchron (hmac.compare_digest), um Timing-Angriffe zu verhindern.
  5. Base64 URL-Safe: Die Daten werden in einer sicheren, textbasierten Form für die Speicherung und Übertragung kodiert.
