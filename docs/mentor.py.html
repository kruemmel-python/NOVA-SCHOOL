Technische Dokumentation: mentor.py

1. Überblick

Die Datei mentor.py definiert die Kernlogik und die Systemanweisungen für einen KI-Code-Mentor namens "Nova Mentor". Diese Klasse oder dieser Modulteil ist darauf ausgelegt, als interaktiver, sokratischer Code-Mentor in einer Schulumgebung zu fungieren.

2. Abhängigkeiten

Das Modul hängt von folgenden internen Komponenten ab:

  • ./ai_service: Enthält die Funktion _prepare_prompt_with_budget, die wahrscheinlich die Eingabeaufforderung (Prompt) für die KI-Modelle formatiert.
  • ./database: Enthält die Klasse SchoolRepository, die für den Zugriff auf schulbezogene Daten zuständig ist.

3. Konstanten und Globaler Zustand

Name Typ Beschreibung
MENTOR_SYSTEM_PROMPT str Die zentrale Systemanweisung, die dem KI-Modell vorgesagt wird, wie es sich als Nova Mentor verhalten soll (sokratisch, nicht sofortige Lösungen, Fokus auf Konzeptverständnis).
MENTOR_INPUT_TOKEN_BUDGET int Eine definierte Budgetgrenze (2400), die wahrscheinlich für die Generierung von Prompts oder die Ressourcenallokation im Kontext der KI-Interaktion relevant ist.

3.1. MENTOR_SYSTEM_PROMPT Details

Dieser String legt das Verhalten des Mentors fest:

  • Rolle: Nova Mentor, ein sokratischer KI-Code-Mentor für eine Schulumgebung.
  • Verhaltensregeln (Kernprinzip):
  • Keine sofortige Lösung: Gib niemals die vollständige Lösung direkt aus.
  • Interaktive Methode: Stelle zuerst 2 bis 4 gezielte Rückfragen oder Denkanstöße.
  • Verweisbarkeit: Verweise explizit auf relevante Stellen im Code oder auf die Fehlermeldung.
  • Konzepterklärung: Erkläre das zugrunde liegende Konzept knapp.
  • Ausnahme (Explizite Anfrage): Wenn der Nutzer explizit nach der Lösung fragt, liefere eine kompakte Musterlösung inklusive Begründung.
  • Ton: Freundlich, klar und technisch präzise.

Die Datei definiert eine Klasse (deren Definition im bereitgestellten Codeausschnitt abgeschnitten ist, aber impliziert wird, dass sie die Mentor-Logik kapselt).

4.3. Importe

  • from __future__ import annotations: Ermöglicht die Verwendung von Typ-Hinten in älteren Python-Versionen.
  • from typing import Any: Wird verwendet, um flexible Typ-Hinweise für die Eingaben oder Ausgaben zu verwenden.
  • from .ai_service import _prepare_prompt_with_budget: Importiert eine interne Funktion zur Prompt-Erstellung.
  • from .database import SchoolRepository: Importiert die Schnittstelle zur Datenbankverwaltung für schulbezogene Daten.

5. Funktionsweise (Impliziert)

Obwohl der Hauptkörper der Klasse fehlt, lässt sich die Funktion der Datei wie folgt ableiten:

  1. Initialisierung: Die Klasse wird wahrscheinlich mit den Systemanweisungen (MENTOR_SYSTEM_PROMPT) und den Ressourcen (z.B. über SchoolRepository) initialisiert.
  2. Interaktion: Die Hauptmethode der Klasse wird die Benutzeranfrage entgegennehmen.
  3. Prompt-Generierung: Die eingehende Anfrage wird mithilfe von _prepare_prompt_with_budget formatiert, wobei die Systemanweisung und die spezifische Benutzeranfrage kombiniert werden.
  4. KI-Ausführung: Der generierte Prompt wird an ein zugrundeliegendes KI-Modell gesendet.
  5. Antwort-Verarbeitung: Die Antwort der KI wird analysiert, um sicherzustellen, dass die Regeln des MENTOR_SYSTEM_PROMPT eingehalten werden (sokratisch, konzeptuell, etc.).

6. Sicherheit und Design-Überlegungen

  • Prompt Engineering: Die strikte Definition des MENTOR_SYSTEM_PROMPT ist ein zentrales Designelement, um eine konsistente und didaktisch wertvolle Interaktion zu gewährleisten.
  • Abgrenzung: Die Trennung von Datenbankzugriff (SchoolRepository) und der eigentlichen KI-Logik (ai_service) deutet auf ein sauberes, modularisiertes Design hin.
  • Ressourcenmanagement: Die Verwendung von MENTOR_INPUT_TOKEN_BUDGET deutet auf eine mögliche Begrenzung der Komplexität oder Kosten der KI-Aufrufe hin.

    • --- Zusammenfassung: mentor.py dient als die zentrale Schnittstelle für einen didaktisch orientierten KI-Code-Mentor, der durch strenge Systemanweisungen dazu angehalten wird, durch Fragen und Konzepterklärungen zu lehren, anstatt sofortige Antworten zu liefern.

Technische Dokumentation: SocraticMentorService

1. Überblick

Die Klasse SocraticMentorService ist ein zentraler Dienst, der die Logik für die Durchführung einer sokratischen Mentoring-Interaktion in einem Entwicklungsprojekt verwaltet. Sie orchestriert die Interaktion zwischen dem Nutzer (Session), dem Repository (für Chat-Historie) und optional einem Curriculum-Service (für kontextbezogene Lernmaterialien), um dem Nutzer bei der Problemlösung, Debugging und Code-Verbesserung zu helfen.

2. Klasse: SocraticMentorService

2.1. Konstruktor (__init__)

Der Konstruktor initialisiert den Dienst und injiziert notwendige Abhängigkeiten. Signaturen:

def __init__(self, repository: SchoolRepository, *, curriculum_service: Any | None = None) -> None:
self.repository = repository
self.curriculum_service = curriculum_service

Parameter:

  • repository: Eine Instanz der SchoolRepository-Klasse, die für den Zugriff auf Chat-Nachrichten und Projektinformationen zuständig ist.
  • curriculum_service (optional, Any | None): Eine Optionale Instanz eines CurriculumService. Wird verwendet, um Lernkontext (Kursziele, Module, Mentor-Regeln) in die Prompt-Generierung einzubinden.

2.2. Methode: thread

Diese Methode dient dazu, die Chat-Historie eines bestimmten Projekts und Nutzers abzurufen. Signaturen:

def thread(self, session: Any, project: dict[str, Any], *, username: str | None = None) -> list[dict[str, Any]]:
# ... Implementierung ...

Beschreibung: Richtet die Chat-Historie für eine spezifische Projekt-Session ab. Logik:

  1. Nutzerbestimmung: Bestimmt den owner (Username oder session.username).
  2. Berechtigungsprüfung: Prüft, ob der owner entweder der aktuelle session.username ist oder ob die session.is_teacher (True) ist. Andernfalls wird eine PermissionError geworfen.
  3. Room Key Generierung: Erzeugt einen eindeutigen Schlüssel (room_key) basierend auf der project_id und dem owner.
  4. Historienabruf: Ruft die letzten 50 Chat-Nachrichten aus dem Repository für diesen room_key ab.
  5. Formatierung: Formatiert die abgerufenen Nachrichten in eine Liste von Dictionaries, wobei die Metadaten (role, author_display_name, text, created_at, mode) extrahiert werden.

    6. Rückgabe: Eine Liste von Dictionaries, die die Chat-Historie darstellen.

2.3. Methode: prepare

Diese Methode ist der Kern der Prompt-Generierung. Sie sammelt alle relevanten Informationen (Projekt, Nutzeranfrage, Code, Lauf-Output, Historie und optionalen Curriculum-Kontext) und fasst sie in einen optimierten Prompt zusammen. Signaturen:

def prepare(
self,
session: Any,
project: dict[str, Any],
*,
prompt: str,
code: str,
path_hint: str,
run_output: str,
course_id: str = "",
module_id: str = "",
) -> dict[str, str]:
# ... Implementierung ...

Beschreibung: Erstellt den finalen Prompt, der an ein LLM gesendet wird, indem er verschiedene Kontextinformationen strukturiert. Schritte:

  1. Historie Abruf: Ruft die Chat-Historie über self.thread() ab.
  2. Historie Kompression: Erstellt eine kompakte Zusammenfassung der letzten 8 Nachrichten (compact_history).
  3. Curriculum Kontext: Wenn self.curriculum_service vorhanden ist, ruft er den Kontext (curriculum_context) ab.
  4. **Prompt Komposition (_compose_prompt):**
  • Basis-Sektionen: Fügt Informationen über das Projektname, den Dateipfad und die aktuelle Frage hinzu.
  • Curriculum-Integration (falls vorhanden): Fügt detaillierte Informationen zu Kurs, Modul, Lernzielen, bereits bestandenen Modulen und spezifischen Mentor-Regeln hinzu.
  • Lauf-Output: Fügt die letzte Ausgabe oder Fehlermeldung ein.
  • Code-Kontext: Fügt den bereitgestellten Code ein.
  • Verlauf: Fügt die gesamte Chat-Historie ein.
  • System-Anweisung: Fügt die allgemeine Anweisung hinzu, wie der Mentor agieren soll (Debugging-Denken, Clean-Code-Hinweise, Tests).
  • Token-Budgeting: Ruft die finale Prompt-Generierung über _prepare_prompt_with_budget auf, um sicherzustellen, dass die Gesamtgröße des Prompts das definierte MENTOR_INPUT_TOKEN_BUDGET nicht überschreitet.

    • Rückgabe: Ein Dictionary mit zwei Werten:

  • prompt: Der final zusammengestellte, optimierte Prompt-String.
  • system_prompt: Die feste Systemanweisung (MENTOR_SYSTEM_PROMPT).

2.4. Methode: store_reply

Diese Methode speichert die Interaktion (Nutzer-Frage und Mentor-Antwort) im Chat-Repository. Signaturen:

def store_reply(
self,
session: Any,
project: dict[str, Any],
*,
prompt: str,
reply: str,
model: str | None = None,
) -> dict[str, Any]:
# ... Implementierung ...

Beschreibung: Speichert die Benutzeranfrage und die Antwort des Mentors im Chat-Repository. Schritte:

  1. Room Key Generierung: Erzeugt den room_key basierend auf der Projekt-ID und dem aktuellen Nutzer.
  2. Benutzer-Nachricht Speichern: Speichert die Benutzeranfrage (prompt) mit der Rolle "user" und dem mode="mentor".
  3. Mentor-Antwort Speichern: Speichert die Antwort (reply) mit der Rolle "assistant" und dem mode="mentor", inklusive optionalem model-Parameter.
  4. Historie Abruf: Ruft die gesamte Chat-Historie für die Session ab (self.thread(session, project)).
  5. Rückgabe: Gibt ein Dictionary zurück, das die Antwort, den verwendeten Modellnamen und die aktualisierte Chat-Historie enthält.

2.5. Statische Methoden

_room_key(project_id: str, username: str) -> str

Generiert einen standardisierten Schlüssel für den Chat-Kontext.

  • Format: f"mentor:{project_id}:{username}"

_compose_prompt(...) -> str

Die private, komplexe Methode, die die gesamte Logik zur Erstellung des finalen LLM-Prompts implementiert. Sie ist verantwortlich für die Strukturierung von Projekt-, Code-, Historie- und Kontextinformationen.

3. Schlüsselkonzepte und Abhängigkeiten

Konzept Beschreibung Abhängigkeiten
Sokratischer Ansatz Der Dienst ist darauf ausgelegt, durch gezielte Fragen, Hinweise und Debugging-Denken den Nutzer zur Selbstfindung und Verbesserung zu führen. LLM-Fähigkeiten (impliziert durch die Prompt-Struktur).
Kontextmanagement Der Dienst kann dynamisch Kontext aus dem Curriculum (Lernziele, Regeln) und aus der Chat-Historie ziehen. SchoolRepository, CurriculumService.
Token-Management Die Methode _prepare_prompt nutzt MENTOR_INPUT_TOKEN_BUDGET und reserved_tokens, um sicherzustellen, dass die generierte Prompt-String die Token-Limits des zugrundeliegenden Modells respektiert. Konstanten (MENTOR_INPUT_TOKEN_BUDGET, reserved_tokens).
Repository-Abstraktion Die Interaktion mit der Speicherung und Abfrage von Chat-Daten erfolgt über die SchoolRepository-Abstraktion. SchoolRepository.
