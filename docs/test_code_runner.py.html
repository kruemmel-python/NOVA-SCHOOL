Technische Dokumentation: test_code_runner.py

Version: 1.0 Datum: 2026-04-06 Autor: Ralf Krümmel

1. Überblick

Diese Datei (test_code_runner.py) scheint eine Testdatei zu sein, die Komponenten aus dem nova_school_server-Modul testet, insbesondere die Klasse CodeRunner und deren Abhängigkeiten. Sie verwendet Mocking-Techniken (unittest.mock.patch) und definiert Platzhalterklassen (_FakeToolSandbox, _FakeRepository), um spezifische Abhängigkeiten zu isolieren und zu testen.

2. Struktur und Komponenten

Die Datei definiert mehrere Klassen, die entweder als Test-Spezialfälle oder als Mock-Implementierungen dienen:

2.1. Importe

Die Datei importiert folgende Module und Klassen:

  • os, tempfile, threading, time, unittest, pathlib.Path, unittest.mock.patch: Standard-Bibliotheken für Systeminteraktionen, temporäre Dateien, Multithreading, Unit-Testing und Mocking.
  • nova_school_server.code_runner.CodeRunner, RunResult, _RawResult: Kernklassen des Code-Runner-Systems.
  • nova_school_server.config.ServerConfig: Konfigurationsklasse.
  • nova_school_server.workspace.WorkspaceManager: Klasse für das Workspace-Management.

2.2. Platzhalterklassen (Mocking-Strukturen)

Diese Klassen dienen dazu, Abhängigkeiten zu simulieren, die im eigentlichen Test nicht direkt implementiert werden sollen.

class _FakeToolSandbox

  • Zweck: Simuliert eine Tool-Sandbox-Funktionalität.
  • Methode: authorize(self, *_args, **_kwargs)
  • Beschreibung: Gibt ein leeres Dictionary ({}) zurück, was impliziert, dass die Authentifizierung erfolgreich ist oder die spezifische Logik der Sandbox für diesen Test ignoriert wird.

class _FakeRepository

  • Zweck: Simuliert eine Repository-Schicht zur Abfrage von Einstellungen.
  • **Konstruktor (__init__):**
  • Erfordert ein settings-Dictionary (dict[str, str]) bei der Initialisierung.
  • **Methode (get_setting):**
  • Parameter: key (der Schlüssel zur gesuchten Einstellung) und ein optionales default.
  • Rückgabe: Gibt den Wert der Einstellung zurück, oder den angegebenen default, falls der Schlüssel nicht existiert.

2.3. Mock-Implementierung der Kernklasse

class _ObservedCodeRunner(CodeRunner)

  • Zweck: Eine erweiterte oder beobachtete Version der CodeRunner-Klasse, die spezifische Zustände speichert.
  • **Konstruktor (__init__):**
  • Erbt von CodeRunner.
  • Initialisiert interne Attribute:
  • self.last_command: list[str] | None: Speichert den zuletzt ausgeführten Befehl.
  • self.last_env: dict[str, str] | None: Speichert die zuletzt verwendeten Umgebungsvariablen.
  • **Methode (_execute):**
  • Beschreibung: Diese Methode wird wahrscheinlich von der Basisklasse übert뤄fit oder überschrieben, um die Ausführung eines Codes zu simulieren.
  • Logik:
  1. Speichert den eingegebenen command in self.last_command.
  2. Kopiert die eingegebene Umgebung (env) in self.last_env.
  3. Richtet eine RunResult-Instanz zurück, die den run_id, language und den command enthält.

3. Zusammenfassung der Funktion

Die Datei dient primär dazu, die Code-Ausführungslogik (CodeRunner) unter kontrollierten Bedingungen zu testen. Durch die Verwendung von _FakeToolSandbox und _FakeRepository wird sichergestellt, dass die Tests nicht von externen, komplexen Ressourcen (wie tatsächliche Tools oder Datenbanken) abhängig sind. Die Klasse _ObservedCodeRunner stellt eine spezifische Implementierung bereit, die den Verlauf der letzten Ausführung (Befehl und Umgebung) protokolliert.

4. Anwendungsfälle (Impliziert)

Basierend auf der Struktur lassen sich folgende Testfälle ableiten:

  1. **Test der CodeRunner-Initialisierung:** Überprüfung, ob die Basisklasse korrekt initialisiert wird.
  2. **Test der _ObservedCodeRunner._execute Logik:** Überprüfung, ob die last_command und last_env korrekt gesetzt werden, wenn die Ausführungsfunktion aufgerufen wird.
  3. Test der Abhängigkeits-Isolation: Sicherstellen, dass die Interaktion mit der Sandbox und dem Repository korrekt simuliert wird, ohne reale Abhängigkeiten zu nutzen.

Technische Dokumentation: _ContainerObservedRunner Klasse

1. Überblick

Die Klasse _ContainerObservedRunner ist eine Implementierung der Klasse CodeRunner und dient dazu, die Ausführung von Befehlen sowohl in einem lokalen Kontext als auch innerhalb eines Containers zu verwalten und zu protokollieren. Sie ist darauf ausgelegt, die notwendigen Schritte für die Ausführung von Code-Tests oder -Aufgaben zu erfassen, einschließlich der Speicherung von Rohbefehlen und der Verwaltung von Umgebungsvariablen und Container-spezifischen Aktionen.

2. Klasse

Name: _ContainerObservedRunner Basisklasse: CodeRunner

3. Eigenschaften (Attributes)

Eigenschaft Typ Beschreibung
self.raw_commands list[list[str]] Eine Liste aller als Rohbefehle (als Listen von Strings) aufgezeichneten Befehle, die von der Runner-Instanz ausgeführt wurden.
self.last_container_command `list[str] None` Speichert den letzten Befehl, der innerhalb eines Containers ausgeführt wurde. Kann None sein, wenn noch kein Container-Befehl ausgeführt wurde.
self.last_container_env `dict[str, str] None` Speichert die Umgebungsvariablen (env), die für den letzten Container-Befehl verwendet wurden. Kann None sein, wenn noch kein Container-Befehl ausgeführt wurde.

4. Methoden (Methods)

4.1. __init__(self, *args, **kwargs) -> None

Zweck: Initialisiert die Klasse und ruft die Konstruktoren der Basisklasse (CodeRunner) auf. Details:

  • Ruft die Konstruktormethode der Basisklasse auf (super().__init__(*args, **kwargs)).
  • Initialisiert die Attribute self.raw_commands, self.last_container_command und self.last_container_env auf ihre Standardwerte ([], None, None).

4.2. _execute_raw(self, command, cwd, stdin_text, env, *, timeout_seconds=None)

Zweck: Simuliert oder protokolliert die Ausführung eines Befehls in einem lokalen Kontext. Parameter:

  • command: Die Liste der Befehle, die ausgeführt werden sollen.
  • cwd: Der aktuelle Arbeitsverzeichnis (Current Working Directory).
  • stdin_text: Der Standardeingabedateninhalt.
  • env: Die Umgebungsvariablen für die Ausführung.
  • timeout_seconds: Optionaler Timeout in Sekunden.

    • Ablauf:

  1. Der eingegebene command wird zur Liste self.raw_commands hinzugefügt.
  2. Der Befehl wird als zusammenhängender String (command_text) erstellt.
  3. Spezialfall (pip install): Wenn der Befehl "pip install" enthält, wird automatisch ein Test-File (demo.py mit value = 1) in das Verzeichnis .nova-python/site-packages im aktuellen Arbeitsverzeichnis (cwd) geschrieben.
  4. Gibt ein _RawResult-Objekt zurück, das die Protokollierung des Rohbefehls enthält.

    5. Rückgabe:

  • Ein _RawResult-Objekt, das die Protokollierung des Rohbefehls enthält.

4.3. _execute_container(self, run_id, language, runtime_executable, image, inner_command, project_root, container_workspace, stdin_text, env, tool_session, permissions)

Zweck: Simuliert oder protokolliert die Ausführung eines Befehls innerhalb eines Docker-Containers. Parameter:

  • run_id: Eine eindeutige Kennung für die Laufzeit.
  • language: Die Programmiersprache.
  • runtime_executable: Der ausführbare Befehl im Container.
  • image: Das Docker-Image, das verwendet wird.
  • inner_command: Die Liste der Befehle, die im Container ausgeführt werden sollen.
  • project_root: Das Stammverzeichnis des Projekts.
  • container_workspace: Das Arbeitsverzeichnis innerhalb des Containers.
  • stdin_text: Der Standardeingabedateninhalt.
  • env: Die Umgebungsvariablen für den Container.
  • tool_session: Ein Objekt für die Tool-Sitzung.
  • permissions: Berechtigungen für die Aktion.

    • Ablauf:

  1. self.last_container_command wird mit dem inner_command gesetzt.
  2. self.last_container_env wird mit der Umgebungsvariable env gesetzt.
  3. Spezialfall (GUI-Snapshot): Wenn der inner_command den String "python_gui_snapshot.sh" enthält:
  • Ein Vorschaubild (gui-preview.png) wird in das Verzeichnis .nova-build innerhalb des container_workspace erstellt.
  • Das Bild wird mit dem Inhalt b"PNG" gespeichert.
  1. Gibt ein RunResult-Objekt zurück, das die Ergebnisse der Container-Ausführung (stdout, stderr, Notes) enthält.

    2. Rückgabe:

  • Ein RunResult-Objekt, das die Ergebnisse der Container-Ausführung protokolliert.

5. Zusammenfassung der Funktionalität

Die Klasse dient als zentraler Mechanismus zur Protokollierung und Verwaltung von Code-Ausführungen, wobei sie spezifische Logik für:

  1. Lokale Befehle (inkl. Abhängigkeitsinstallation).
  2. Container-Befehle (inkl. Umgebungsspeicherung und spezifischer Dateioperationen wie Bild-Generierung).

6. Anwendungsfälle

  • Test-Frameworks: Zur Erfassung und Speicherung aller Befehle, die für die Durchführung von Unit-Tests oder Integrationstests benötigt werden.
  • CI/CD-Pipelines: Zur automatisierten Dokumentation der Schritte, die in verschiedenen Umgebungen (lokal vs. Container) ausgeführt werden.
  • Debugging: Zur Überprüfung, welche Befehle und welche Umgebungsvariablen bei einer bestimmten Ausführung verwendet wurden.

Technische Dokumentation: test_code_runner.py - Session-Management-Klassen

1. Überblick

Diese Dokumentation beschreibt die Struktur und die Implementierung der Klassen _Session und _TeacherSession in der Datei test_code_runner.py. Diese Klassen dienen dazu, den Benutzerkontext, Rollen und Berechtigungen innerhalb eines Simulations- oder Runner-Systems zu verwalten.

2. Klassenstruktur

2.1. Klasse _Session

Die Klasse _Session bildet die Basisklasse für alle Benutzer- oder Sitzungszustände. Sie definiert die Standardeigenschaften und die grundlegenden Berechtigungen, die für eine typische Benutzerrolle gelten. Attribute:

Attribut Typ Beschreibung Standardwert
username str Der eindeutige Benutzername der Sitzung. "student"
role str Die Rolle des Benutzers (z.B. "student", "teacher"). "student"
is_teacher bool Flag, das anzeigt, ob der Benutzer als Lehrer konfiguriert ist. False
permissions dict Ein Dictionary, das die spezifischen Berechtigungen des Benutzers definiert. { "run.python": True, "run.html": True, "web.access": False }

**Details zu permissions:** Das permissions-Dictionary definiert die Zugriffsberechtigungen auf verschiedene Aktionen.

  • "run.python": Erlaubt das Ausführen von Python-Code.
  • "run.html": Erlaubt das Ausführen von HTML-Code.
  • "web.access": Erlaubt den Zugriff auf Webressourcen.

    • Konfiguration (Standardwerte):

  • Benutzername: "student"
  • Rolle: "student"
  • **is_teacher:** False
  • Berechtigungen:
  • "run.python": True
  • "run.html": True
  • "web.access": False

2.2. Klasse _TeacherSession

Die Klasse _TeacherSession erbt von _Session und erweitert diese spezifisch für Benutzer mit der Rolle des Lehrers. Erweiterungen:

  • **username:** Wird auf "teacher" gesetzt.
  • **role:** Wird auf "teacher" gesetzt.
  • **is_teacher:** Wird explizit auf True gesetzt.

    • Verhalten: Diese Klasse übernimmt alle Attribute und Methoden von _Session, fügt aber die spezifische Konfiguration für Lehrer hinzu, insbesondere die explizite Kennzeichnung als Lehrer (is_teacher = True). **Konfiguration (_TeacherSession):**

  • Benutzername: "teacher"
  • Rolle: "teacher"
  • **is_teacher:** True
  • **permissions:** Erhält die Standardberechtigungen von _Session (z.B. "run.python": True, "web.access": False), es sei denn, diese werden in der Klasse explizit überschrieben (was in dieser Definition nicht geschehen ist).

3. Zusammenfassung und Nutzungsszenarien

Klasse Hauptzweck Schlüsselmerkmale Typische Anwendung
_Session Basis-Konfiguration Definiert Standardwerte für Benutzer und Berechtigungen. Generierung von Standard-Sitzungsobjekten.
_TeacherSession Spezialisierte Konfiguration Übernimmt Basisdaten und setzt die Rolle und den Lehrer-Status auf True. Verwaltung von Lehrer-Sitzungen, die spezielle Rechte benötigen.

Beispielhafte Anwendung (Konzeptuell):

# Erstellung einer Standard-Studenten-Sitzung
student_session = _Session()
print(f"Username: {student_session.username}, Is Teacher: {student_session.is_teacher}")
# Erstellung einer Lehrer-Sitzung
teacher_session = _TeacherSession()
print(f"Username: {teacher_session.username}, Role: {teacher_session.role}, Is Teacher: {teacher_session.is_teacher}")

4. Fazit

Die Implementierung nutzt das Prinzip der Klassenhererbung (Inheritance), um eine gemeinsame Basisstruktur (_Session) zu definieren und spezifische Erweiterungen (_TeacherSession) für unterschiedliche Benutzerrollen zu ermöglichen. Dies sorgt für eine saubere und wartbare Verwaltung von Benutzerdaten und deren zugehörigen Zugriffsberechtigungen.
