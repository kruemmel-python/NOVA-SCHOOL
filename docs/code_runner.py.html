Technische Dokumentation: code_runner.py

1. Überblick

Die Datei code_runner.py dient als zentraler Orchestrator für das Ausführen von Code in verschiedenen Umgebungen (Container) basierend auf der Sprache der Datei. Sie implementiert die Logik zur Erkennung der Programmiersprache, zur Auswahl des entsprechenden Docker-Images und zur Durchführung des Code-Executions. Das Skript nutzt verschiedene Bibliotheken wie hashlib für Dateisummen, os und pathlib für Dateimanagement, subprocess für das Ausführen von Shell-Befehlen (wie docker), threading für parallele Operationen und dataclasses für die Strukturierung von Daten.

2. Struktur und Abhängigkeiten

2.1 Importe

Das Skript importiert verschiedene Module und Bibliotheken:

  • Standardbibliotheken: hashlib, os, re, shlex, shutil, subprocess, threading, time, textwrap, uuid.
  • Interne Module:
  • config.ServerConfig: Enthält Serverkonfigurationen.
  • container_seccomp.resolve_seccomp_profile_option: Verantwortlich für die Auflösung von Seccomp-Profiloptionen.
  • permissions.allowed_tool_names: Definiert die erlaubten Tool-Namen für die Ausführung.
  • workspace.WorkspaceManager: Managiert den Arbeitsbereich.

2.2 Konfiguration und Konstanten

Das Skript definiert mehrere wichtige Mappings und Muster, die für die Spracherkennung und die Container-Konfiguration entscheidend sind:

2.2.1 Sprach- und Erweiterungs-Mappings

  • **LANGUAGE_TO_PERMISSION (Dict):** Mappt die Programmiersprache auf den entsprechenden Befehl für die Ausführung in einem Container (z.B. "python" wird zu "run.python").
  • **EXTENSION_TO_LANGUAGE (Dict):** Mappt Dateiendungen auf die Programmiersprache (z.B. .py wird zu "python").

2.2.2 Docker-Image-Definitionen

  • **DEFAULT_CONTAINER_IMAGES (Dict):** Definiert Standard-Docker-Images für die wichtigsten Sprachen (z.B. python verwendet python:3.12-slim).

2.2.3 Regex-Muster für Code-Analyse

Diese Muster werden zur Erkennung spezifischer Code-Elemente verwendet, um die Sprache zu bestimmen oder spezielle GUI-Anforderungen zu identifizieren:

  • **JAVA_PACKAGE_PATTERN (re.Pattern):** Wird verwendet, um Java-Paketnamen zu identifizieren (z.B. package com.example;).
  • **_PYTHON_GUI_IMPORT_PATTERNS (Tuple):** Eine Liste von Tupeln, die Muster definieren, um Importe von GUI-Bibliotheken (wie tkinter, PyQt, wx) zu erkennen.
  • **_PYTHON_GUI_USAGE_PATTERNS (Tuple):** Muster, um die Verwendung von GUI-Initialisierungen (z.B. mainloop(), QApplication()) zu erkennen.
  • **_PYTHON_TRACEBACK_WRAPPER_PATHS (Tuple):** Pfade, die als mögliche Orte für Python-Tracebacks dienen.
  • **_PYTHON_TRACEBACK_WRAPPER_PATTERNS (Tuple):** Regex-Muster, um spezifische Python-Traceback-Zeilen zu identifizieren.

3. Datenstrukturen

3.1 @dataclass(slots=True) CodeRunner

Die Klasse CodeRunner (impliziert durch die Importe, obwohl die Definition hier abgeschnitten ist) wird verwendet, um die notwendigen Informationen für die Code-Ausführung zu kapseln.

  • **slots=True:** Reduziert den Speicherverbrauch der Instanz, indem es die Verwendung von __dict__ vermeidet.

4. Kernfunktionen (Implizit)

Obwohl die Hauptlogik in den folgenden Teilen der Datei liegt, impliziert die Struktur folgende Hauptaufgaben:

  1. Dateianalyse: Lesen der bereitgestellten Datei, um die Dateiendung zu bestimmen.
  2. Sprachbestimmung: Basierend auf der Dateiendung wird die Programmiersprache identifiziert.
  3. Container-Konfiguration: Auswahl des passenden Docker-Images und der notwendigen Befehle (LANGUAGE_TO_PERMISSION).
  4. Code-Handling: Verarbeitung des Codes (impliziert durch die Nutzung von shlex und subprocess zur Ausführung).
  5. GUI-Analyse (Python-spezifisch): Anwendung der definierten Regex-Muster, um festzustellen, ob eine GUI-Umgebung benötigt wird und welche Bibliotheken verwendet werden.
  6. Parallelisierung: Nutzung von threading zur effizienten Verwaltung mehrerer Code-Ausführungen.

5. Schlüsselkonzepte und Designmuster

  • Configuration-Driven Design: Die Abhängigkeiten (Sprachen, Images, Muster) sind zentral in Konstanten und Mappings definiert, was die Wartbarkeit und Erweiterbarkeit erleichtert.
  • Regex-basierte Analyse: Die Verwendung von komplexen Regular Expressions ermöglicht eine flexible Erkennung von Code-Strukturen, die über einfache Dateiendungen hinausgehen (z.B. GUI-Importe).
  • Modulare Struktur: Die Trennung von Konfiguration, Container-Management und Code-Analyse in verschiedene Importe deutet auf ein gut strukturiertes, modulares Design hin.
  • Thread-Safety: Die Verwendung von threading deutet darauf hin, dass das System darauf ausgelegt ist, mehrere Code-Ausführungen gleichzeitig zu verarbeiten.

6. Zusammenfassung

code_runner.py ist das Herzstück eines automatisierten Code-Ausführers. Es transformiert eine Dateiendung in eine vollständige Ausführungsstrategie, die das korrekte Container-Image lädt und die notwendigen Befehle für die Code-Execution bereitstellt. Die detaillierten Regex-Muster zeigen eine starke Fokussierung auf die spezifischen Anforderungen von Python-Umgebungen, insbesondere im Hinblick auf GUI-Bibliotheken.

Technische Dokumentation: RunScheduler Klasse

1. Überblick

Die Klasse RunScheduler ist verantwortlich für die Verwaltung und Steuerung der Ausführung von Aufgaben (oder "Leasen") basierend auf bestimmten Benutzerrollen und globalen/individuellen Limits. Sie implementiert einen Warteschlangen- und Aktivierungsmechanismus, der sicherstellt, dass Ressourcen (oder Ausführungen) gemäß festgelegten Prioritäten und Kapazitäten zugewiesen werden.

2. Klasse: RunScheduler

2.1. Konstruktor (__init__)

Initialisiert die Scheduler-Instanz und setzt die notwendigen internen Zustände. Parameter:

  • repository: Any | None = None: Eine Referenz auf ein Repository-Objekt, das zur Abfrage von Konfigurationswerten (Settings) verwendet wird. Wenn None, werden Standardwerte verwendet.

    • Attribute:

  • self.repository: Das Repository-Objekt zur Konfigurationsabfrage.
  • self._condition: Ein threading.Condition-Objekt, das für die Synchronisation von Zugriffsoperationen und Warteschlangenzugriffen verwendet wird.
  • self._queue: list[dict[str, Any]]: Die Warteschlange für eingehende Anfragen (Requests).
  • self._active_total: int: Die aktuelle Gesamtzahl der aktivierten Ressourcen/Leasen.
  • self._active_by_owner: dict[str, int]: Eine Dictionary, das die Anzahl der aktivierten Ressourcen pro Benutzer (owner_username) speichert.
  • self._sequence: int: Ein Zähler, der die sequentielle Nummerierung der Anfragen verzeichnet.

2.2. Methoden

2.2.1. acquire(self, owner_username: str, role: str) -> SchedulerLease

Diese Methode wird aufgerufen, wenn ein Benutzer eine Ausführung anfordern möchte. Sie führt die folgenden Schritte durch:

  1. Anfrageerstellung: Erstellt eine request-Dictionary mit Metadaten wie request_id (UUID-basiert), owner_username, role, priority (basierend auf der Rolle) und dem Zeitstempel der Einreichung (submitted_at).
  2. Synchronisation: Verriegelt den Zugriff auf die interne Warteschlange (self._condition).
  3. Sequenzierung: Erhöht den internen Sequenzzähler (self._sequence) und fügt die Anfrage zur Warteschlange hinzu.
  4. Priorisierte Sortierung: Sortiert die gesamte Warteschlange basierend auf der Priorität (numerisch) und der Sequenznummer (numerisch).
  5. Aktivitätsprüfung:
  • Es wird geprüft, ob die aktuelle Anfrage die höchste Priorität hat und ob die Bedingungen für die Aktivierung erfüllt sind (self._can_activate).
  • Wenn die Bedingungen erfüllt sind:
  • Die Anfrage wird aus der Warteschlange entfernt.
  • self._active_total wird um 1 erhöht.
  • Die Zählung für den owner_username wird inkrementiert.
  • Ein SchedulerLease-Objekt wird erstellt und zurückgegeben, das alle relevanten Informationen (Lease-ID, Zeit bis zur Aktivierung, Warteschlangenposition) enthält.
  • Wenn die Bedingungen nicht erfüllt sind:
  • Die Methode wartet (self._condition.wait(timeout=0.1)) auf eine Benachrichtigung, um erneut zu prüfen, ob die Bedingungen erfüllt sind oder ob andere Anfragen priorisiert werden müssen.

    • Rückgabe: Ein Objekt vom Typ SchedulerLease, falls die Anfrage erfolgreich aktiviert wurde.

2.2.2. release(self, lease: SchedulerLease | None) -> None

Diese Methode wird aufgerufen, wenn ein zugewiesener Lease beendet wird.

  1. Validierung: Prüft, ob ein gültiges lease-Objekt übergeben wurde.
  2. Synchronisation: Verriegelt den Zugriff auf die interne Zustände.
  3. Aktualisierung der Aktivität:
  • self._active_total wird um 1 reduziert (mindestens 0).
  • Die Zählung für den owner_username wird um 1 reduziert.
  • Falls die Zählung für den Benutzer auf 0 fällt, wird der Eintrag aus self._active_by_owner entfernt.
  1. Benachrichtigung: Ruft self._condition.notify_all() auf, um andere wartende Threads zu informieren, dass sich der Zustand geändert hat.

2.2.3. _can_activate(self, owner_username: str, role: str) -> bool

Prüft die aktuellen Kapazitäten, um eine Aktivierung zu erlauben:

  • Globale Limit: Ist die Gesamtanzahl der aktiven Ressourcen (self._active_total) kleiner als das globale Limit (self._global_limit())?
  • Individuelles Limit: Ist die Anzahl der aktiven Ressourcen für den spezifischen Benutzer (self._active_by_owner.get(owner_username, 0)) kleiner als das Limit für seine Rolle (self._per_owner_limit(role))?

2.2.4. _global_limit(self) -> int

Litet das globale Limit für die Anzahl der gleichzeitig aktiven Ressourcen aus der Konfiguration ab.

  • Konfiguration: Liest den Wert für den Schlüssel "scheduler_max_concurrent_global".
  • Standardwert: Wenn der Schlüssel nicht existiert oder fehlschlägt, wird der Standardwert 4 verwendet.
  • Mindestwert: Der zurückgegebene Wert wird auf max(1, value) gesetzt.

2.2.5. _per_owner_limit(self, role: str) -> int

Litet das spezifische Limit für die Anzahl der gleichzeitig aktiven Ressourcen für eine bestimmte Rolle ab.

  1. Rollen-Mapping: Mappt die eingegebene role auf einen Konfigurationsschlüssel (z.B. "student" $\rightarrow$ "scheduler_max_concurrent_student").
  2. Standardwert-Mapping: Mappt die role auf einen Standardwert (z.B. "student" $\rightarrow$ 1).
  3. Konfigurationsabfrage: Liest den Wert für den entsprechenden Schlüssel aus dem Repository.
  4. Mindestwert: Der zurückgegebene Wert wird auf max(1, value) gesetzt.

2.2.6. _setting_int(self, key: str, default: int, *, minimum: int = 1) -> int

Eine private Hilfsmethode zur sicheren Abfrage und Validierung von Integer-Einstellungen aus dem Repository.

  • Logik: Versucht, den Wert für den key aus dem Repository zu lesen. Fängt alle Exception (z.B. KeyError, ValueError) ab und gibt den default Wert zurück.
  • Validierung: Stellt sicher, dass der zurückgegebene Wert mindestens den Wert von minimum hat.

2.2.7. _priority_for_role(self, role: str) -> int

Definiert die Prioritätswerte für verschiedene Rollen, um die Sortierung in acquire zu steuern:

  • admin: 0 (Höchste Priorität)
  • teacher: 1
  • student: 2 (Niedrigste Priorität)
  • Fehlende Rolle: Gibt den Wert 2 zurück (Standard für "student").

3. Abhängigkeiten und Datenstrukturen

  • **threading.Condition:** Wird für das Thread-Safety-Management verwendet, um den Zugriff auf die Warteschlange und die Zähler zu synchronisieren.
  • **uuid.uuid4().hex[:12]:** Wird verwendet, um eindeutige, kurze Identifikatoren für Anfragen zu generieren.
  • **time.perf_counter():** Wird verwendet, um präzise Zeitstempel für die Einreichung und die Berechnung der Wartezeit zu erfassen.
  • **SchedulerLease (Extern):** Ein extern definiertes Objekt, das die Details einer aktivierten Ausführung speichert.

4. Zusammenfassung der Funktionsweise

Die RunScheduler fungiert als zentraler Gatekeeper für Ressourcenallokation. Anfragen werden in einer priorisierten Warteschlange gesammelt. Die Aktivierung einer Anfrage ist nur möglich, wenn sowohl die globale Kapazität als auch die individuelle Kapazität für den jeweiligen Benutzer und seine Rolle nicht überschritten werden. Die gesamte Logik ist durch threading.Condition sichergestellt, um Race Conditions bei gleichzeitigen Anfragen zu vermeiden.

1. Überblick

2. Klasse: _RawResult

Zweck: Speichert alle relevanten Informationen, die aus einer Code-Ausführung resultieren. Beschreibung: Die Klasse _RawResult ist eine Datenstruktur (wahrscheinlich eine dataclass oder eine einfache Klasse), die die Ausgabe, Fehlerinformationen und Zeitmessungen einer Code-Ausführung kapselt. Attribute:

Attribut Typ Beschreibung
stdout str Die Standardausgabe (Standard Output) des ausgeführten Codes.
stderr str Die Standardfehlerausgabe (Standard Error) des ausgeführten Codes.
returncode int Der Rückgabecode des ausgeführten Prozesses (typischerweise 0 für Erfolg, nicht-null für Fehler).
duration_ms int Die Dauer der Code-Ausführung, gemessen in Millisekunden (ms).
command `list[str] None` Eine Optionale Liste von Strings, die den vollständigen Befehl oder die Kommandosyntax darstellen, die ausgeführt wurde. Kann None sein, wenn der Befehl nicht protokolliert wurde.

3. Verwendung und API

Verwendungsszenario: Diese Klasse wird typischerweise verwendet, um die Ergebnisse von Funktionen oder Methoden zu sammeln, die externe Befehle ausführen. Sie ermöglicht eine konsistente und leicht zugängliche Darstellung komplexer Ausführungsdaten. Beispiel (Konzeptuell):

# Angenommene Nutzung
result = _RawResult(

)

4. Technische Details und Design-Überlegungen

  • Datenintegrität: Durch die explizite Definition der Felder (stdout, stderr, returncode, duration_ms, command) wird sichergestellt, dass alle kritischen Metriken einer Code-Ausführung erfasst werden.
  • Flexibilität: Das optional definierte Attribut command erlaubt es, die Struktur der Klasse zu halten, während sie auch Fälle unterstützt, in denen der Befehl selbst nicht relevant ist (z.B. bei internen Framework-Aufrufen).
  • Typisierung: Die Verwendung von Python-Typen (str, int, list[str] | None) verbessert die Lesbarkeit und die Wartbarkeit des Codes.
