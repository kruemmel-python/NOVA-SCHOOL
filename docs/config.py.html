Technische Dokumentation: config.py

1. Überblick

Die Datei config.py dient als zentrale Konfigurationsdatei für die Anwendung. Sie definiert Konstanten für Schlüssel, die in Konfigurationsdateien (wahrscheinlich JSON-Dateien) verwendet werden, und definiert eine Dataclass-Struktur, die zur Strukturierung der Konfigurationen ermöglicht.

2. Abhängigkeiten

  • json: Für die Verarbeitung von JSON-Konfigurationsdaten.
  • os: Für den Zugriff auf das Betriebssystem (möglicherweise zur Pfadverwaltung oder Umgebungsvariablen).
  • dataclasses: Zur Erstellung von Datenstrukturen (speziell für die Konfigurationsstruktur).
  • pathlib.Path: Für die objektorientierte Handhabung von Dateipfaden.
  • typing: Für Typisierung (z.B. Any).

3. Konstanten und Definitionen

3.1. RUNTIME_FILE_CONFIG_KEYS

Dies ist eine Tupel von Strings, die die Schlüssel definieren, die in einer Konfigurationsdatei (wahrscheinlich einer JSON-Datei) für die Laufzeitkonfiguration (RUNTIME_FILE_CONFIG) erwartet werden. Zweck: Dient als zentrale Referenz für die erwarteten Konfigurationsparameter. Definition:

RUNTIME_FILE_CONFIG_KEYS = (
"host",
"port",
"session_ttl_seconds",
"run_timeout_seconds",
"live_run_timeout_seconds",
"tenant_id",
)

Erklärte Schlüssel:

Schlüssel Beschreibung Typ (Impliziert)
host Der Hostname oder die IP-Adresse des Servers. String
port Der Netzwerkport, auf dem der Dienst laust. Integer/String
session_ttl_seconds Die Zeitdauer (in Sekunden), bevor eine Sitzung abgelaufen ist. Integer
run_timeout_seconds Die maximale Zeit (in Sekunden), die ein Hauptlaufprozess aufhalten darf. Integer
live_run_timeout_seconds Die maximale Zeit (in Sekunden), die ein Live-Laufprozess aufhalten darf. Integer
tenant_id Die eindeutige Kennung des Tenants oder der Untermiete. String/Integer

3.2. Dataclass: Konfigurationsstruktur

Die Datei definiert eine dataclass, die als Struktur für die Konfigurationen dient. Definition:

@dataclass(slots=True)class <|end|>

Details:

  • **@dataclass**: Dekoriert die Klasse, um automatisch Methoden wie __init__, __repr__ und andere Dataclass-Funktionen zu generieren.
  • **slots=True**: Eine Optimierung, die die Anzahl der Attribute reduziert und den Speicherverbrauch senkt, indem es verhindert, dass für jede Instanz eine neue __dict__ erstellt wird. Dies ist nützlich für Konfigurationsobjekte, die viele Instanzen erzeugen könnten.

    • Hinweis: Die tatsächliche Definition der Dataclass (die Attribute, die sie enthält) fehlt im bereitgestellten Code-Snippet. Es wird angenommen, dass diese Klasse die Daten aus der Konfigurationsdatei parst und strukturiert.

4. Verwendung und API-Design (Impliziert)

Basierend auf der Struktur lässt sich ableiten, dass die Anwendung folgende Schritte durchführt:

  1. Konfigurationsladen: Eine JSON-Datei wird geladen.
  2. Validierung/Mapping: Die Schlüssel aus RUNTIME_FILE_CONFIG_KEYS werden verwendet, um die entsprechenden Werte aus der JSON-Datei zu extrahieren.
  3. Objektinitialisierung: Die extrahierten Werte werden verwendet, um eine Instanz der definierten Dataclass zu erstellen.

    4. Beispielhafte Nutzung (Hypothetisch):

# Angenommen, es existiert eine Klasse, die die Konfiguration parst
def load_config(json_data: dict) -> YourConfigClass:
# Hier würde die Logik die keys aus RUNTIME_FILE_CONFIG_KEYS verwenden,
# um die Daten zu extrahieren und in die Dataclass zu mappen.
config_instance = YourConfigClass(
host=json_data.get("host"),
port=json_data.get("port"),
# ... weitere Attribute
)
return config_instance
# Beispiel für die Nutzung
# config = load_config(json.load(open("config.json")))
# print(config.host)

5. Zusammenfassung und Wartungshinweise

  • Struktur: Die Verwendung von dataclasses sorgt für eine klare, wartbare Struktur der Konfiguration.
  • Konstanten: Die explizite Definition von RUNTIME_FILE_CONFIG_KEYS macht die Abhängigkeiten der Konfiguration transparent.
  • Wartung: Sollten neue Laufzeitkonfigurationsparameter hinzugefügt werden, muss sowohl die Liste RUNTIME_FILE_CONFIG_KEYS als auch die Definition der Dataclass entsprechend aktualisiert werden.

Technische Dokumentation: ServerConfig Klasse

1. Übersicht

Die Klasse ServerConfig dient dazu, die Konfiguration für einen Server-Prozess zu definieren und zu verwalten. Sie hält alle notwendigen Pfade, Netzwerkparameter und spezifischen Einstellungen für die Anwendung bereit.

2. Klasse: ServerConfig

Die Klasse ServerConfig ist eine Klasse, die die Struktur und die initialen Werte für die Serverkonfiguration abbildet.

2.1. Attribute

Die Klasse definiert die folgenden Attribute:

Attribut Typ Beschreibung Standardwert
base_path Path Der absolute Pfad zur Basis des Konfigurationsverzeichnisses. Nicht initialisiert (muss über from_base_path gesetzt werden)
data_path Path Der Pfad zum Verzeichnis für Daten. Nicht initialisiert
docs_path Path Der Pfad zum Verzeichnis für Dokumentation. Nicht initialisiert
users_workspace_path Path Der Pfad zum Workspace für Benutzer. Nicht initialisiert
groups_workspace_path Path Der Pfad zum Workspace für Gruppen. Nicht initialisiert
static_path Path Der Pfad zum Verzeichnis für statische Dateien. Nicht initialisiert
database_path Path Der Pfad zur Datenbankdatei. Nicht initialisiert
host str Der Netzwerk-Host, auf dem der Server lauscht. "0.0.0.0"
port int Der Netzwerk-Port, auf dem der Server lauscht. 8877
session_ttl_seconds int Die Lebensdauer der Sitzung in Sekunden (Time-To-Live). 43200
run_timeout_seconds int Die Zeit, die der Server für eine normale Ausführung hat, bevor er beendet wird. 20
live_run_timeout_seconds int Die Zeit, die der Server für eine Live-Ausführung hat, bevor er beendet wird. 300
tenant_id str Der eindeutige Identifier des Tenants. "nova-school"
school_name str Der Name der Schule/des Servers. "Nova School Server"

2.2. Methoden

from_base_path(cls, base_path: Path) -> "ServerConfig"

Dies ist eine Class Method, die die primäre Methode zur Initialisierung der ServerConfig ist. Sie liest die Konfiguration basierend auf einem bereitgestellten Basis-Pfad und kombiniert diese mit Umgebungsvariablen oder einem Konfigurations-Payload. Funktionsweise:

  1. Pfad-Resolution: Der eingegebene base_path wird aufgelöst (resolve(strict=False)).
  2. Payload-Laden: Es wird eine Funktion load_server_config_payload(base_path) aufgerufen, um spezifische Konfigurationsdaten zu laden.
  3. Paket-Pfad-Resolution: Es wird der Pfad zum Paket (resolve_package_path(base_path)) berechnet.
  4. **Konfigurations-Resolver (env_or_payload):** Eine interne Helper-Funktion wird definiert, um den Wert für ein gegebenes Schlüssel-Wert-Paar zu bestimmen:
  • Zuerst wird der Wert aus den Umgebungsvariablen (z.B. NOVA_SCHOOL_HOST) abgerufen.
  • Falls nicht vorhanden, wird der Wert aus dem Konfigurations-Payload (payload.get(key)) abgerufen.
  • Als letzter Fall wird der Standardwert (default) verwendet.
  1. Pfad-Berechnung: Die verschiedenen Pfade werden relativ zum base_path konstruiert:
  • data_path = base_path / "data"
  • docs_path = data_path / "docs"
  • workspaces_path = data_path / "workspaces"
  • users_workspace_path = workspaces_path / "users"
  • groups_workspace_path = workspaces_path / "groups"
  • static_path = package_path / "static"
  • database_path = data_path / "school.db"
  1. Instanziierung: Die Klasse wird mit den berechneten Werten und den durch die Resolveren ermittelten Werten für die Standardwerte initialisiert.

3. Abhängigkeiten

  • Path (von pathlib): Für die Verwaltung von Dateipfaden.
  • os: Für den Zugriff auf Umgebungsvariablen (os.environ.get).
  • Externe Funktionen (nicht im Code sichtbar, aber impliziert):
  • load_server_config_payload(base_path)
  • resolve_package_path(base_path)

4. Verwendung (Usage)

Die Konfiguration wird typischerweise durch Aufruf der Klasse ServerConfig über die Klasse-Methode from_base_path erstellt:

from pathlib import Path
# Angenommen, load_server_config_payload und resolve_package_path sind definiert
# Beispiel: Initialisierung
config = ServerConfig.from_base_path(Path("/home/user/server_root"))
# Zugriff auf Konfigurationswerte
print(f"Server Host: {config.host}")
print(f"Datenpfad: {config.data_path}")
print(f"Datenbankpfad: {config.database_path}")
print(f"Session TTL: {config.session_ttl_seconds} Sekunden")

5. Design-Überlegungen

  • Konfigurationsflexibilität: Die Klasse kombiniert statische Standardwerte mit der Möglichkeit, diese durch Umgebungsvariablen oder externe Payloads überschreiben zu lassen. Dies ermöglicht eine flexible Konfiguration über verschiedene Deployment-Umgebungen hinweg.
  • Pfadmanagement: Die Verwendung von pathlib.Path stellt sicher, dass alle Pfade plattformunabhängig und korrekt behandelt werden.
  • Separation von Logik: Die Logik zur Abfrage von Konfigurationswerten (env_or_payload) ist sauber von der eigentlichen Pfadkonstruktion getrennt.
  • Klassen-Methode: Die Verwendung einer @classmethod für die Initialisierung ist üblich, wenn die Konfiguration stark von externen Faktoren (wie einem Startpfad) abhängt.

Technische Dokumentation: Konfiguration und Pfadverwaltung

Übersicht

Diese Dokumentation beschreibt die Funktionen, die für die Verwaltung von Pfaden, das Laden und Speichern von Serverkonfigurationsdaten verantwortlich sind, wie sie in der Datei config.py implementiert sind.

Funktionen

1. resolve_package_path(base_path: Path) -> Path

Beschreibung: Diese Funktion dient dazu, einen spezifischen Pfad innerhalb einer gegebenen Basis-URL zu einem eindeutigen, gültigen Pfad zu auflösen, der die Struktur eines Pakets (insbesondere die Existenz von __init__.py und einem static-Verzeichnis) widerspiegelt. Parameter:

  • base_path: Path: Der Ausgangspunkt des Pfadverarbeitungs-Prozesses.

    • Rückgabe:

  • Path: Der auflösende, gültige Pfad.

    • Logik:

  1. Basis-Pfad auflösen: Der eingegebene base_path wird aufgelöst (.resolve(strict=False)).
  2. Kandidatenliste: Es werden drei Hauptkandidaten für den korrekten Pfad geprüft:
  • base_path / "nova_school_server"
  • base_path
  • package_root (Der Verzeichnisbaum des aktuellen Skripts, __file__, inklusive des Elternverzeichnisses).
  1. Validierung: Für jeden Kandidaten wird geprüft, ob die Bedingung erfüllt ist:
  • Das Verzeichnis resolved / "__init__.py" existiert.
  • Das Verzeichnis resolved / "static" existiert.
  1. Rückgabe: Wenn die Bedingung für einen Kandidaten erfüllt ist, wird dieser aufgelöste Pfad zurückgegeben.
  2. Standardfall: Falls keine der spezifischen Kandidaten die Bedingung erfüllt, wird der package_root als Standardwert zurückgegeben.

    3. Anwendungsfall: Nützlich, um dynamisch den korrekten Pfad zu einem Server- oder Paket-Root zu bestimmen, basierend auf verschiedenen möglichen Konfigurationsstrukturen. ---

2. load_server_config_payload(base_path: Path) -> dict[str, Any]

Beschreibung: Diese Funktion lädt die Konfigurationsdaten aus einer JSON-Datei, die im angegebenen base_path gesucht wird. Parameter:

  • base_path: Path: Der Verzeichnisbaum, in dem die server_config.json Datei gesucht werden soll.

    • Rückgabe:

  • dict[str, Any]: Ein Dictionary, das die geladenen Konfigurationsdaten enthält. Wird leer ({}) zurückgegeben, wenn die Datei nicht existiert oder eine Fehlermeldung beim Parsen auftritt.

    • Logik:

  1. Pfadkonstruktion: Der vollständige Pfad zu server_config.json wird konstruiert und aufgelöst.
  2. Existenzprüfung: Es wird geprüft, ob die Datei existiert. Falls nicht, wird ein leeres Dictionary zurückgegeben.
  3. JSON-Parsing: Die Datei wird in UTF-8-Kodierung gelesen und versucht, sie mit json.loads() zu parsen.
  4. Fehlerbehandlung: Bei jeglichen Exception (z.B. Syntaxfehler im JSON) wird die Funktion sicher mit einem leeren Dictionary zurückgegeben.
  5. Typ-Check: Falls das Parsen erfolgreich ist, wird sichergestellt, dass das Ergebnis ein Dictionary ist, andernfalls wird ein leeres Dictionary zurückgegeben.

    6. Anwendungsfall: Zur dynamischen Lesezugriff auf die Serverkonfiguration. ---

3. save_server_config_payload(base_path: Path, updates: dict[str, Any]) -> dict[str, Any]

Beschreibung: Diese Funktion lädt die aktuelle Konfiguration, kombiniert sie mit den bereitgestellten Updates und speichert das resultierende JSON-Dokument zurück in der Konfigurationsdatei. Parameter:

  • base_path: Path: Der Verzeichnisbaum, in dem die server_config.json Datei gespeichert werden soll.
  • updates: dict[str, Any]: Ein Dictionary mit den Schlüssel-Wert-Paaren, die in die bestehende Konfiguration hinzugefügt oder überschrieben werden sollen.

    • Rückgabe:

  • dict[str, Any]: Das aktualisierte Konfigurations-Dictionary, das erfolgreich gespeichert wurde.

    • Logik:

  1. Basis-Pfad auflösen: Der base_path wird aufgelöst.
  2. Laden: Die aktuelle Konfiguration wird über load_server_config_payload geladen.
  3. Aktualisierung: Die geladene Konfiguration wird mit den updates mithilfe der .update()-Methode kombiniert.
  4. Speichern:
  • Der aktualisierte Payload wird in eine JSON-String-Repräsentation umgewandelt (json.dumps) mit ensure_ascii=False und indent=2 für eine lesbare Formatierung.
  • Der String wird in die Datei server_config.json im angegebenen base_path geschrieben.
  1. Rückgabe: Das finale, aktualisierte Konfigurations-Dictionary wird zurückgegeben.

    2. Anwendungsfall: Zur persistenten Speicherung und Aktualisierung der Serverkonfiguration.

Abhängigkeiten

  • **pathlib.Path**: Für die Manipulation von Dateipfaden.
  • **json**: Für das Serialisieren und Deserialisieren von JSON-Daten.
  • **Any**: Typ-Hint für die flexible Struktur der Konfigurationsdaten.

Technische Dokumentation: Konfiguration der Laufzeitumgebung

Übersicht

Diese Dokumentation beschreibt die Funktionen, die zur Verwaltung und Abgleich der Konfigurationen für die aktive und gespeicherte Laufzeitumgebung verwendet werden. Die Funktionen in config.py dienen dazu, die Konfigurationen aus dem ServerConfig-Objekt zu extrahieren, sie aus einer gespeicherten Datei zu laden und zu prüfen, ob ein Neustart der Laufzeitumgebung erforderlich ist.

Funktionen

1. active_runtime_config(config: ServerConfig) -> dict[str, Any]

Beschreibung: Diese Funktion extrahiert die aktuellen, aktiven Laufzeitkonfigurationsparameter aus einem ServerConfig-Objekt und formatiert sie in ein Dictionary. Parameter:

  • config: Ein Objekt vom Typ ServerConfig, das die aktuelle Konfiguration enthält.

    • Rückgabe:

  • Ein Dictionary (dict[str, Any]), das die aktiven Laufzeitkonfigurationswerte enthält.

    • Details: Die Werte aus config.host, config.port, config.session_ttl_seconds, config.run_timeout_seconds, config.live_run_timeout_seconds und config.tenant_id werden extrahiert und in die entsprechenden Typen (String für Host/TenantID, Integer für Zeitwerte) konvertiert. Beispiel (Konzeptuell):

# Wenn config.host = "127.0.0.1" und config.port = 8080
# Rückgabe:
# {
#     "host": "127.0.0.1",
#     "port": 8080,
#     "session_ttl_seconds": <Wert aus config.session_ttl_seconds>,
#     # ... weitere Werte
# }

---

2. def stored_runtime_config(base_path: Path, config: ServerConfig) -> dict[str, Any]

Beschreibung: Diese Funktion lädt die gespeicherte Laufzeitkonfiguration aus einer angegebenen Datei (base_path) und kombiniert sie mit den aktiven Konfigurationen, um eine konsolidierte Konfiguration zurückzugeben. Sie verwendet die aktiven Konfigurationen als Fallback, falls Werte in der gespeicherten Datei fehlen. Parameter:

  • base_path: Ein Path-Objekt, das den Pfad zur Datei enthält, aus der die gespeicherte Konfiguration geladen werden soll.
  • config: Ein Objekt vom Typ ServerConfig, das die aktuelle Konfiguration enthält.

    • Rückgabe:

  • Ein Dictionary (dict[str, Any]), das die konsolidierte Laufzeitkonfiguration enthält.

    • Details:

  1. Payload Laden: Zuerst wird die Konfigurationspayload aus der Datei mittels load_server_config_payload(base_path) geladen.
  2. Aktive Konfiguration: Die aktiven Konfigurationen werden über active_runtime_config(config) ermittelt.
  3. Konsolidierung: Die Werte aus dem payload werden verwendet, um die Werte aus der aktiven Konfiguration zu überschreiben, falls im Payload spezifische Werte vorhanden sind.
  • Priorisierung: Die Werte aus dem payload werden verwendet, sofern sie existieren. Andernfalls werden die Werte aus der aktiven Konfiguration (active) verwendet.
  • Typkonvertierung: Die Werte aus dem Payload werden entsprechend konvertiert (z.B. port zu int).

    • Beispiel (Konzeptuell):

# Wenn payload.get("port") existiert, wird dieser Wert verwendet, ansonsten wird active["port"] verwendet.
# Rückgabe:
# {
#     "host": "Wert aus payload oder active["host"]",
#     "port": <int-Wert aus payload oder active["port"]>,
#     # ... weitere Werte
# }

---

3. def runtime_config_requires_restart(active: dict[str, Any], stored: dict[str, Any]) -> bool

Beschreibung: Diese Funktion prüft, ob die aktive Laufzeitkonfiguration signifikant von der gespeicherten Konfiguration abweicht. Wenn Unterschiede in den definierten Konfigurationsparametern festgestellt werden, wird als True zurückgegeben, was auf einen Neustart der Laufzeitumgebung hindeutet. Parameter:

  • active: Das Dictionary mit den aktuellen, aktiven Laufzeitkonfigurationen (aus active_runtime_config).
  • stored: Das Dictionary mit den gespeicherten Laufzeitkonfigurationen (aus stored_runtime_config).

    • Rückgabe:

  • True, wenn die Konfigurationen nicht identisch sind (eine Änderung erforderlich).
  • False, wenn die Konfigurationen identisch sind.

    • Details:

  1. Iterierung: Die Funktion durchläuft alle Schlüssel (key) in der globalen Konstante RUNTIME_FILE_CONFIG_KEYS.
  2. Vergleich: Für jeden Schlüssel wird geprüft, ob der String-Wert der aktiven Konfiguration (str(active.get(key))) mit dem String-Wert der gespeicherten Konfiguration (str(stored.get(key))) übereinstimmt.
  3. Fehlererkennung: Sobald ein Schlüssel gefunden wird, für den die Werte nicht identisch sind, wird sofort True zurückgegeben.
  4. Erfolg: Wenn alle Schlüssel erfolgreich verglichen wurden und keine Diskrepanzen gefunden wurden, wird False zurückgegeben.

    5. Wichtige Abhängigkeiten:

  • RUNTIME_FILE_CONFIG_KEYS: Eine globale Konstante, die die Liste aller Konfigurationsschlüssel definiert, die verglichen werden sollen.

Zusammenfassung der Datenflüsse

Funktion Eingabe(n) Hauptzweck Ausgabe
active_runtime_config ServerConfig Aktive Konfiguration extrahieren. Aktive Konfigurations-Dict
stored_runtime_config Path, ServerConfig Speicherten Payload laden und mit aktiven Konfigurationen fusionieren. Konsolidierte Konfigurations-Dict
runtime_config_requires_restart Aktive & gespeicherte Dicts Abgleich der Konfigurationen zur Feststellung eines Neustarts. Boolean (True/False)
