Technische Dokumentation: test_analysis_archive_builder.py

Version: 1.0 Datum: 2023-10-27 Autor: [Ihr Name/Team] Beschreibung: Diese Datei enthält eine Sammlung von Unit-Tests, die die Funktionalität der Funktion build_source_analysis_archive aus dem Modul nova_school_server.analysis_archive_builder validieren. Die Tests konzentrieren sich auf die korrekte Erstellung eines Archivs (ZIP-Datei) aus einer gegebenen Quelle. ---

1. Überblick

Die Datei test_analysis_archive_builder.py dient als Testumgebung für die Hauptfunktion, die für die Erstellung von Analysearchiven zuständig ist. Sie nutzt das unittest-Framework, um sicherzustellen, dass die Archivierungslogik (wahrscheinlich das Sammeln und Komprimieren von Analyseergebnissen) gemäß den Spezifikationen funktioniert.

2. Abhängigkeiten

  • subprocess: Wird verwendet, um möglicherweise externe Befehle auszuführen (z.B. für das Komprimieren, falls nicht direkt durch zipfile abgedeckt).
  • tempfile: Wird verwendet, um temporäre Dateien für Tests zu erstellen.
  • unittest: Das Standard-Framework für die Testausführung.
  • zipfile: Wird verwendet, um die eigentliche ZIP-Datei zu erstellen und zu verifizieren.
  • pathlib.Path: Wird für die moderne und objektorientierte Handhabung von Dateipfaden verwendet.
  • nova_school_server.analysis_archive_builder.build_source_analysis_archive: Die Funktion, die getestet wird.

3. Struktur und Inhalt

Die Datei enthält eine einzelne Klasse, die die Testlogik kapselt.

3.1. Importe

Die notwendigen Bibliotheken und die zu testende Funktion werden importiert:

from __future__ import annotations
import subprocess
import tempfile
import unittest
import zipfile
from pathlib import Path
from nova_school_server.analysis_archive_builder import build_source_analysis_archive

3.2. Testklasse

Die Tests sind in einer Klasse organisiert, die von unittest.TestCase erbt. (Der Rest des Codes, der die Testmethoden enthält, ist in der bereitgestellten Snippet-Struktur impliziert.) Zweck der Testmethoden (Impliziert): Die Testmethoden werden darauf abzielen, verschiedene Szenarien für die Archivierung zu prüfen, wie zum Beispiel:

  1. Erfolgreiche Archivierung: Testet den Fall, in dem die Funktion erfolgreich eine ZIP-Datei erstellt.
  2. Fehlende Eingabe: Testet, wie die Funktion mit fehlenden oder ungültigen Eingabedaten umgeht (Erwartet: Ausnahme oder spezifische Fehlermeldung).
  3. Inhaltliche Integrität: Testet, ob alle erwarteten Analyseergebnisse korrekt in das Archiv aufgenommen werden.
  4. Ressourcenmanagement: Testet, ob temporäre Dateien korrekt bereinigt werden.

4. Funktionsweise der Hauptfunktion (Referenz)

Die Funktion build_source_analysis_archive ist das zentrale Objekt der Tests. Sie ist verantwortlich für:

  1. Die Sammlung aller relevanten Analyseergebnisse aus der Quelle.
  2. Die Organisation dieser Ergebnisse in eine strukturierte Form.
  3. Die Erstellung einer komprimierten Datei (ZIP-Archiv).
  4. Rückgabe des Pfades zur erstellten Archivdatei.

5. Teststrategie

Die Teststrategie basiert auf Unit Testing und Integration Testing (da sie die Interaktion zwischen der Analyselogik und der Dateiverarbeitung testen).

  • Mocking/Setup: Es wird erwartet, dass die Tests geeignete Testdaten (simulierte Analyseergebnisse) bereitstellen, um die Funktion unter kontrollierten Bedingungen zu testen.
  • Isolation: Jeder Test sollte idealerweise isoliert sein, um sicherzustellen, dass ein Fehler in einem Test nicht zu einem anderen Test führt.
  • Ressourcen-Cleanup: Die Verwendung von tempfile deutet darauf hin, dass die Tests sicherstellen sollen, dass erstellte Dateien nach Abschluss des Tests gelöscht werden.

6. Erwartete Ergebnisse

Bei erfolgreicher Ausführung der Tests sollte die folgende Aussage gelten:

  • Die Funktion build_source_analysis_archive liefert ein korrekt strukturiertes und komprimiertes Archiv, wenn die Eingabedaten gültig sind.
  • Die Funktion wirft angemessene Fehler, wenn die Eingabedaten fehlerhaft sind.

7. Anmerkungen und Verbesserungsvorschläge

  • Erweiterung der Testfälle: Es sollte geprüft werden, ob alle möglichen Zustände der Analyseergebnisse (z.B. leere Ergebnisse, sehr große Ergebnisse, Ergebnisse mit Sonderzeichen) abgedeckt sind.
  • Fehlerbehandlung: Die spezifischen Fehler, die build_source_analysis_archive wirft (z.B. FileNotFoundError, ZipError), sollten explizit in den Testmethoden abgefroren werden, um die Robustheit zu erhöhen.
  • Dokumentation der Eingaben/Ausgaben: Für die Testfälle selbst sollte eine klare Beschreibung der erwarteten Eingabedaten und der erwarteten Ausgabe (z.B. "Erwarte, dass die Datei /tmp/test_archive.zip erstellt wird") hinzugefügt werden.

Technische Dokumentation: test_analysis_archive_builder.py

1. Überblick

Diese Datei enthält eine Reihe von Unit-Tests, die die Funktionalität der Funktion build_source_analysis_archive (die in einer anderen Datei definiert ist) validieren. Insbesondere wird geprüft, dass das Archivierungssystem korrekt funktioniert, indem es Textquellen beibehält und binäre Dateien ignoriert.

2. Zielsetzung der Tests

Das Hauptziel dieser Tests ist es sicherzustellen, dass das SourceAnalysisArchiveBuilder die folgenden Regeln korrekt umsetzt:

  1. Textquellen: Dateien, die als Quellcode oder Dokumentation betrachtet werden (z.B. .py, .md, .html), müssen im Archiv enthalten sein.
  2. Binäre Dateien: Dateien, die als reine Binärdaten (z.B. Modelle, Bilder, Archive) betrachtet werden, dürfen nicht im Archiv enthalten sein.
  3. Strukturierung: Die Dateien müssen korrekt unter einem definierten Präfix im Archiv gespeichert werden.

3. Test-Details

Testfall: test_build_source_analysis_archive_keeps_text_source_and_skips_binaries

Dieser Test simuliert ein Git-Repository mit einer Mischung aus verschiedenen Dateitypen und führt die Archivierung durch. Setup (Setup-Phase):

  1. Temporäre Umgebung: Ein temporäres Verzeichnis wird erstellt, um das Test-Repository zu isolieren.
  2. Repository Initialisierung:
  • Ein Git-Repository wird initialisiert (self._git(repo, "init")).
  • Konfigurationsdateien (user.name, user.email) werden hinzugefügt.
  1. Dateien und Verzeichnisse erstellt:
  • Textquellen (Sollten enthalten sein):
  • pyproject.toml (Konfigurationsdatei)
  • server.py (Python-Code)
  • static/index.html (HTML-Datei)
  • Docs/guide.md (Markdown-Dokumentation)
  • LIT/README.md (Markdown-Dokumentation)
  • Binäre Dateien (Sollten ignoriert werden):
  • LIT/model.gguf (Modell-Datei)
  • Model/offline.gguf (Modell-Datei)
  • dist/old.zip (Archivdatei)
  • image.png (Bilddatei)
  1. Commit: Die Dateien werden mit einer Commit-Nachricht versehen.

    2. Aktion (Execution-Phase):

  2. Die Funktion build_source_analysis_archive(repo, output_dir=repo / "out") wird aufgerufen, um das Archiv zu erstellen.

    3. Assertion (Erwartete Ergebnisse):

  3. Existenz des Archivs: Es wird sichergestellt, dass die erzeugte Archivdatei (result.archive_path) existiert.
  4. Versionsprüfung: Die Version im Archiv (result.version) muss mit der im Commit ("0.1.12") übereinstimmen.
  5. Inhaltliche Validierung (Archiv-Namensliste):
  • Erwartet (Textquellen): server.py, static/index.html, Docs/guide.md, LIT/README.md müssen im Archiv enthalten sein.
  • Erwartet (Textquellen-Ausschluss): Docs/guide.html muss nicht enthalten sein (da es eine HTML-Datei ist, die im Test nicht explizit als Textquelle behandelt wurde, oder es ist eine binäre/spezielle Datei).
  • Erwartet (Binär-Ausschluss): LIT/model.gguf, Model/offline.gguf, dist/old.zip, und image.png müssen nicht im Archiv enthalten sein.

4. Hilfsmethoden

Methode: _git(repo: Path, *args: str) -> None

  • Zweck: Eine private Hilfsmethode zur Ausführung von Git-Befehlen innerhalb des angegebenen Repositorys.
  • Funktionalität: Führt einen subprocess.run aufzurufen, um Git-Befehle (z.B. init, add, commit) auszuführen.
  • Fehlerbehandlung: Prüft den returncode des Git-Befehls. Bei einem Fehler wird eine AssertionError mit der Fehlermeldung des Git-Befehls (stdout/stderr) geworfen.

Die erfolgreiche Ausführung dieses Tests bestätigt, dass die Klassifizierung von Dateien (basierend auf Dateiendungen oder Kontext) für die Archivierung korrekt implementiert ist.
