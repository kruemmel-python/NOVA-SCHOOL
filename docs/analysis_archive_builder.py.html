Technische Dokumentation: analysis_archive_builder.py

1. Überblick

Die Python-Datei analysis_archive_builder.py dient dazu, einen Archivierungs- und Build-Prozess für eine Analyse zu automatisieren. Sie ermöglicht es, bestimmte Dateien und Verzeichnisse zu identifizieren, zu filtern und diese in ein ZIP-Archiv zu komprimieren. Das Skript verwendet Argumenten-Parsing (argparse) für die Konfiguration und definiert spezifische Regeln für die Auswahl von Dateien und Verzeichnissen, die entweder ausgeschlossen oder erlaubt sind.

2. Funktionen und Klassen

2.1. Globale Konfiguration (Constants)

Das Skript definiert mehrere globale Konstanten, die die Regeln für die Archivierung festlegen:

  • **SKIP_DIR_NAMES**: Eine Menge von Verzeichnissnamen, die ignoriert werden sollen (z.B. .git, venv, node_modules, dist, Model).
  • **SKIP_FILE_SUFFIXES**: Eine Menge von Dateiendungen, die ignoriert werden sollen (z.B. .pyc, .dll, .png, .exe).
  • **ALLOWED_SUFFIXES**: Eine Menge von Dateiendungen, die für die Archivierung in Betracht gezogen werden sollen (z.B. .py, .js, .json, .xml).
  • **ALLOWED_FILE_NAMES**: Eine Menge von spezifischen Dateinamen, die explizit in das Archiv aufgenommen werden sollen (z.B. .gitignore, Dockerfile, requirements.txt).

2.2. Dataclass: SourceAnalysisArchiveBuildResult

Diese dataclass wird verwendet, um das Ergebnis des Archivierungs- und Build-Prozesses strukturiert zu speichern.

Attribut Typ Beschreibung
version str Die Version des generierten Archivs.
archive_path Path Der vollständige Pfad zum erstellten ZIP-Archiv.
file_count int Die Gesamtzahl der Dateien, die im Archiv enthalten sind.

3. Hauptlogik (Impliziert)

Obwohl der bereitgestellte Code nur die Definitionen und die Dataclass enthält, impliziert die Struktur folgende Hauptfunktionen (basierend auf den Importen und Konstanten):

  1. Argumenten-Parsing: Verwendung von argparse, um von der Kommandozeile wichtige Parameter (z.B. Quellverzeichnis, Zielpfad) zu erhalten.
  2. Verzeichnis-Traversal: Durchlaufen des angegebenen Quellverzeichnisses.
  3. Filterung:
  • Ausschluss: Überprüfung, ob ein Verzeichnis in SKIP_DIR_NAMES enthalten ist.
  • Ausschluss: Überprüfung, ob eine Datei eine Endung aus SKIP_FILE_SUFFIXES hat.
  • Einschluss: Überprüfung, ob eine Datei eine Endung aus ALLOWED_SUFFIXES hat oder ob sie einen Namen aus ALLOWED_FILE_NAMES hat.
  1. Archivierung: Sammlung der gefilterten Dateien und Erstellung eines ZIP-Archivs an dem vom Benutzer definierten Zielpfad.
  2. Ergebnis-Rückgabe: Rückgabe eines Objekts vom Typ SourceAnalysisArchiveBuildResult.

4. Konstanten-Details

4.1. SKIP_DIR_NAMES

Diese Liste dient der schnellen Identifizierung und Ignorierung von Standard-Projektstrukturen, temporären Dateien und Umgebungsvariablen, um unnötige oder irrelevante Inhalte aus dem Archiv zu entfernen.

4.2. SKIP_FILE_SUFFIXES

Diese Liste filtert Dateien, die typischerweise nicht für die Analyse oder das finale Deployment relevant sind (z.B. Bytecode-Dateien, Cache-Dateien, ausführbare Binärdateien).

4.3. ALLOWED_SUFFIXES

Diese Liste definiert die primären Programmier- und Konfigurationsdateiendungen, die für die Analyse relevant sind.

4.4. ALLOWED_FILE_NAMES

Diese Liste stellt sicher, dass wichtige Metadaten- oder Build-Dateien (wie Dockerfile, requirements.txt) explizit erfasst werden, unabhängig von ihrer Endung.

5. Nutzungshinweise

Dieses Skript ist für den automatisierten Aufbau von Analyse-Backups oder Deployment-Paketen konzipiert. Annahme: Das Skript muss mit einem Quellverzeichnis und einem Zielpfad gestartet werden, die über die Kommandozeile über argparse definiert werden. Erwartete Ausgabe: Bei erfolgreicher Ausführung wird ein Objekt vom Typ SourceAnalysisArchiveBuildResult zurückgegeben, das den Pfad zum erstellten Archiv und die Anzahl der enthaltenen Dateien enthält. --- Ende der Dokumentation

Technische Dokumentation: analysis_archive_builder.py

Übersicht

Diese Datei enthält Funktionen zur Erstellung von Analyse-Archiven für ein Projekt namens "NovaSchoolAnalyzer". Die Hauptfunktionen umfassen die automatische Versionserkennung des Projekts und die Kompression der Analyse-Dateien in ein ZIP-Archiv.

Funktionen

1. detect_project_version(base_path: Path) -> str

Beschreibung: Diese Funktion scannt die pyproject.toml-Datei im angegebenen Pfad, um die Versionsnummer des Projekts zu extrahieren. Parameter:

  • base_path: Path: Der Pfad zum Projektverzeichnis, in dem die pyproject.toml liegt.

    • Rückgabe:

  • str: Die extrahierte Versionsnummer (z.B. "1.2.3"). Wenn die Datei nicht gefunden wird oder keine Versionszeile mit dem Präfix "version = " gefunden wird, wird standardmäßig "0.1.0" zurückgegeben.

    • Details:

  1. Sie konstruiert den vollständigen Pfad zu pyproject.toml.
  2. Prüft, ob die Datei existiert. Falls nicht, gibt sie "0.1.0" zurück.
  3. Liest den Inhalt der Datei.
  4. Iteriert über alle Zeilen und sucht nach einer Zeile, die mit "version = " beginnt.
  5. Extrahiert den Wert nach dem Gleichheitszeichen (=) und entfernt eventuelle umgebenden Anführungszeichen (" oder ').

    6. ---

2. `build_source_analysis_archive(

base_path: Path, *, output_dir: Path | None = None, version: str | None = None, ) -> SourceAnalysisArchiveBuildResult **Beschreibung:** Diese Hauptfunktion erstellt ein ZIP-Archiv aller relevanten Quellanalyse-Dateien aus dem base_path und speichert es im angegebenen oder standardmäßig im dist-Verzeichnis des Basisverzeichnisses. Sie verwendet entweder eine übergeordnete version` oder erkennt die Version automatisch. Parameter:

  • base_path: Path: Der Pfad zum Verzeichnis, das die Analyse-Dateien enthält.
  • output_dir: Path | None = None: Optionaler Zielpfad für das Archiv. Wenn None angegeben wird, wird das Verzeichnis base_path/dist verwendet.
  • version: str | None = None: Optionaler Versionsstring. Wenn dieser angegeben wird, wird er verwendet; andernfalls wird die Version automatisch erkannt.

    • Rückgabe:

  • SourceAnalysisArchiveBuildResult: Ein Objekt, das Informationen über das erstellte Archiv enthält.

    • Details:

  1. Pfad-Resolution: Der base_path wird aufgelöst (resolve(strict=False)). Das output_dir wird entweder explizit gesetzt oder als base_path/dist definiert und wird erstellt (mkdir(parents=True, exist_ok=True)).
  2. Versionsbestimmung: Die Versionsnummer wird entweder aus dem optionalen version-Parameter übernommen oder durch die Funktion detect_project_version(base_path) ermittelt.
  3. Archivnamen: Der Name des Archivs wird als NovaSchoolAnalyzer-v<version_text>.zip formatiert.
  4. Bereinigung: Falls das Archiv bereits existiert, wird es gelöscht (unlink()), bevor es neu erstellt wird.
  5. Archiv-Erstellung:
  • Es wird ein zipfile.ZipFile mit Komprimierung (ZIP_DEFLATED) und Kompressionslevel 9 geöffnet.
  • Die Funktion _iter_source_analysis_files(base_path) wird aufgerufen, um eine Liste der relativen Pfade der zu inkludierenden Dateien zu erhalten.
  • Jede Datei wird aus base_path relativ zum Archivpfad geschrieben und unter dem Präfix root_prefix gespeichert.
  • Die Anzahl der inkludierten Dateien wird erfasst.
  1. Rückgabe: Ein SourceAnalysisArchiveBuildResult wird zurückgegeben, das die verwendete Versionsnummer, den Pfad zum Archiv und die Anzahl der Dateien enthält.

    2. ---

Abhängigkeiten

  • pathlib.Path (aus dem pathlib-Modul)
  • zipfile (aus dem zipfile-Modul)
  • SourceAnalysisArchiveBuildResult (Muss in einem anderen Teil der Datei definiert sein)
  • _iter_source_analysis_files (Eine externe, nicht gezeigte Funktion, die die zu archivierenden Dateien liefert)

Verwendung (Beispiel)

Um das Archiv zu erstellen, könnte man es wie folgt aufrufen:

from pathlib import Path
# Angenommen, SourceAnalysisArchiveBuildResult und _iter_source_analysis_files sind definiert
# 1. Definieren des Basisverzeichnisses
project_root = Path("./my_analysis_project")
# 2. Erstellen des Archivs (Version wird automatisch erkannt)
result = build_source_analysis_archive(
base_path=project_root,
output_dir=Path("./dist_archives")
)
print(f"Archiv erstellt: {result.archive_path}")
print(f"Dateien im Archiv: {result.file_count}")

Alternative (Explizite Version):

result_v1 = build_source_analysis_archive(
base_path=project_root,
version="2.0.0"
)

Technische Dokumentation: analysis_archive_builder.py

Übersicht

Diese Datei enthält Funktionen zur Iteration und Filterung von Dateien, die als Kandidaten für eine Analyse-Archivierung dienen sollen. Insbesondere die Funktion _iter_source_analysis_files ist dafür verantwortlich, eine Liste von relativen Pfaden zu generieren, die bestimmte Kriterien erfüllen, bevor sie für die weitere Verarbeitung verwendet werden.

Funktionen

_iter_source_analysis_files(base_path: Path) -> list[Path]

Diese Funktion durchläuft alle Dateien, die von Git verfolgt werden, im angegebenen base_path, und filtert sie basierend auf einer Reihe von Kriterien, um eine sortierte Liste von relevanten Quelldateien zurückzugeben. Parameter:

  • base_path: Path: Der absolute Pfad zum Verzeichnis, in dem die Git-Repository-Operationen durchgeführt werden sollen.

    • Rückgabe:

  • list[Path]: Eine sortierte Liste von Path-Objekten, die die relativen Pfade der qualifizierten Quelldateien enthalten.

    • Beschreibung:

  1. Initialisiert eine leere Liste files.
  2. Ruft die interne Funktion _git_tracked_files(base_path) auf, um alle Dateien zu erhalten, die von Git verfolgt werden.
  3. Iteriert über die zurückgegebenen relativen Pfade:
  • Konstruiert den vollständigen Pfad (candidate) zu der Datei im base_path.
  • Filter 1 (Existenz): Prüft, ob der konstruierte Pfad tatsächlich eine Datei ist (candidate.is_file()). Wenn nicht, wird die Datei übersprungen.
  • Filter 2 (Ignorieren): Ruft _should_skip_source_analysis_file(relative_path) auf. Wenn diese Funktion True zurückgibt, wird die Datei übersprungen.
  • Filter 3 (Erlaubnis): Ruft _is_allowed_source_file(relative_path) auf. Wenn diese Funktion False zurückgibt, wird die Datei übersprungen.
  • Filter 4 (Typ): Ruft _is_probably_text_file(candidate) auf. Wenn diese Funktion False zurückgibt (d.h., es ist wahrscheinlich keine Textdatei), wird die Datei übersprungen.
  • Wenn alle Prüfungen erfolgreich sind, wird der relative Pfad zur Liste files hinzugefügt.
  1. Gibt die final sortierte Liste der relativen Pfade zurück.

_git_tracked_files(base_path: Path) -> list[Path]

Diese interne Funktion verwendet das git ls-files-Kommando, um eine Liste aller Dateien zurückzugeben, die vom Git-Repository im angegebenen Verzeichnis verfolgt werden. Parameter:

  • base_path: Path: Der absolute Pfad zum Git-Repository-Verzeichnis.

    • Rückgabe:

  • list[Path]: Eine Liste von Path-Objekten, die die relativen Pfade der Git-verfolgten Dateien enthalten.

    • Beschreibung:

  1. Führt das Shell-Kommando git ls-files im angegebenen Verzeichnis (cwd=base_path) aus.
  2. Erwartet einen erfolgreichen Ausführungscode (result.returncode == 0).
  3. Fehlerbehandlung: Wenn der Ausführungscode nicht 0 ist, wird eine RuntimeError mit einer zusammenfassenden Fehlermeldung (aus stderr oder stdout) geworfen.
  4. Ergebnisverarbeitung: Wenn der Befehl erfolgreich ist, werden die Ausgabe (stdout) in Zeilen aufgeteilt und jede Zeile wird zu einem Path-Objekt konvertiert, wobei Leerzeichen entfernt werden.

Abhängigkeiten

  • pathlib.Path: Für die Handhabung von Dateipfaden.
  • subprocess: Für die Ausführung externer Befehle (git).

Anmerkungen

  • Die Logik der Filterung in _iter_source_analysis_files ist sequenziell und basiert auf der Annahme, dass die Kriterien von oben nach unten angewendet werden (jede vorherige Prüfung führt zum Überspringen der Datei, falls sie fehlschlägt).
  • Die spezifischen Implementierungen der Helper-Funktionen (_should_skip_source_analysis_file, _is_allowed_source_file, _is_probably_text_file) sind in dieser Dokumentation nicht detailliert beschrieben, da sie externe Logik enthalten.

Technische Dokumentation: analysis_archive_builder.py

Übersicht

Diese Datei enthält eine Reihe von internen Funktionen, die zur Filterung und Bewertung von Dateien dienen, die für einen Analyse-Archiv-Builder in Verwendung sind. Die Funktionen bestimmen, ob eine Datei übersprungen werden soll (_should_skip_source_analysis_file), ob sie als Quelle für die Analyse betrachtet werden soll (_is_allowed_source_file), und ob sie wahrscheinlich Textdaten enthält (_is_probably_text_file).

Funktionen

1. _should_skip_source_analysis_file(relative_path: Path) -> bool

Diese Funktion entscheidet, ob eine spezifische Datei, basierend auf ihrem Pfad und Namen, übersprungen werden soll. Beschreibung: Überprüft verschiedene Kriterien, um festzustellen, ob eine Datei als Quelle für die Analyse ignoriert werden sollte. Parameter:

  • relative_path: Path: Der relative Pfad zur Datei, die bewertet werden soll.

    • Rückgabetyp:

  • bool: True, wenn die Datei übersprungen werden soll; False, andernfalls nicht.

    • Logikdetails:

  1. Verzeichnisprüfung: Prüft, ob Teile des Pfades (außer dem letzten Teil) in der Liste SKIP_DIR_NAMES enthalten sind.
  2. Spezialpfadprüfung: Prüft, ob der Pfad exakt "Linux/project" ist.
  3. Temporäre Dateien: Prüft, ob der Dateiname mit _tmp_ oder tmp_ beginnt.
  4. Dump-Dateien: Prüft, ob der Dateiname mit _analysis_dump.md oder .codedump.md endet.
  5. Dateiendung: Prüft, ob die Dateiendung (suffix) in der Liste SKIP_FILE_SUFFIXES enthalten ist.
  6. Dokumentenprüfung: Prüft, ob der Pfad mit "Docs" beginnt und die Endung .html hat.

    7. ---

2. _is_allowed_source_file(relative_path: Path) -> bool

Diese Funktion bestimmt, ob eine Datei als gültige Quelle für die Analyse betrachtet werden darf. Beschreibung: Prüft, ob der Dateiname oder die Dateiendung der Datei in den definierten Listen der erlaubten Namen oder Endungen enthalten ist. Parameter:

  • relative_path: Path: Der relative Pfad zur Datei, die bewertet werden soll.

    • Rückgabetyp:

  • bool: True, wenn die Datei erlaubt ist; False, andernfalls nicht.

    • Logikdetails:

  1. Name-Prüfung: Prüft, ob der Dateiname (relative_path.name) in der Liste ALLOWED_FILE_NAMES enthalten ist.
  2. Endung-Prüfung: Prüft, ob die Dateiendung (in Kleinbuchstaben) in der Liste ALLOWED_SUFFIXES enthalten ist.

    3. ---

3. _is_probably_text_file(path: Path) -> bool

Diese Funktion bewertet, ob eine Datei wahrscheinlich Textinhalt enthält, indem sie einen kleinen Teil der Datei liest und die Kodierung prüft. Beschreibung: Leset die ersten 8192 Bytes der Datei und versucht, diese als UTF-8 zu dekodieren. Sie prüft auf Null-Byte-Zeichen (\x00) und versucht bei Fehlern eine Dekodierung mit utf-8-sig. Parameter:

  • path: Path: Der Pfad zur Datei, die analysiert werden soll.

    • Rückgabetyp:

  • bool: True, wenn die Datei als Text betrachtet wird; False, andernfalls nicht.

    • Logikdetails:

  1. Bytes-Lesung: Liest die ersten 8192 Bytes der Datei (path.read_bytes()[:8192]).
  2. Null-Byte-Prüfung: Wenn ein Null-Byte (b"\x00") im Lesevorgang gefunden wird, wird die Funktion sofort auf False gesetzt.
  3. UTF-8 Dekodierung: Versucht, die Bytes als UTF-8 zu dekodieren.
  4. Fallback-Dekodierung: Falls die UTF-8-Dekodierung fehlschlägt (UnicodeDecodeError), versucht sie, die Bytes als UTF-8 mit Signatur (utf-8-sig) zu dekodieren.
  5. Fehlversuch: Wenn beide Dekodierungsversuche fehlschlagen, wird die Funktion auf False gesetzt.
  6. Erfolg: Wird die Dekodierung erfolgreich abgeschlossen, wird True zurückgegeben.

Abhängigkeiten und Konstanten (Impliziert)

Die Funktionalität dieser Funktionen hängt von externen globalen Variablen ab, die in der Datei definiert sein müssen:

  • **SKIP_DIR_NAMES**: Eine Liste von Pfadkomponenten, die bei einem Pfad-Match zu Überspringen führen.
  • **SKIP_FILE_SUFFIXES**: Eine Liste von Dateiendungen, die bei einem Dateiendungs-Match zu Überspringen führen.
  • **ALLOWED_FILE_NAMES**: Eine Liste von Dateinamen, die als Quelle erlaubt sind.
  • **ALLOWED_SUFFIXES**: Eine Liste von Dateiendungen, die als Quelle erlaubt sind.

Zusammenfassung der Nutzungsszenarien

Diese Funktionen dienen als Gatekeeper im Analyse-Archiv-Builder:

  1. Filterung (Skip): _should_skip_source_analysis_file entscheidet, welche Dateien überhaupt in den Analyseprozess aufgenommen werden.
  2. Validierung (Allow): _is_allowed_source_file stellt sicher, dass die identifizierten Quellen den definierten Spezifikationen entsprechen.
  3. Typ-Prüfung (Content Check): _is_probably_text_file hilft, Dateien zu identifizieren, die textbasierte Analyse durchlaufen sollten, indem sie die Datenintegrität (insbesondere die Anwesenheit von Null-Bytes) prüft.

Technische Dokumentation: analysis_archive_builder.py

1. Überblick

Die Python-Skriptdatei analysis_archive_builder.py dient dazu, ein Quellcode-basiertes ZIP-Archiv für die Analyse von KI-Projekten zu erstellen. Das Skript wird über die Kommandozeile (CLI) ausgeführt, um einen bestimmten Projektordner zu verarbeiten und das resultierende Archiv an einen definierten Speicherort zu speichern.

2. Funktionsweise

Die Hauptfunktion des Skripts ist die Orchestrierung des Archivierprozesses, indem es Argumente von der Kommandozeile entgegennimmt und die Kernlogik zur Erstellung des Archivs aufruft.

2.1. Hauptfunktion (main())

Die main-Funktion ist der zentrale Einstiegspunkt des Skripts und übernimmt folgende Schritte:

  1. Argumenten-Parsing: Sie initialisiert ein argparse.ArgumentParser, um die notwendigen Kommandozeilenargumente zu definieren.
  2. Argument-Erfassung: Sie parst die übergebenen Argumente (z.B. den Basispfad, den Ausgabedirektor und eine Versionsnummer).
  3. Pfad-Resolutions: Sie konvertiert die übergebenen Pfade in Path-Objekte und stellt sicher, dass der base_path absolut aufgelöst wird (resolve(strict=False)).
  4. Ausgabepfad-Konfiguration: Sie konstruiert den vollständigen Ausgabepfad (output_dir). Wenn der übergebene output-dir nicht absolut ist, wird er relativ zum base_path konstruiert.
  5. Archiv-Erstellung: Sie ruft die Kernfunktion build_source_analysis_archive auf, um das eigentliche Archiv zu erstellen.
  6. Ergebnis-Ausgabe: Sie gibt den Pfad zum erstellten Archiv aus.

2.2. Abhängigkeiten

  • **pathlib.Path:** Für die Manipulation und Auflösung von Dateipfaden.
  • **argparse:** Für die Definition und Verarbeitung von Kommandozeilenargumenten.
  • **build_source_analysis_archive(...):** Eine externe Funktion (nicht im bereitgestellten Code sichtbar), die die eigentliche Logik zur Erstellung des ZIP-Archivs implementiert.

3. Kommandozeilenargumente

Das Skript akzeptiert folgende Argumente:

Argument Typ Standardwert Beschreibung Pflicht
base_path String . Der Wurzelpfad des Projekts, aus dem die Analyse durchgeführt werden soll. Nein (Standard: .)
--output-dir String dist Der Name des Verzeichnisses, in dem das resultierende ZIP-Archiv gespeichert werden soll. Nein (Standard: dist)
--version String "" Eine optionale Versionsnummer, die in den Archivmetadaten aufgenommen werden soll. Nein

4. Code-Analyse (Auszug)

def main() -> None:
parser = argparse.ArgumentParser(description="Build a source-only ZIP archive for AI analysis.")
parser.add_argument("base_path", nargs="?", default=".", help="Projektwurzel")
parser.add_argument("--output-dir", default="dist", help="Ausgabeordner fuer das ZIP-Archiv")
parser.add_argument("--version", default="", help="Optionale Versionsueberschreibung")
args = parser.parse_args()
base_path = Path(args.base_path).resolve(strict=False)
output_dir = Path(args.output_dir)
if not output_dir.is_absolute():
output_dir = base_path / output_dir
result = build_source_analysis_archive(base_path, output_dir=output_dir, version=args.version or None)
print(result.archive_path)
if __name__ == "__main__":
main()

4.1. Pfad-Handling

  • **base_path = Path(args.base_path).resolve(strict=False):** Der bereitgestellte Basispfad wird in ein Path-Objekt umgewandelt und aufgelöst. strict=False erlaubt es, wenn der Pfad nicht existiert, trotzdem weiterzuarbeiten (obwohl die nachfolgende Logik davon abhängt, ob die Dateien tatsächlich existieren).
  • **output_dir = Path(args.output_dir):** Der gewünschte Ausgabedirektor wird initialisiert.
  • **if not output_dir.is_absolute(): output_dir = base_path / output_dir:** Dies ist eine wichtige Logik. Wenn der Benutzer einen relativen Pfad für --output-dir angibt (z.B. logs/), wird dieser Pfad relativ zum base_path konstruiert, um eine konsistente Archivstruktur zu gewährleisten.

4.2. Kernlogik

  • **result = build_source_analysis_archive(...):** Hier wird die eigentliche Verarbeitung des Quellcodes und die Erstellung des ZIP-Archivs durch die externe Funktion ausgelöst.
  • **print(result.archive_path):** Das Skript gibt den finalen Pfad zum erstellten Archiv an die Standardausgabe aus.

5. Verwendung (Beispiel)

Um das Skript auszuführen, wird es mit dem Basisordner als erstes Argument aufgerufen:

python analysis_archive_builder.py /pfad/zu/meinem/projekt

Beispiel mit Optionen: Um das Archiv in einem separaten Ordner namens analysis_output zu speichern und eine Version hinzuzufügen:

python analysis_archive_builder.py /pfad/zu/meinem/projekt --output-dir analysis_output --version v1.2.3

Erwartete Ausgabe (Beispiel):

/pfad/zu/meinem/projekt/dist/analysis_project_v1.2.3.zip
