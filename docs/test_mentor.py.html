3. Hauptkomponenten

3.1. Klasse _Session

Diese Klasse dient zur Strukturierung und Speicherung der Kontextinformationen eines Benutzers während einer Sitzung oder eines Mentoring-Prozesses. Attribute:

  • username: str: Der eindeutige Benutzername.
  • is_teacher: bool = False: Flag, ob der Benutzer ein Lehrer ist (Standardwert: False).
  • permissions: dict | None = None: Ein Dictionary, das spezifische Berechtigungen des Benutzers speichert (Standardwert: leeres Dictionary {}).
  • group_ids: list[str] | None = None: Eine Liste von IDs von Gruppen, zu denen der Benutzer gehört (Standardwert: leere Liste []).
  • user: dict: Ein internes Dictionary, das die öffentliche Darstellung des Benutzers enthält, z.B. {"display_name": username.title()}.

    • **Konstruktor (__init__):** Initialisiert die Attribute und setzt Standardwerte für permissions und group_ids.

Technische Dokumentation: Funktion _bundle_archive(secret: str) -> bytes

1. Übersicht

Die Funktion _bundle_archive ist eine interne Funktion, die eine komplexe Datenstruktur (Manifest, Kursinformationen, Mentor-Regeln) basierend auf einem geheimen Schlüssel (secret) serialisiert, signiert und in ein ZIP-Archiv verpackt. Das Ergebnis ist die binäre Datenstruktur des erstellten ZIP-Archivs. Zweck: Erstellung eines standardisierten, verschlüsselten und signierten Archivs, das spezifische Lernmaterialien und Mentor-Anweisungen für einen bestimmten Kurs enthält. Eingabe:

  • secret: str: Ein geheimes Passwort oder Schlüssel, der für die kryptografische Signatur verwendet wird.

    • Ausgabe:

  • bytes: Die binären Daten des erstellten ZIP-Archivs.

2. Funktionsweise (Algorithmus)

Die Funktion folgt mehreren Schritten, um das finale Archiv zu generieren:

  1. Manifest-Erstellung: Ein Dictionary (manifest) wird erstellt, das Metadaten über das Archiv selbst enthält (z.B. bundle_id, schema_version, Erstellungsdatum).
  2. Kurs-Datenstruktur: Ein detailliertes Dictionary (course) wird definiert, das alle spezifischen Informationen zu einem Kurs (z.B. python-grundlagen) enthält, einschließlich Module, Quizfragen und Abschlussprüfungen.
  3. Mentor-Regel-Definition: Eine spezifische Mentor-Regel (mentor_rule) wird definiert, die Anweisungen für einen bestimmten Kurs und Modul enthält.
  4. Kanonische Serialisierung: Die gesamte Struktur (Manifest, Kurs, Mentor-Regel) wird zu einem JSON-String serialisiert (canonical). Dabei werden sort_keys=True und separators=(",", ":") verwendet, um eine konsistente Darstellung zu gewährleisten.
  5. Signatur-Erstellung:
  • SHA-256-Hash: Der canonical-String wird mittels SHA-256 gehasht (canonical_sha256).
  • HMAC-Signatur: Eine HMAC-SHA256-Signatur wird berechnet, wobei der secret als Schlüssel verwendet wird, um die Integrität des canonical-Strings zu verifizieren (signature).
  1. Archiv-Erstellung (ZIP):
  • Ein io.BytesIO-Buffer wird erstellt, um die Daten im Speicher zu speichern.
  • Ein zipfile.ZipFile wird auf diesen Buffer geschrieben.
  • Inhalte des Archivs:
  • manifest.json: Das serialisierte Manifest.
  • signature.json: Die serialisierte Signaturinformation.
  • courses/python-grundlagen.json: Die serialisierte Kursstruktur.
  • mentor_rules/python-grundlagen-p01.json: Die serialisierte Mentor-Regel.
  1. Rückgabe: Die gesamten binären Daten des erstellten ZIP-Archivs werden über buffer.getvalue() zurückgegeben.

3. Abhängigkeiten

Die Funktion benötigt die folgenden Standard-Python-Module:

  • time: Für die Erstellung des Zeitstempels.
  • json: Für die Serialisierung der Daten in JSON-Formate.
  • hashlib: Für die Berechnung des SHA-256-Hashes.
  • hmac: Für die Berechnung der HMAC-Signatur.
  • io: Für die Verwendung von io.BytesIO als in-memory-Speicher.
  • zipfile: Für die Erstellung des ZIP-Archivs.

4. Datenstruktur-Schema (Output-Archiv)

Das generierte ZIP-Archiv enthält folgende Dateien:

Dateiname Inhaltstyp Beschreibung
manifest.json JSON Metadaten und Versionsinformationen des Bundles.
signature.json JSON Die berechnete SHA-256-Hash und die HMAC-SHA256-Signatur.
courses/python-grundlagen.json JSON Die vollständige Struktur des Kurses, einschließlich aller Module und Prüfungen.
mentor_rules/python-grundlagen-p01.json JSON Die spezifische Mentor-Anweisung für das erste Modul des Kurses.

5. Sicherheit und Integrität

Die Funktion implementiert eine grundlegende Integritätsprüfung durch kryptografische Mechanismen:

  • Integrität: Die canonical_sha256 stellt sicher, dass der ursprüngliche JSON-String unverändert ist.
  • Authentizität: Die signature (HMAC-SHA256) stellt sicher, dass die Daten von einer autorisierten Quelle stammen und nicht manipuliert wurden, da sie nur mit dem secret generiert werden kann.

  • Die Datenintegrität und -authentizität des Pakets gewährleistet werden muss.

Technische Dokumentation: MentorTests

1. Überblick

Diese Dokumentation beschreibt die Struktur und die Testfälle der Klasse MentorTests, die Teil eines größeren Testsets (vermutlich für eine Mentor-Funktionalität) ist. Die Tests konzentrieren sich auf die korrekte Verarbeitung von Kontextinformationen, die Begrenzung von Eingaben (Prompt-Truncation) und die korrekte Generierung von Mentor-Anweisungen.

2. Zielsetzung der Tests

Das Hauptziel dieser Testklasse ist es, die Funktionalität der SocraticMentorService (über die Mentor-Klasse) zu validieren, insbesondere in folgenden Bereichen:

  1. Kontext-Inklusion: Sicherstellen, dass der generierte Prompt den notwendigen Kurskontext und Modulinformationen enthält.
  2. Anweisungserstellung: Verifizieren, dass die Mentor-Anweisung spezifische, handlungsleitende Fragen generiert.
  3. Kontext-Trunkierung: Überprüfen, dass sehr lange Eingaben (Code, Laufzeit-Output) korrekt verkürzt werden, um die Verarbeitungseffizienz zu gewährleisten.

3. Teststruktur und Setup

3.1. Klasse: MentorTests

Diese Klasse verwendet das Standard-unittest-Framework, um die Methoden zu definieren, die die Funktionalität des Mentor-Systems testen.

3.2. Methode: setUp(self) -> None

Die setUp-Methode wird vor jedem einzelnen Test aufgerufen, um einen konsistenten und isolierten Testumgebungszustand zu erstellen. Ablauf:

  1. Temporäre Umgebung: Ein temporäres Verzeichnis (self.tmp) wird erstellt, um Dateien und Datenbanken für den Testlauf zu isolieren.
  2. Repository Initialisierung:
  • Ein SchoolRepository wird mit einer Datenbankdatei (school.db) in das temporäre Verzeichnis initialisiert.
  • Eine spezifische Einstellung (curriculum_bundle_secret) wird gesetzt.
  • Ein Benutzer (student) wird erstellt mit spezifischen Berechtigungen (curriculum.use: True, mentor.use: True).
  1. Service-Setup:
  • Der CurriculumService wird mit dem Repository instanziiert.
  • Ein Administrator-Session (admin) wird erstellt, um die notwendigen Berechtigungen für die Import-Funktionalität zu simulieren.
  • Die Import-Funktionalität (import_bundle_archive) wird aufgerufen, um einen Bundle zu importieren und zu aktivieren.
  1. Mentor-Instanziierung:
  • Der SocraticMentorService (self.mentor) wird mit dem Repository und dem CurriculumService initialisiert.
  1. Session-Setup:
  • Eine _Session (self.session) für einen Studenten wird erstellt, der die notwendigen Berechtigungen besitzt.
  1. Projekt-Definition:
  • Eine definierte Projektstruktur (self.project) wird festgelegt.

3.3. Methode: tearDown(self) -> None

Die tearDown-Methode wird nach Abschluss jedes Tests aufgerufen, um die Ressourcen freizugeben:

  1. Repository Schließen: Die Datenbankverbindung wird geschlossen (self.repository.close()).
  2. Temporäre Umgebung Aufräumen: Das temporäre Verzeichnis wird bereinigt (self.tmp.cleanup()).

4. Testfälle (Test Cases)

4.1. Testfall: test_prepare_includes_curriculum_context_and_bundle_mentor_rules

Ziel: Validieren, dass die mentor.prepare-Funktion den generierten Prompt mit allen erwarteten Kontextinformationen und spezifischen Mentor-Anweisungen füllt. Input:

  • Prompt: "Warum sehe ich keine Ausgabe?"
  • Code: "print(\"Hallo\")"
  • Kontext: course_id="python-grundlagen", module_id="p01_einstieg"

    • Erwartete Assertions:

  • Der generierte Prompt muss den Kurskontext ("Kurskontext: Python Grundlagen") enthalten.
  • Der generierte Prompt muss den aktuellen Modulkontext ("Aktuelles Modul: Einfuhrung in Python und Programmieren") enthalten.
  • Der generierte Prompt muss eine spezifische Mentor-Vorgabe enthalten (z.B. "Frage zuerst nach der Funktion von print()").
  • Der generierte Prompt muss eine allgemeine Frage enthalten ("Welche Zeile zeigt etwas auf dem Bildschirm?").

4.2. Testfall: test_prepare_trims_large_runtime_context

Ziel: Validieren, dass die Funktion bei sehr großen Eingaben (Code und Laufzeit-Output) den Prompt korrekt verkürzt (Truncation), um die Verarbeitungsgrenzen einzuhalten. Input:

  • Code: Eine sehr lange Liste von print(index) Aufrufen (6000 Zeilen).
  • Laufzeit-Output: Eine sehr lange Liste von Fehlermeldungen (2000 Zeilen).

    • Erwartete Assertions:

  • Der generierte Prompt muss die ursprüngliche Frage enthalten.
  • Der generierte Prompt muss eine spezifische Aufgabe enthalten ("Aufgabe: Fuehre den Nutzer ueber Fragen").
  • Der generierte Prompt muss den Marker für Prompt-Truncation (PROMPT_TRUNCATION_MARKER.strip()) enthalten, was belegt, dass die Begrenzung aktiv wurde.
  • Die Länge des generierten Prompts muss unter einem definierten Schwellenwert liegen (hier: < 9000 Zeichen), was die erfolgreiche Trunkierung bestätigt.

Die Testsuite stellt sicher, dass die Mentor-Funktionalität nicht nur logisch korrekt ist, sondern auch robust gegenüber großen Datenmengen ist (durch Trunkierung) und kontextsensitiv ist (durch korrekte Inkorporation von Kurs- und Modulinformationen). Die erfolgreiche Ausführung dieser Tests bestätigt die korrekte Implementierung der Kontext-Handling- und Prompt-Generierungslogik im SocraticMentorService.
