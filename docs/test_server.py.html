Technische Dokumentation: test_server.py

Version: 1.0 Datum: 2026-04-06 Autor: Ralf Krümmel

1. Überblick

Diese Datei (test_server.py) dient wahrscheinlich als Test- oder Setup-Skript für einen Server, der mit der NovaSchoolApplication und NovaSchoolRequestHandler interagiert. Sie definiert grundlegende Klassenstrukturen (_Repo und _App), die für die Konfiguration und das Repository-Management des Servers zuständig sind.

2. Struktur und Komponenten

Die Datei definiert zwei Hauptklassen: _Repo und _App.

2.1. Klasse _Repo

Diese Klasse ist verantwortlich für das Abrufen von Konfigurationsdaten (Settings) aus einem bereitgestellten Dictionary. Zweck: Bereitstellung einer einfachen Schnittstelle zum Abrufen von Einstellungen. Attribute:

  • self.settings: dict[str, str]: Ein Dictionary, das die Konfigurationsparameter des Servers speichert.

    • Methoden:

  • __init__(self, settings: dict[str, str]) -> None: Initialisiert die Klasse mit dem Konfigurations-Dictionary.
  • get_setting(self, key: str, default=None) -> str | None: Gibt den Wert der Schlüssel (key) aus dem self.settings zurück. Wenn der Schlüssel nicht existiert, wird der vom Benutzer bereitgestellte default Wert verwendet (oder None, falls kein Default angegeben ist).

2.2. Klasse _App

Diese Klasse stellt die Hauptanwendungseinheit dar und initialisiert das Repository sowie die spezifische Konfiguration. Zweck: Initialisierung der Hauptanwendung mit Repository-Zugriff und Port-Konfiguration. Attribute:

  • self.repository: _Repo: Eine Instanz der _Repo-Klasse, die für den Zugriff auf die Einstellungen zuständig ist.
  • self.config: Eine interne Klasse (oder eine einfache Klasse), die die spezifischen Konfigurationsparameter der Anwendung hält.

    • **Konstruktor (__init__):**

  • __init__(self, settings: dict[str, str], port: int = 8877) -> None:
  • Initialisiert das self.repository mit den übergebenen settings.
  • Definiert eine interne Klasse namens Config, die die Anwendungskonfiguration hält und standardmäßig einen port von 8877 verwendet.

    • **Interne Klasse Config:**

  • Diese Klasse ist eine einfache, nicht-erweiterbare Klasse, die nur das Attribut port enthält.

3. Abhängigkeiten

  • __future__ import annotations: Wird verwendet, um moderne Typ-Hinting-Syntax zu ermöglichen.
  • import tempfile: Importiert das tempfile-Modul (wahrscheinlich für temporäre Dateisystemoperationen in den Tests).
  • import unittest: Importiert das unittest-Modul (bestätigt, dass dies ein Test-Kontext ist).
  • from pathlib import Path: Importiert das pathlib-Modul für objektorientierte Dateisystemoperationen.
  • from nova_school_server.server import NovaSchoolApplication, NovaSchoolRequestHandler: Importiert die Hauptlogik-Klassen des Servers.

4. Verwendung und Anwendungsfälle

Die Klassen _Repo und _App dienen als Low-Level-Abstraktionen für die Konfiguration und das Datenmanagement des Servers.

  1. Konfigurationsmanagement: _Repo.get_setting() wird verwendet, um dynamisch Einstellungen aus einer zentralen Quelle (dem settings-Dictionary) abzurufen.
  2. Anwendungsinitialisierung: _App.__init__() wird verwendet, um die gesamte Anwendungsumgebung einzurichten, indem sie das Repository und die spezifische Port-Konfiguration initialisiert.

5. Zusammenfassung der Klassenstruktur

Klasse Hauptverantwortung Schlüsselmethoden
_Repo Abrufen von Konfigurationsdaten. get_setting(key, default)
_App Initialisierung der Anwendung, Verwaltung von Repository und Port-Konfiguration. __init__(settings, port)

--- Hinweis: Die Datei scheint primär die Setup- und Konfigurationsschicht zu definieren. Die eigentliche Geschäftslogik (Server-Start, Request-Handling) wird wahrscheinlich in NovaSchoolApplication und NovaSchoolRequestHandler implementiert.

Technische Dokumentation: test_server.py

Übersicht

Diese Datei enthält eine Reihe von Unit-Tests, die die Funktionalität der Klasse NovaSchoolRequestHandler und der zugrunde liegenden Komponenten (wie _App und NovaSchoolApplication) validieren. Die Tests konzentrieren sich insbesondere auf:

  1. Cookie-Header-Verhalten: Wie Secure-Attribute in Cookies basieren, wenn der Host HTTPS ist oder ein X-Forwarded-Proto-Header vorhanden ist.
  2. URL-Generierung: Wie die URL für die Zertifikatsprüfung (_certificate_verification_url) basierend auf dem öffentlichen Host und Forwarded-Proto konstruiert wird.
  3. Dateisystem-Sicherheit: Die Ablehnung von Dateitour-Operationen (../) durch die Funktion _resolve_relative_file.
  4. Konfiguration von Binärpfaden: Die korrekte Auflösung des Pfades für den Standard-LITLM-Binary-Pfad unter bestimmten Bedingungen.

Struktur der Datei

Die Datei enthält eine Testklasse, die von der Standardbibliothek unittest verwendet wird.

Klasse: RequestHandlerTlsTests

Diese Klasse erbt von unittest.TestCase und enthält spezifische Testmethoden.

1. Methode: _handler(self, settings: dict[str, str] | None = None, headers: dict[str, str] | None = None) -> NovaSchoolRequestHandler

  • Zweck: Eine private Hilfsmethode, die eine Instanz von NovaSchoolRequestHandler erstellt.
  • Funktionalität:
  • Erstellt eine Instanz von NovaSchoolRequestHandler über den Konstruktor (NovaSchoolRequestHandler.__new__).
  • Initialisiert die handler.application mit den bereitgestellten settings (oder einem leeren Dictionary, falls None übergeben wird).
  • Initialisiert die handler.headers mit den bereitgestellten headers (oder einem leeren Dictionary).
  • Gibt die konfigurierte handler zurück.

2. Testfälle für Cookie-Header

Diese Tests prüfen, ob der Cookie-Header korrekt mit dem Secure-Attribut versehen wird, wenn die Umgebung HTTPS verwendet.

  • **test_cookie_header_uses_secure_when_public_host_is_https():**
  • Szenario: Der öffentliche Host ist https://nova.schule.local.
  • Erwartung: Der zurückgegebene Cookie-Header enthält ; Secure.
  • **test_cookie_header_uses_secure_when_forwarded_proto_is_https():**
  • Szenario: Der Host-Header ist nova.schule.local und X-Forwarded-Proto ist https.
  • Erwartung: Der zurückgegebene Cookie-Header enthält ; Secure.
  • **test_clear_cookie_header_uses_secure_when_forwarded_proto_is_https():**
  • Szenario: Gleiches Szenario wie oben, aber für die Methode _clear_cookie_header().
  • Erwartung: Der zurückgegebene Cookie-Header enthält ; Secure.

3. Testfälle für Zertifikatsprüfung (URL-Generierung)

Diese Tests validieren, wie die URL für die Zertifikatsprüfung konstruiert wird, basierend auf der Konfiguration.

  • **test_certificate_verification_url_prefers_https_public_host():**
  • Szenario: server_public_host ist https://nova.schule.local.
  • Erwartung: Die generierte URL beginnt mit https://nova.schule.local/certificate/verify?certificate_id=....
  • **test_certificate_verification_url_uses_forwarded_https_for_bare_public_host():**
  • Szenario: server_public_host ist nova.schule.local, aber X-Forwarded-Proto ist https.
  • Erwartung: Die generierte URL beginnt mit https://nova.schule.local/certificate/verify?certificate_id=... (Hinweis: Die URL scheint den vollständigen Hostname zu verwenden, was auf eine korrekte Logik zur Host-Resolution hindeutet).

4. Testfall für Dateisystem-Sicherheit

  • **test_resolve_relative_file_rejects_traversal():**
  • Szenario: Versucht, einen relativen Pfad (../secret.txt) von einem temporären Verzeichnis zu einem Verzeichnis außerhalb des Verzeichnisses zu auflösen.
  • Erwartung: Es wird eine PermissionError (oder eine ähnliche Fehlermeldung, die auf eine Sicherheitsbeschränkung hinweist) geworfen, was die Traversal-Angriffe verhindert.

5. Testfall für Konfiguration von Binärpfaden

  • **test_default_litertlm_binary_path_accepts_linux_lit_binary():**
  • Szenario: Ein temporäres Verzeichnis wird erstellt, und ein spezifischer Pfad (base_path/Linux/LIT/lit.linux_x86_64) wird erstellt und mit Inhalt gefüllt.
  • Erwartung: Die Methode app.default_litertlm_binary_path() muss den absoluten, aufgelösten Pfad zum erstellten Binary-Datei zurückgeben. Dies stellt sicher, dass die Konfiguration korrekt auf das tatsächliche Dateisystem-Objekt verweist.

Zusammenfassung der Testziele

Die Testsuite stellt sicher, dass die NovaSchoolRequestHandler robuste und sichere Verhaltensweisen bei der Verarbeitung von HTTP/HTTPS-Kontexten, der Generierung von Ressourcen-URLs und der Handhabung von Dateipfaden gewährleistet.
