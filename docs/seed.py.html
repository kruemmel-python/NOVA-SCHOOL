Technische Dokumentation: seed.py

1. Überblick

Die Datei seed.py dient dazu, initialisierende Daten, Konfigurationen und möglicherweise Testdaten für eine Anwendung zu bereitstellen. Sie importiert Komponenten aus verschiedenen Modulen (z.B. Authentifizierung, Datenbank, Dokumentationskatalog, Berechtigungen und Workspace-Management) und definiert spezifische Konfigurationswerte oder Berechtigungssets.

2. Struktur und Importe

Die Datei importiert Abhängigkeiten aus verschiedenen internen Modulen:

  • **from .auth import AuthService**: Importiert die Klasse AuthService, die wahrscheinlich für Authentifizierungsfunktionen zuständig ist.
  • **from .code_runner import DEFAULT_CONTAINER_IMAGES**: Importiert eine Konstante oder eine Liste von Standard-Container-Bildern für Code-Runner.
  • **from .database import SchoolRepository**: Importiert die Klasse SchoolRepository, die für die Interaktion mit der Datenbank (speziell für Informationen zu Schulen) verantwortlich ist.
  • **from .docs_catalog import DocumentationCatalog**: Importiert die Klasse DocumentationCatalog, die wahrscheinlich die Struktur oder den Katalog der verfügbaren Dokumentationen verwaltet.
  • **from .permissions import ROLE_DEFAULTS**: Importiert eine Konfiguration oder einen Satz von Standardberechtigungen basierend auf Rollen.
  • **from .workspace import WorkspaceManager, slugify**: Importiert die Klassen WorkspaceManager zur Verwaltung von Arbeitsbereichen und die Funktion slugify zur URL-freundlichen Umbenennung von Strings.

3. Konstanten und Konfigurationen

3.1. DEMO_GROUP_PERMISSIONS

Diese globale Variable definiert ein spezifisches Set von Berechtigungen, das für eine "Demo-Gruppe" zugewiesen ist.

DEMO_GROUP_PERMISSIONS = {
"workspace.group": True,
"chat.use": True,
"docs.read": True,
}

Beschreibung: Dieses Dictionary speichert die Berechtigungen für eine bestimmte Gruppe (vermutlich eine Test- oder Demo-Gruppe). Es zeigt an, dass diese Gruppe Zugriff auf die Gruppenverwaltung, die Nutzung von Chats und das Lesen von Dokumenten besitzt.

4. Funktionen

Die Datei enthält derzeit keine explizit definierten Funktionen, außer der Importstruktur und der Definition der Konfiguration. (Hinweis: Der Code-Block endet hier, was darauf hindeutet, dass die eigentlichen Initialisierungs- oder Seed-Funktionen in den folgenden Teilen der Datei folgen würden.)

5. Zusammenfassung der Verantwortlichkeiten

seed.py fungiert als zentraler Einstiegspunkt für die Initialisierung von Systemkomponenten und die Festlegung von Standardkonfigurationen für Tests, Demonstrationen oder die Bereitstellung von Startdaten. Schlüsselkomponenten:

  • Importmanagement für alle Kernmodule.
  • Definition von spezifischen Berechtigungssets (DEMO_GROUP_PERMISSIONS).

6. Anmerkungen und Verbesserungsvorschläge

  1. Funktionsdefinition: Die Datei scheint derzeit nur Importe und eine Konfiguration zu enthalten. Es wird erwartet, dass hier die eigentlichen seed-Funktionen (z.B. seed_database(), seed_users(), etc.) definiert werden, die die Datenbank und andere Dienste initialisieren.
  2. Dokumentation der Importe: Für größere Projekte wäre es hilfreich, eine kurze Erklärung zu liefern, warum diese spezifischen Module importiert werden (z.B. "Importiert SchoolRepository zur Initialisierung der Schuldatenbank").
  3. Typ-Hinweise: Da es sich um eine Python-Datei handelt, sollten die definierten Strukturen (wie DEMO_GROUP_PERMISSIONS) durch Docstrings oder Type Hinting weiter spezifiziert werden, falls sie komplexer werden.

Technische Dokumentation: bootstrap_application Funktion

1. Übersicht

Die Funktion bootstrap_application dient dazu, die notwendigen Initialisierungen und Konfigurationen für die Nova School Server-Umgebung vorzunehmen. Sie stellt sicher, dass die notwendigen Dokumentationen gesät werden, Systemparameter konfiguriert werden, Benutzer und Gruppen erstellt werden, sowie die ersten Demo-Projekte im Repository initialisiert werden.

2. Funktion

def bootstrap_application(repository: SchoolRepository, auth_service: AuthService, docs_catalog: DocumentationCatalog, workspace_manager: WorkspaceManager) -> dict[str, object]:
# ... (Implementierung)

Parameter:

Name Typ Beschreibung
repository SchoolRepository Schnittstelle zum Repository, um Einstellungen zu setzen, Projekte zu erstellen und Benutzer/Gruppen zu verwalten.
auth_service AuthService Schnittstelle zur Authentifizierung, um Benutzer zu erstellen und zu verwalten.
docs_catalog DocumentationCatalog Schnittstelle zur Verwaltung von Dokumentationen, um sicherzustellen, dass Seed-Dokumentationen vorhanden sind.
workspace_manager WorkspaceManager Schnittstelle zur Verwaltung von Arbeitsbereichen und Projekten (z.B. Materialisierung von Projekten).

Rückgabe:

Typ Beschreibung
dict[str, object] Ein Dictionary, das die Ergebnisse der Initialisierung enthält, typischerweise: seed_users, seed_group und role_defaults.

3. Funktionsbeschreibung (Schritt-für-Schritt)

Die Funktion führt die folgenden Hauptschritte durch:

3.1. Dokumentationsinitialisierung

  1. Dokumentationen sicherstellen: Ruft docs_catalog.ensure_seed_docs() auf, um sicherzustellen, dass alle notwendigen Seed-Dokumentationen vorhanden sind.

3.2. Systemkonfiguration (Repository Settings)

Die Funktion setzt eine Vielzahl von Konfigurationsparametern im repository über repository.put_setting(). Diese Einstellungen betreffen verschiedene Komponenten wie:

  • Server- und Proxy-Konfiguration: school_name, server_public_host, web_proxy_url, web_proxy_no_proxy, web_proxy_required.
  • LLaMA-Modell-Konfiguration: Pfade, Aliasse, Backend (vulkan), Kontextgröße (llamacpp_ctx_size), GPU-Layer (llamacpp_gpu_layers), Threads, Schlafzeiten und Port (llamacpp_port).
  • LITERLM-Modell-Konfiguration: Pfade, Aliasse, Backend (cpu), Schlafzeiten und Port (litertlm_port).
  • On-Device-Konfiguration: Maximale Tokens, Top-K-Werte, Temperatur und Zufallsseed.
  • Container-Konfiguration: Backend (container), Runtime (docker), Speicher-/CPU-Limits, PID-Limits, Dateigrößenlimits, Seccomp-Einstellungen und Bild-Definitionen für verschiedene Sprachen (Python, Node, C++, Java, Rust).
  • Scheduler-Konfiguration: Maximale gleichzeitige Konkurrenzwerte für globale, studentische und Lehrer-Rollen.

3.3. Benutzer- und Gruppenmanagement

  1. Benutzererstellung: Erstellt drei spezifische Benutzerkonten:
  • admin: Mit Passwort NovaSchool!admin (Rolle: Server Admin).
  • teacher: Mit Passwort NovaSchool!teacher (Rolle: Lehrkraft).
  • student: Mit Passwort NovaSchool!student (Rolle: Schueler Demo).
  1. Gruppenerstellung: Erstellt eine Gruppe namens "class-1a" mit der Beschreibung "Demo-Klasse fuer den Nova School Server" und spezifischen Berechtigungen (DEMO_GROUP_PERMISSIONS).
  2. Benutzerzuweisung: Fügt den Benutzer "student" zur Gruppe "class-1a" hinzu.
  3. Workspace-Ordner: Erstellt Profile-Ordner für jeden Seed-Benutzer (user und group).

3.4. Projektinitialisierung (Demo-Projekte)

Die Funktion prüft, ob bestimmte Demo-Projekte bereits existieren, und erstellt sie bei Bedarf:

  1. Projekt "Python Labor":
  • Bedingung: Prüft, ob ein Projekt mit Owner user, Key student und Slug python-labor existiert.
  • Aktion: Erstellt das Projekt (Python Labor) mit Template python, Runtime python und main.py. Materialisiert das Projekt.
  1. Projekt "Web Labor":
  • Bedingung: Prüft, ob ein Projekt mit Owner group, Key class-1a und Slug web-labor existiert.
  • Aktion: Erstellt das Projekt (Web Labor) mit Template frontend-lab, Runtime html und index.html. Materialisiert das Projekt.
  1. Projekt "Distributed Playground":
  • Bedingung: Prüft, ob ein Projekt mit Owner user, Key teacher und Slug distributed-playground existiert.
  • Aktion: Erstellt das Projekt (Distributed Playground) mit Template distributed-system, Runtime python und services/coordinator.py. Materialisiert das Projekt.

3.5. Rückgabewert

Die Funktion gibt ein strukturiertes Dictionary zurück, das die initialisierten Daten enthält:

  • **seed_users**: Eine Liste der erstellten Benutzer mit ihren Benutzernamen und Passwörtern.
  • **seed_group**: Der Name der erstellten Gruppe (class-1a).
  • **role_defaults**: Ein Dictionary, das Standard-Rollenkonfigurationen für student, teacher und admin bereitstellt.

Die Funktion hängt von folgenden externen Abhängigkeiten und Konstanten ab (die in der bereitgestellten Code-Sektion nicht definiert sind, aber impliziert werden):
